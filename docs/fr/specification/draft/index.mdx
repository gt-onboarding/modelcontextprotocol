<div id="enable-section-numbers" />

Le [Model Context Protocol](https://modelcontextprotocol.io) (MCP) est un protocole ouvert qui
permet une intégration fluide entre les applications LLM, les sources de données externes et les
outils. Que vous développiez un IDE propulsé par l’IA, amélioriez une interface de chat ou créiez
des flux de travail d’IA personnalisés, MCP fournit une méthode standardisée pour connecter les LLM au contexte
dont ils ont besoin.

Cette spécification définit les exigences faisant autorité du protocole, sur la base du
schéma TypeScript dans
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts).

Pour des guides de mise en œuvre et des exemples, consultez
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Les mots-clés « MUST », « MUST NOT », « REQUIRED », « SHALL », « SHALL NOT », « SHOULD », « SHOULD
NOT », « RECOMMENDED », « NOT RECOMMENDED », « MAY » et « OPTIONAL » dans ce document doivent être
interprétés comme décrit dans [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] lorsqu’ils apparaissent, et uniquement dans ce cas,
en lettres majuscules, comme indiqué ici.

<div id="overview">
  ## Vue d’ensemble
</div>

Le MCP fournit une manière standardisée pour les applications de :

* Partager des informations contextuelles avec des modèles de langage
* Exposer des outils et des capacités aux systèmes d’IA
* Créer des intégrations et des workflows composables

Le protocole utilise des messages [JSON-RPC](https://www.jsonrpc.org/) 2.0 pour établir
la communication entre :

* **Hôtes** : applications LLM qui initient les connexions
* **Clients** : connecteurs au sein de l’application hôte
* **Serveurs** : services qui fournissent du contexte et des capacités

Le MCP s’inspire en partie du
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), qui
standardise la manière d’ajouter la prise en charge des langages de programmation dans tout un écosystème d’outils de développement. De manière similaire, le MCP standardise la façon d’intégrer du contexte supplémentaire
et des outils dans l’écosystème des applications d’IA.

<div id="features">
  ### Fonctionnalités
</div>

Les serveurs peuvent proposer aux clients les fonctionnalités suivantes :

* **Ressources** : Contexte et données, à utiliser par l’utilisateur ou le modèle d’IA
* **Invites** : Modèles de messages et workflows paramétrés pour les utilisateurs
* **Outils** : Fonctions que le modèle d’IA peut exécuter

Les clients peuvent proposer les fonctionnalités suivantes aux serveurs :

* **Échantillonnage** : Comportements agentiques initiés par le serveur et interactions LLM récursives
* **Racines** : Requêtes initiées par le serveur concernant les périmètres d’URI ou de système de fichiers dans lesquels opérer
* **Élicitation** : Requêtes initiées par le serveur pour obtenir des informations supplémentaires auprès des utilisateurs

<div id="additional-utilities">
  ### Fonctions supplémentaires
</div>

* Configuration
* Suivi de la progression
* Annulation
* Signalement des erreurs
* Journalisation

<div id="security-and-trust-safety">
  ## Sécurité et confiance & sûreté
</div>

Le Protocole de contexte de modèle (MCP) offre de puissantes fonctionnalités via un accès arbitraire aux données
et des cheminements d’exécution de code. Ce pouvoir s’accompagne de considérations importantes en matière de sécurité et de confiance
que tous les implémenteurs doivent aborder avec rigueur.

<div id="key-principles">
  ### Principes clés
</div>

1. **Consentement et contrôle de l’utilisateur**
   * Les utilisateurs doivent donner un consentement explicite à tout accès aux données et à toute opération, et les comprendre
   * Les utilisateurs doivent garder la maîtrise des données partagées et des actions effectuées
   * Les implémenteurs devraient fournir des interfaces claires pour examiner et autoriser les activités

2. **Confidentialité des données**
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’exposer des données utilisateur à des serveurs
   * Les hôtes ne doivent pas transmettre des données de ressources ailleurs sans le consentement de l’utilisateur
   * Les données utilisateur doivent être protégées par des contrôles d’accès appropriés

3. **Sécurité des Outils**
   * Les Outils permettent l’exécution de code arbitraire et doivent être traités avec la prudence
     appropriée.
     * En particulier, les descriptions du comportement des outils, telles que les annotations, doivent être
       considérées comme non fiables, sauf si elles proviennent d’un Serveur MCP de confiance.
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’invoquer un outil
   * Les utilisateurs doivent comprendre ce que fait chaque outil avant d’en autoriser l’utilisation

4. **Contrôles d’Échantillonnage LLM**
   * Les utilisateurs doivent approuver explicitement toute demande d’Échantillonnage LLM
   * Les utilisateurs devraient contrôler :
     * Si un échantillonnage a lieu ou non
     * L’invite réelle qui sera envoyée
     * Quels résultats le serveur peut voir
   * Le protocole limite intentionnellement la visibilité du serveur sur les Invites

<div id="implementation-guidelines">
  ### Directives d’implémentation
</div>

Bien que le MCP ne puisse pas imposer ces principes de sécurité au niveau du protocole,
les implémenteurs **DEVRAIENT** :

1. Intégrer des parcours de consentement et d’autorisation robustes dans leurs applications
2. Fournir une documentation claire sur les implications en matière de sécurité
3. Mettre en place des contrôles d’accès et des protections des données adaptés
4. Suivre les meilleures pratiques de sécurité dans leurs intégrations
5. Prendre en compte les implications en matière de confidentialité dans la conception de leurs fonctionnalités
