<Info>**Révision du protocole** : 2025-03-26</Info>

Le Protocole de contexte de modèle (MCP) se compose de plusieurs composants clés qui fonctionnent de concert :

* **Protocole de base** : types de messages JSON-RPC principaux
* **Gestion du cycle de vie** : initialisation de la connexion, négociation des capacités et
  contrôle de session
* **Fonctionnalités du serveur** : Ressources, Invites et Outils exposés par les serveurs
* **Fonctionnalités du client** : Échantillonnage et listes de répertoires racines fournies par les clients
* **Utilitaires** : aspects transversaux comme la journalisation et l’auto-complétion des arguments

Toutes les implémentations **DOIVENT** prendre en charge le protocole de base et les composants de gestion du cycle de vie.
D’autres composants **PEUVENT** être implémentés en fonction des besoins spécifiques de l’application.

Ces couches du protocole établissent une séparation claire des responsabilités tout en permettant des interactions riches entre clients et serveurs. La conception modulaire permet aux implémentations de prendre en charge exactement les fonctionnalités dont elles ont besoin.

<div id="messages">
  ## Messages
</div>

Tous les messages échangés entre les Clients MCP et les Serveurs MCP **DOIVENT** respecter la spécification
[JSON-RPC 2.0](https://www.jsonrpc.org/specification). Le protocole définit
les types de messages suivants :

<div id="requests">
  ### Requêtes
</div>

Les requêtes sont envoyées du client vers le serveur, ou l’inverse, pour initier une opération.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Les requêtes **DOIVENT** inclure un identifiant de type chaîne ou entier.
* Contrairement à JSON-RPC de base, l’identifiant **NE DOIT PAS** être `null`.
* L’identifiant de la requête **NE DOIT PAS** avoir été utilisé auparavant par l’émetteur au cours de la même session.

<div id="responses">
  ### Réponses
</div>

Les réponses sont envoyées suite aux requêtes et contiennent le résultat de l’opération ou une erreur.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* Les réponses **DOIVENT** inclure le même ID que la requête correspondante.
* Les **réponses** sont ensuite classées en **résultats** ou en **erreurs**. Un `result` ou une `error` **DOIT** être défini. Une réponse **NE DOIT PAS** définir les deux.
* Les résultats **PEUVENT** avoir n’importe quelle structure d’objet JSON, tandis que les erreurs **DOIVENT** inclure au minimum un code d’erreur et un message.
* Les codes d’erreur **DOIVENT** être des entiers.

<div id="notifications">
  ### Notifications
</div>

Les notifications sont envoyées du client au serveur, ou l’inverse, sous forme de message unidirectionnel.
Le destinataire **NE DOIT PAS** envoyer de réponse.

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Les notifications **NE DOIVENT PAS** inclure d’identifiant.

<div id="batching">
  ### Traitement par lots
</div>

JSON-RPC définit également un moyen de
[traiter plusieurs requêtes et notifications par lots](https://www.jsonrpc.org/specification#batch),
en les envoyant dans un tableau. Les implémentations MCP **PEUVENT** prendre en charge l’envoi de lots JSON-RPC,
mais **DOIVENT** prendre en charge la réception de lots JSON-RPC.

<div id="auth">
  ## Authentification
</div>

Le MCP fournit un cadre d’[autorisation](/fr/specification/2025-03-26/basic/authorization) à utiliser avec HTTP.
Les implémentations utilisant un transport HTTP **DEVRAIENT** se conformer à cette spécification,
tandis que celles utilisant le transport STDIO **NE DEVRAIENT PAS** la suivre,
et devraient plutôt récupérer les informations d’identification depuis l’environnement.

En outre, clients et serveurs **PEUVENT** négocier leurs propres stratégies d’authentification et
d’autorisation personnalisées.

Pour participer aux discussions et contribuer à l’évolution des mécanismes d’auth du MCP, rejoignez-nous
sur
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
pour aider à façonner l’avenir du protocole !

<div id="schema">
  ## Schéma
</div>

La spécification complète du protocole est définie par un
[schéma TypeScript](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.ts).
Il s’agit de la référence faisant autorité pour tous les messages et structures du protocole.

Il existe également un
[schéma JSON](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.json),
généré automatiquement à partir de cette référence TypeScript, destiné à être utilisé avec
divers outils d’automatisation.
