title: Spécification
---

[Model Context Protocol](https://modelcontextprotocol.io) (MCP) est un protocole ouvert qui
permet une intégration fluide entre les applications LLM, les sources de données externes et les
outils. Que vous développiez un IDE propulsé par l’IA, amélioriez une interface de chat ou créiez
des workflows d’IA personnalisés, MCP fournit une manière standardisée de connecter les LLM au contexte
dont ils ont besoin.

Cette spécification définit les exigences de protocole faisant autorité, d’après le
schéma TypeScript dans
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.ts).

Pour des guides de mise en œuvre et des exemples, consultez
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Les mots‑clés « MUST », « MUST NOT », « REQUIRED », « SHALL », « SHALL NOT », « SHOULD », « SHOULD
NOT », « RECOMMENDED », « NOT RECOMMENDED », « MAY » et « OPTIONAL » dans ce document doivent être
interprétés comme décrit dans [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] lorsque, et seulement lorsque, ils
apparaissent en majuscules, comme illustré ici.

<div id="overview">
  ## Vue d’ensemble
</div>

Le MCP fournit une méthode standardisée permettant aux applications de :

* Partager des informations contextuelles avec des modèles de langage
* Exposer des Outils et des capacités aux systèmes d’IA
* Créer des intégrations et des flux de travail composables

Le protocole utilise des messages [JSON-RPC](https://www.jsonrpc.org/) 2.0 pour établir la communication entre :

* **Hôtes** : applications LLM qui initient les connexions
* **Clients** : connecteurs au sein de l’application hôte
* **Serveurs** : services qui fournissent du contexte et des capacités

Le MCP s’inspire en partie du
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), qui
standardise la manière d’ajouter la prise en charge de langages de programmation à l’échelle d’un écosystème entier d’outils de développement. De façon similaire, le MCP standardise l’intégration de contexte supplémentaire et d’Outils dans l’écosystème des applications d’IA.

<div id="base-protocol">
  ### Protocole de base
</div>

* Format de messages [JSON-RPC](https://www.jsonrpc.org/)

<div id="features">
  ### Fonctionnalités
</div>

Les serveurs peuvent proposer aux clients l’une des fonctionnalités suivantes :

* **Ressources** : contexte et données, à utiliser par l’utilisateur ou le modèle d’IA
* **Invites** : messages et workflows paramétrés pour les utilisateurs
* **Outils** : fonctions que le modèle d’IA peut exécuter

Les clients peuvent proposer la fonctionnalité suivante aux serveurs :

* **Échantillonnage** : comportements agentiques initiés par le serveur et interactions LLM récursives

<div id="additional-utilities">
  ### Outils supplémentaires
</div>

* Configuration
* Suivi de l’avancement
* Annulation
* Signalement des erreurs
* Journalisation

<div id="security-and-trust-safety">
  ## Sécurité, confiance et sûreté
</div>

Le Protocole de contexte de modèle (MCP) offre des capacités puissantes grâce à l’accès arbitraire aux données
et à des chemins d’exécution de code. Cette puissance s’accompagne de considérations importantes en matière de sécurité et de confiance
que tous les implémenteurs doivent aborder avec soin.

<div id="key-principles">
  ### Principes clés
</div>

1. **Consentement et contrôle de l’utilisateur**
   * Les utilisateurs doivent donner un consentement explicite et comprendre tous les accès aux données et opérations
   * Les utilisateurs doivent conserver le contrôle sur les données partagées et les actions effectuées
   * Les implémenteurs doivent fournir des interfaces claires pour examiner et autoriser les activités

2. **Confidentialité des données**
   * Les Hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’exposer des données utilisateur aux Serveurs
   * Les Hôtes ne doivent pas transmettre des données de Ressources ailleurs sans le consentement de l’utilisateur
   * Les données utilisateur doivent être protégées par des contrôles d’accès appropriés

3. **Sécurité des Outils**
   * Les Outils représentent une exécution de code arbitraire et doivent être traités avec la prudence appropriée.
     * En particulier, les descriptions du comportement des outils, telles que les annotations, doivent être considérées comme non fiables, sauf si elles proviennent d’un Serveur de confiance.
   * Les Hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’invoquer un outil
   * Les utilisateurs doivent comprendre ce que fait chaque outil avant d’en autoriser l’utilisation

4. **Contrôles d’Échantillonnage LLM**
   * Les utilisateurs doivent approuver explicitement toute demande d’échantillonnage LLM
   * Les utilisateurs doivent contrôler :
     * Si l’échantillonnage a lieu ou non
     * L’invite exacte qui sera envoyée
     * Quels résultats le Serveur peut voir
   * Le protocole limite intentionnellement la visibilité du Serveur sur les Invites

<div id="implementation-guidelines">
  ### Directives de mise en œuvre
</div>

Bien que le MCP lui-même ne puisse pas imposer ces principes de sécurité au niveau du protocole,
les implémenteurs DEVRAIENT :

1. Intégrer dans leurs applications des parcours robustes de consentement et d’autorisation
2. Fournir une documentation claire sur les implications en matière de sécurité
3. Mettre en place des contrôles d’accès et des protections des données appropriés
4. Suivre les meilleures pratiques de sécurité dans leurs intégrations
5. Prendre en compte les implications en matière de confidentialité dans la conception de leurs fonctionnalités
