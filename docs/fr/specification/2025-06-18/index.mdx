<div id="enable-section-numbers" />

Le [Protocole de contexte de modèle](https://modelcontextprotocol.io) (MCP) est un protocole ouvert qui
permet une intégration fluide entre les applications à base de LLM et les sources de données externes ainsi que
les outils. Que vous développiez un IDE piloté par l’IA, amélioriez une interface de chat, ou créiez
des workflows d’IA personnalisés, MCP offre une manière standardisée de connecter les LLM au contexte
dont ils ont besoin.

Cette spécification définit les exigences de protocole faisant autorité, basées sur le
schéma TypeScript dans
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts).

Pour des guides de mise en œuvre et des exemples, rendez-vous sur
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Les mots-clés « MUST », « MUST NOT », « REQUIRED », « SHALL », « SHALL NOT », « SHOULD », « SHOULD
NOT », « RECOMMENDED », « NOT RECOMMENDED », « MAY » et « OPTIONAL » dans ce document doivent être
interprétés comme décrit dans [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] lorsque, et seulement lorsque, ils
apparaissent en majuscules, comme indiqué ici.

<div id="overview">
  ## Aperçu
</div>

Le MCP offre une méthode standardisée permettant aux applications de :

* Partager des informations contextuelles avec des modèles de langage
* Exposer des outils et des capacités aux systèmes d’IA
* Créer des intégrations et des workflows composables

Le protocole utilise des messages [JSON-RPC](https://www.jsonrpc.org/) 2.0 pour établir la
communication entre :

* **Hôtes** : applications LLM qui initient les connexions
* **Clients** : connecteurs au sein de l’application hôte
* **Serveurs** : services qui fournissent du contexte et des capacités

Le MCP s’inspire en partie du
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), qui
standardise la manière d’ajouter la prise en charge des langages de programmation à l’échelle d’un écosystème complet d’outils de développement. De façon similaire, le MCP standardise la manière d’intégrer du contexte supplémentaire et des outils dans l’écosystème des applications d’IA.

<div id="features">
  ### Fonctionnalités
</div>

Les serveurs peuvent offrir aux clients les fonctionnalités suivantes :

* **Ressources** : Contexte et données, utilisables par l’utilisateur ou le modèle d’IA
* **Invites** : Modèles de messages et de workflows destinés aux utilisateurs
* **Outils** : Fonctions que le modèle d’IA peut exécuter

Les clients peuvent offrir aux serveurs les fonctionnalités suivantes :

* **Échantillonnage** : Comportements agentiques initiés par le serveur et interactions récursives avec des LLM
* **Racines** : Demandes initiées par le serveur concernant les périmètres d’URI ou de système de fichiers dans lesquels opérer
* **Élicitation** : Demandes initiées par le serveur pour obtenir des informations supplémentaires auprès des utilisateurs

<div id="additional-utilities">
  ### Utilitaires supplémentaires
</div>

* Configuration
* Suivi de la progression
* Annulation
* Signalement des erreurs
* Journalisation

<div id="security-and-trust-safety">
  ## Sécurité, confiance et sûreté
</div>

Le Protocole de contexte de modèle (MCP) offre des capacités puissantes grâce à un accès arbitraire aux données
et à des parcours d’exécution de code. Ce pouvoir s’accompagne de considérations majeures en matière de sécurité et de confiance
que tous les implémenteurs doivent aborder avec rigueur.

<div id="key-principles">
  ### Principes clés
</div>

1. **Consentement et contrôle de l’utilisateur**
   * Les utilisateurs doivent donner un consentement explicite et comprendre tout accès aux données et toutes les opérations
   * Les utilisateurs doivent garder le contrôle sur les données partagées et les actions effectuées
   * Les implémenteurs doivent fournir des interfaces claires pour examiner et autoriser les activités

2. **Confidentialité des données**
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’exposer des données utilisateur aux serveurs
   * Les hôtes ne doivent pas transmettre des données de ressources ailleurs sans le consentement de l’utilisateur
   * Les données des utilisateurs doivent être protégées par des contrôles d’accès appropriés

3. **Sécurité des Outils**
   * Les Outils correspondent à l’exécution de code arbitraire et doivent être traités avec une
     prudence adéquate.
     * En particulier, les descriptions du comportement d’un outil, telles que les annotations, doivent être
       considérées comme non fiables, sauf si elles proviennent d’un serveur de confiance.
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’invoquer un outil
   * Les utilisateurs doivent comprendre ce que fait chaque outil avant d’en autoriser l’utilisation

4. **Contrôles d’échantillonnage LLM**
   * Les utilisateurs doivent approuver explicitement toute demande d’échantillonnage LLM
   * Les utilisateurs doivent pouvoir contrôler :
     * Si un échantillonnage a lieu ou non
     * L’invite exacte qui sera envoyée
     * Les résultats que le serveur peut voir
   * Le protocole limite intentionnellement la visibilité du serveur sur les invites

<div id="implementation-guidelines">
  ### Directives de mise en œuvre
</div>

Bien que le MCP lui-même ne puisse pas imposer ces principes de sécurité au niveau du protocole,
les implémenteurs **DEVRAIENT** :

1. Intégrer des parcours de consentement et d’autorisation robustes dans leurs applications
2. Fournir une documentation claire des implications en matière de sécurité
3. Mettre en place des contrôles d’accès et des protections des données appropriés
4. Suivre les meilleures pratiques de sécurité dans leurs intégrations
5. Prendre en compte les implications en matière de confidentialité dans la conception de leurs fonctionnalités
