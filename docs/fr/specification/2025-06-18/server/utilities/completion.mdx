<div id="enable-section-numbers" />

<Info>**Révision du protocole** : 2025-06-18</Info>

Le Protocole de contexte de modèle (MCP) propose un moyen standardisé pour que les serveurs offrent des suggestions d’autocomplétion des arguments pour les Invites et les URI de Ressources. Cela permet des expériences riches, proches d’un IDE, où les utilisateurs reçoivent des suggestions contextuelles lors de la saisie des valeurs d’arguments.

<div id="user-interaction-model">
  ## Modèle d’interaction utilisateur
</div>

L’achèvement dans le MCP est conçu pour prendre en charge des expériences utilisateur interactives similaires à l’autocomplétion de code dans un IDE.

Par exemple, les applications peuvent afficher des suggestions d’achèvement dans un menu déroulant ou une fenêtre contextuelle au fur et à mesure que les utilisateurs saisissent du texte, avec la possibilité de filtrer et de sélectionner parmi les options disponibles.

Cependant, les implémentations sont libres d’exposer l’achèvement via n’importe quel modèle d’interface qui répond à leurs besoins&mdash;le protocole en lui-même n’impose aucun modèle d’interaction utilisateur spécifique.

<div id="capabilities">
  ## Capacités
</div>

Les serveurs qui prennent en charge les complétions **DOIVENT** déclarer la capacité `completions` :

```json
{
  "capabilities": {
    "completions": {}
  }
}
```

<div id="protocol-messages">
  ## Messages du protocole
</div>

<div id="requesting-completions">
  ### Demande de complétions
</div>

Pour obtenir des suggestions de complétion, les clients envoient une requête `completion/complete` indiquant
ce qui doit être complété via un type de référence :

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "completion/complete",
  "params": {
    "ref": {
      "type": "ref/prompt",
      "name": "code_review"
    },
    "argument": {
      "name": "language",
      "value": "py"
    }
  }
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "completion": {
      "values": ["python", "pytorch", "pyside"],
      "total": 10,
      "hasMore": true
    }
  }
}
```

Pour les Invites ou modèles d’URI avec plusieurs arguments, les clients doivent inclure les complétions précédentes dans l’objet `context.arguments` afin de fournir du contexte pour les requêtes suivantes.

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "completion/complete",
  "params": {
    "ref": {
      "type": "ref/prompt",
      "name": "code_review"
    },
    "argument": {
      "name": "framework",
      "value": "fla"
    },
    "context": {
      "arguments": {
        "language": "python"
      }
    }
  }
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "completion": {
      "values": ["flask"],
      "total": 1,
      "hasMore": false
    }
  }
}
```

<div id="reference-types">
  ### Types de références
</div>

Le protocole prend en charge deux types de références de complétion :

| Type           | Description                          | Exemple                                             |
| -------------- | ------------------------------------ | --------------------------------------------------- |
| `ref/prompt`   | Fait référence à une Invite par nom  | `{"type": "ref/prompt", "name": "code_review"}`     |
| `ref/resource` | Fait référence à l’URI d’une Ressource | `{"type": "ref/resource", "uri": "file:///{path}"}` |

<div id="completion-results">
  ### Résultats de complétion
</div>

Les serveurs renvoient un tableau de valeurs de complétion classées par pertinence, avec :

- Jusqu’à 100 éléments par réponse
- Nombre total de correspondances disponibles (facultatif)
- Booléen indiquant si d’autres résultats sont disponibles

<div id="message-flow">
  ## Flux des messages
</div>

```mermaid
sequenceDiagram
    participant Client
    participant Server

    Note over Client: L’utilisateur saisit un argument
    Client->>Server: completion/complete
    Server-->>Client: Suggestions de complétion

    Note over Client: L’utilisateur continue à saisir
    Client->>Server: completion/complete
    Server-->>Client: Suggestions affinées
```

<div id="data-types">
  ## Types de données
</div>

<div id="completerequest">
  ### CompleteRequest
</div>

- `ref`: Un `PromptReference` ou un `ResourceReference`
- `argument`: Objet contenant :
  - `name`: Nom de l’argument
  - `value`: Valeur actuelle
- `context`: Objet contenant :
  - `arguments`: Un mapping des noms d’arguments déjà résolus vers leurs valeurs.

<div id="completeresult">
  ### CompleteResult
</div>

- `completion`: Objet contenant :
  - `values`: Liste de suggestions (max. 100)
  - `total`: Nombre total de correspondances (optionnel)
  - `hasMore`: Indicateur de résultats supplémentaires

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les serveurs **DEVRAIENT** renvoyer des erreurs JSON-RPC standard pour les cas d’échec courants :

- Méthode introuvable : `-32601` (Fonctionnalité non prise en charge)
- Nom d’invite invalide : `-32602` (Paramètres invalides)
- Arguments requis manquants : `-32602` (Paramètres invalides)
- Erreurs internes : `-32603` (Erreur interne)

<div id="implementation-considerations">
  ## Considérations d’implémentation
</div>

1. Les serveurs **DEVRAIENT** :
   - Renvoyer des suggestions triées par pertinence
   - Mettre en œuvre une recherche floue lorsque c’est pertinent
   - Limiter le débit des requêtes de complétion
   - Valider toutes les entrées

2. Les clients **DEVRAIENT** :
   - Antirebondir (debouncer) les requêtes de complétion rapides
   - Mettre en cache les résultats de complétion lorsque c’est pertinent
   - Gérer avec souplesse les résultats manquants ou partiels

<div id="security">
  ## Sécurité
</div>

Les implémentations DOIVENT :

- Valider toutes les entrées de complétion
- Mettre en œuvre une limitation de débit appropriée
- Contrôler l’accès aux suggestions sensibles
- Prévenir les fuites d’informations liées aux complétions
