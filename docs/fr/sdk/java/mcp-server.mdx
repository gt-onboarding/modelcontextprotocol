<div id="server-features">
  ## Fonctionnalités du serveur
</div>

Le Serveur MCP est un composant essentiel de l’architecture du Protocole de contexte de modèle (MCP) qui fournit des outils, des ressources et des capacités aux clients. Il implémente la partie serveur du protocole et est responsable de :

* Exposer des outils que les clients peuvent découvrir et exécuter.
  Prendre en charge des schémas d’entrée et de sortie et renvoyer des contenus structurés et non structurés.
* Gérer des ressources avec des modèles d’accès basés sur des URI
* Fournir des modèles d’invites et traiter les demandes d’invite
* Prendre en charge la négociation de capacités avec les clients
* Implémenter des opérations de protocole côté serveur
* Gérer des connexions clientes concurrentes
* Fournir une journalisation structurée, un suivi de progression et des notifications
* Fournir des implémentations de transports serveur `STDIO`, `Streamable-HTTP` et `SSE` sans nécessiter de frameworks web externes.
* **En option**, dépendances de transport spécifiques à Spring `io.modelcontextprotocol.sdk:mcp-spring-webflux`, `io.modelcontextprotocol.sdk:mcp-spring-webmvc` pour les utilisateurs de [Spring AI](https://docs.spring.io/spring-ai/reference/api/mcp/mcp-server-boot-starter-docs.html).

<Tip>
  Cette [démo de démarrage rapide](/fr/quickstart/server), basée sur Spring AI MCP, vous montrera
  comment créer un serveur MCP.
</Tip>

Le serveur prend en charge des API synchrones et asynchrones, permettant une intégration flexible dans différents contextes applicatifs.

<Tabs>
  <Tab title="API synchrone">
    ```java
    // Create a server with custom configuration
    McpSyncServer syncServer = McpServer.sync(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .capabilities(ServerCapabilities.builder()
            .resources(false, true)  // Enable resource support
            .tools(true)             // Enable tool support
            .prompts(true)           // Enable prompt support
            .logging()               // Enable logging support
            .completions()           // Enable completions support
            .build())
        .build();

    // Register tools, resources, and prompts
    syncServer.addTool(syncToolSpecification);
    syncServer.addResource(syncResourceSpecification);
    syncServer.addPrompt(syncPromptSpecification);

    // Close the server when done
    syncServer.close();
    ```

    ### Préserver les thread-locals dans les gestionnaires

    `McpSyncServer` délègue l’exécution à un `McpAsyncServer` sous-jacent.
    L’exécution des gestionnaires pour les outils, les ressources, etc. peut ne pas se produire sur le thread appelant la méthode.
    Dans ce cas, les thread-locals sont perdus, ce qui peut rompre certaines fonctionnalités de frameworks reposant sur un travail lié au thread.
    Pour garantir que l’exécution se fasse sur le thread appelant et que les thread-locals restent disponibles, définissez `McpServer.sync(...).immediateExecution(true)`.

    <Note>
      Ceci ne concerne que les serveurs synchrones. Les serveurs asynchrones utilisent la pile réactive et ne doivent pas s’appuyer sur des thread-locals.
    </Note>
  </Tab>

  <Tab title="API asynchrone">
    ```java
    // Create an async server with custom configuration
    McpAsyncServer asyncServer = McpServer.async(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .capabilities(ServerCapabilities.builder()
            .resources(false, true)     // Enable resource support
            .tools(true)                // Enable tool support
            .prompts(true)              // Enable prompt support
            .logging()                  // Enable logging support
            .completions()              // Enable completions support
            .build())
        .build();

    // Register tools, resources, and prompts
    asyncServer.addTool(asyncToolSpecification)
        .doOnSuccess(v -> logger.info("Tool registered"))
        .subscribe();

    asyncServer.addResource(asyncResourceSpecification)
        .doOnSuccess(v -> logger.info("Resource registered"))
        .subscribe();

    asyncServer.addPrompt(asyncPromptSpecification)
        .doOnSuccess(v -> logger.info("Prompt registered"))
        .subscribe();

    // Close the server when done
    asyncServer.close()
        .doOnSuccess(v -> logger.info("Server closed"))
        .subscribe();
    ```
  </Tab>
</Tabs>

<div id="server-transport-providers">
  ## Fournisseurs de transport côté serveur
</div>

La couche de transport du SDK MCP est responsable de la gestion des communications entre clients et serveurs.
Elle propose différentes implémentations pour prendre en charge divers protocoles et modes de communication.
Le SDK inclut plusieurs implémentations de fournisseurs de transport intégrés :

<Tabs>
  <Tab title="STDIO">
    Créer un transport basé sur un processus :

    ```java
    StdioServerTransportProvider transportProvider = new StdioServerTransportProvider(new ObjectMapper());
    ```

    Assure la gestion bidirectionnelle des messages JSON-RPC via les flux d’entrée/sortie standard, avec traitement non bloquant des messages, sérialisation/désérialisation et prise en charge d’un arrêt en douceur.

    Fonctionnalités clés :

    <ul>
      <li>Communication bidirectionnelle via stdin/stdout</li>
      <li>Prise en charge de l’intégration basée sur les processus</li>
      <li>Mise en place et configuration simples</li>
      <li>Implémentation légère</li>
    </ul>
  </Tab>

  <Tab title="WebFlux">
    <p>Transport serveur SSE et HTTP diffusé en continu basé sur WebFlux. [Nécessite la dépendance `mcp-spring-webflux`](/fr/sdk/java/mcp-overview#dependencies).</p>

    <Tabs>
      <Tab title="HTTP diffusé en continu (WebFlux)">
        ```java
        @Configuration
        class McpConfig {
            @Bean
            WebFluxStreamableServerTransportProvider webFluxStreamableServerTransportProvider(ObjectMapper mapper) {
              	return this.mcpStreamableServerTransportProvider = WebFluxStreamableServerTransportProvider.builder()
                        .objectMapper(mapper)
                        .messageEndpoint("/mcp")
                        .build();
            }

            @Bean
            RouterFunction<?> mcpRouterFunction(WebFluxStreamableServerTransportProvider transportProvider) {
                return transportProvider.getRouterFunction();
            }
        }
        ```

        <p>Implémente la spécification du transport HTTP diffusé en continu et fournit :</p>

        <ul>
          <li>Communication bidirectionnelle via HTTP diffusé en continu</li>
          <li>Diffusion HTTP réactive avec WebFlux</li>
          <li>Connexions clientes concurrentes via des points de terminaison HTTP diffusé en continu</li>
          <li>Routage des messages et gestion des sessions</li>
          <li>Arrêt en douceur</li>
        </ul>

        <Note>
          L’implémentation actuelle ne prend pas en charge la reprise faute de stockage de session
        </Note>

        <Tip>
          Dans des environnements distribués avec plusieurs instances de Serveur MCP, un routage correct des messages est requis.
          Pour atteindre l’évolutivité sans surcharge d’infrastructure importante, envisagez l’implémentation de transport serveur « Stateless Streamable-HTTP ».
        </Tip>
      </Tab>

<div id="server-capabilities">
  ## Capacités du serveur
</div>

Le serveur peut être configuré avec diverses capacités :

```java
var capabilities = ServerCapabilities.builder()
    .resources(false, true)  // Prise en charge des ressources avec notifications de modifications de liste
    .tools(true)             // Prise en charge des outils avec notifications de modifications de liste
    .prompts(true)           // Prise en charge des invites avec notifications de modifications de liste
    .logging()               // Activer la journalisation (activée par défaut avec le niveau INFO)
    .build();
```

<div id="tool-specification">
  ### Spécification des Outils
</div>

Le Protocole de contexte de modèle (MCP) permet aux serveurs [d’exposer des Outils](/fr/specification/2024-11-05/server/tools/) qui peuvent être invoqués par des modèles de langage.
Le SDK Java permet d’implémenter des spécifications d’Outils avec leurs fonctions de gestion.
Les Outils permettent aux modèles d’IA d’effectuer des calculs, d’accéder à des API externes, d’interroger des bases de données et de manipuler des fichiers :

<Tabs>
  <Tab title="Sync">
    ```java
    // Spécification d’outil synchrone
    var schema = """
                {
                  "type" : "object",
                  "id" : "urn:jsonschema:Operation",
                  "properties" : {
                    "operation" : {
                      "type" : "string"
                    },
                    "a" : {
                      "type" : "number"
                    },
                    "b" : {
                      "type" : "number"
                    }
                  }
                }
                """;
    var syncToolSpecification = new McpServerFeatures.SyncToolSpecification(
        new Tool("calculator", "Basic calculator", schema),
        (exchange, arguments) -> {
            // Implémentation de l’outil
            return new CallToolResult(result, false);
        }
    );
    ```
  </Tab>

  <Tab title="Async">
    ```java
    // Spécification d’outil asynchrone
    var schema = """
                {
                  "type" : "object",
                  "id" : "urn:jsonschema:Operation",
                  "properties" : {
                    "operation" : {
                      "type" : "string"
                    },
                    "a" : {
                      "type" : "number"
                    },
                    "b" : {
                      "type" : "number"
                    }
                  }
                }
                """;
    var asyncToolSpecification = new McpServerFeatures.AsyncToolSpecification(
        new Tool("calculator", "Basic calculator", schema),
        (exchange, arguments) -> {
            // Implémentation de l’outil
            return Mono.just(new CallToolResult(result, false));
        }
    );
    ```
  </Tab>
</Tabs>

La spécification d’Outil comprend une définition d’Outil avec `name`, `description` et `parameter schema`, suivie d’un gestionnaire d’appels qui implémente la logique de l’Outil.
Le premier argument de la fonction est `McpAsyncServerExchange` pour l’interaction avec le client, et le second est une map des arguments de l’Outil.

<div id="resource-specification">
  ### Spécification de ressource
</div>

Spécification d’une ressource avec sa fonction de gestion.
Les ressources fournissent du contexte aux modèles d’IA en exposant des données telles que : contenu de fichiers, enregistrements de base de données, réponses d’API, informations système, état de l’application.
Exemple de spécification de ressource :

<Tabs>
  <Tab title="Sync">
    ```java
    // Spécification de ressource synchrone
    var syncResourceSpecification = new McpServerFeatures.SyncResourceSpecification(
        new Resource("custom://resource", "name", "description", "mime-type", null),
        (exchange, request) -> {
            // Implémentation de la lecture de la ressource
            return new ReadResourceResult(contents);
        }
    );
    ```
  </Tab>

  <Tab title="Async">
    ```java
    // Spécification de ressource asynchrone
    var asyncResourceSpecification = new McpServerFeatures.AsyncResourceSpecification(
        new Resource("custom://resource", "name", "description", "mime-type", null),
        (exchange, request) -> {
            // Implémentation de la lecture de la ressource
            return Mono.just(new ReadResourceResult(contents));
        }
    );
    ```
  </Tab>
</Tabs>

La spécification de ressource se compose de définitions de ressource et d’un gestionnaire de lecture de ressource.
La définition de la ressource inclut `name`, `description` et le `MIME type`.
Le premier argument de la fonction qui gère les requêtes de lecture de ressource est un `McpAsyncServerExchange` avec lequel le serveur peut interagir avec le client connecté.
Le second argument est un `McpSchema.ReadResourceRequest`.

<div id="prompt-specification">
  ### Spécification des invites
</div>

Dans le cadre des [fonctionnalités d’invites](/fr/specification/2024-11-05/server/prompts/), le MCP propose une méthode standardisée permettant aux serveurs d’exposer des modèles d’invite aux clients.
La Spécification des invites définit un modèle structuré pour les interactions avec les modèles d’IA, qui prend en charge un formatage cohérent des messages, la substitution de paramètres, l’injection de contexte, le formatage des réponses et la mise en gabarit des instructions.

<Tabs>
  <Tab title="Sync">
    ```java
    // Spécification d’invite synchrone
    var syncPromptSpecification = new McpServerFeatures.SyncPromptSpecification(
    ```

<div id="using-sampling-from-a-server">
  ### Utiliser l’échantillonnage depuis un serveur
</div>

Pour utiliser les [fonctionnalités d’échantillonnage](/fr/specification/2024-11-05/client/sampling/), vous avez besoin d’un client compatible qui le prend en charge.
Aucune configuration particulière du serveur n’est nécessaire, mais vérifiez la prise en charge de l’échantillonnage côté client avant d’envoyer des requêtes.
En savoir plus sur la [prise en charge de l’échantillonnage côté client](fr/./mcp-client#sampling-support).

Lorsqu’un client compatible se connecte à un serveur avec état, le serveur peut demander des générations de modèles de langue :

<Tabs>
  <Tab title="API synchrone">
    ```java
    // Create a server
    McpSyncServer server = McpServer.sync(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .build();

    // Define a tool that uses sampling
    var calculatorTool = new McpServerFeatures.SyncToolSpecification(
        new Tool("ai-calculator", "Performs calculations using AI", schema),
        (exchange, arguments) -> {
            // Check if client supports sampling
            if (exchange.getClientCapabilities().sampling() == null) {
                return new CallToolResult("Client does not support AI capabilities", false);
            }

            // Create a sampling request
            McpSchema.CreateMessageRequest request = McpSchema.CreateMessageRequest.builder()
                .messages(List.of(new McpSchema.SamplingMessage(McpSchema.Role.USER,
                    new McpSchema.TextContent("Calculate: " + arguments.get("expression")))
                .modelPreferences(McpSchema.ModelPreferences.builder()
                    .hints(List.of(
                        McpSchema.ModelHint.of("claude-3-sonnet"),
                        McpSchema.ModelHint.of("claude")
                    ))
                    .intelligencePriority(0.8)  // Prioritize intelligence
                    .speedPriority(0.5)         // Moderate speed importance
                    .build())
                .systemPrompt("You are a helpful calculator assistant. Provide only the numerical answer.")
                .maxTokens(100)
                .build();

            // Request sampling from the client
            McpSchema.CreateMessageResult result = exchange.createMessage(request);

            // Process the result
            String answer = result.content().text();
            return new CallToolResult(answer, false);
        }
    );

    // Add the tool to the server
    server.addTool(calculatorTool);
    ```
  </Tab>

  <Tab title="API asynchrone">
    ```java
    // Create a server
    McpAsyncServer server = McpServer.async(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .build();

    // Define a tool that uses sampling
    var calculatorTool = new McpServerFeatures.AsyncToolSpecification(
        new Tool("ai-calculator", "Performs calculations using AI", schema),
        (exchange, arguments) -> {
            // Check if client supports sampling
            if (exchange.getClientCapabilities().sampling() == null) {
                return Mono.just(new CallToolResult("Client does not support AI capabilities", false));
            }

            // Create a sampling request
            McpSchema.CreateMessageRequest request = McpSchema.CreateMessageRequest.builder()
                .content(new McpSchema.TextContent("Calculate: " + arguments.get("expression")))
                .modelPreferences(McpSchema.ModelPreferences.builder()
                    .hints(List.of(
                        McpSchema.ModelHint.of("claude-3-sonnet"),
                        McpSchema.ModelHint.of("claude")
                    ))
                    .intelligencePriority(0.8)  // Prioritize intelligence
                    .speedPriority(0.5)         // Moderate speed importance
                    .build())
                .systemPrompt("You are a helpful calculator assistant. Provide only the numerical answer.")
                .maxTokens(100)
                .build();

            // Request sampling from the client
            return exchange.createMessage(request)
                .map(result -> {
                    // Process the result
                    String answer = result.content().text();
                    return new CallToolResult(answer, false);
                });
        }
    );

    // Add the tool to the server
    server.addTool(calculatorTool)
        .subscribe();
    ```
  </Tab>
</Tabs>

L’objet `CreateMessageRequest` vous permet de spécifier : `Content` — le texte ou l’image d’entrée pour le modèle,
`Model Preferences` — des indications et des priorités pour la sélection du modèle, `System Prompt` — des instructions sur le comportement du modèle et
`Max Tokens` — la longueur maximale de la réponse générée.

<div id="logging-support">
  ### Prise en charge de la journalisation
</div>

Le serveur fournit des fonctionnalités de journalisation structurée permettant d’envoyer aux clients des messages de journal avec différents niveaux de gravité. Les notifications de journal ne peuvent être envoyées que dans le cadre d’une session client existante, par exemple lors d’appels d’outils, de ressources ou d’invites.

Par exemple, nous pouvons envoyer un message de journal depuis une fonction de gestionnaire d’outil.
Côté client, vous pouvez enregistrer un consommateur de journalisation pour recevoir les messages de journal du serveur et définir le niveau minimal de journalisation afin de filtrer les messages.

```java
var mcpClient = McpClient.sync(transport)
        .loggingConsumer(notification -> {
            System.out.println("Received log message: " + notification.data());
        })
        .build();

mcpClient.initialize();

mcpClient.setLoggingLevel(McpSchema.LoggingLevel.INFO);

// Call the tool that sends logging notifications
CallToolResult result = mcpClient.callTool(new McpSchema.CallToolRequest("logging-test", Map.of()));
```

Le serveur peut envoyer des messages de journal en utilisant l’objet `McpAsyncServerExchange`/`McpSyncServerExchange` dans la fonction du gestionnaire d’outil/ressource/invite :

```java
var tool = new McpServerFeatures.AsyncToolSpecification(
    new McpSchema.Tool("logging-test", "Test logging notifications", emptyJsonSchema),
    (exchange, request) -> {

      exchange.loggingNotification( // Utiliser l’exchange pour envoyer des messages de journal
          McpSchema.LoggingMessageNotification.builder()
            .level(McpSchema.LoggingLevel.DEBUG)
            .logger("test-logger")
            .data("Debug message")
            .build())
        .block();

      return Mono.just(new CallToolResult("Logging test completed", false));
    });

var mcpServer = McpServer.async(mcpServerTransportProvider)
  .serverInfo("test-server", "1.0.0")
  .capabilities(
    ServerCapabilities.builder()
      .logging() // Activer la prise en charge de la journalisation
      .tools(true)
      .build())
  .tools(tool)
  .build();
```

Les clients peuvent contrôler le niveau minimal de journalisation qu’ils reçoivent via la requête `mcpClient.setLoggingLevel(level)`. Les messages en dessous du niveau défini seront filtrés.
Niveaux de journalisation pris en charge (par ordre de gravité croissante) : DEBUG (0), INFO (1), NOTICE (2), WARNING (3), ERROR (4), CRITICAL (5), ALERT (6), EMERGENCY (7)

<div id="error-handling">
  ## Gestion des erreurs
</div>

Le SDK propose une gestion des erreurs complète via la classe McpError, couvrant la compatibilité du protocole, la communication de transport, la messagerie JSON-RPC, l’exécution des Outils, la gestion des Ressources, la gestion des Invites, les délais d’expiration et les problèmes de connexion. Cette approche unifiée garantit une gestion des erreurs cohérente et fiable pour les opérations synchrones comme asynchrones.
