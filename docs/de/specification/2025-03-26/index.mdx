[Model Context Protocol](https://modelcontextprotocol.io) (MCP) ist ein offenes Protokoll, das
eine nahtlose Integration zwischen LLM-Anwendungen und externen Datenquellen sowie
Werkzeugen ermöglicht. Ob Sie eine KI-gestützte IDE entwickeln, eine Chatoberfläche erweitern oder
benutzerdefinierte KI-Workflows erstellen – MCP bietet eine standardisierte Möglichkeit, LLMs mit dem Kontext zu verbinden,
den sie benötigen.

Diese Spezifikation definiert die maßgeblichen Protokollanforderungen, basierend auf dem
TypeScript-Schema in
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.ts).

Für Implementierungsleitfäden und Beispiele besuchen Sie
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Die Schlüsselwörter „MUST“, „MUST NOT“, „REQUIRED“, „SHALL“, „SHALL NOT“, „SHOULD“, „SHOULD
NOT“, „RECOMMENDED“, „NOT RECOMMENDED“, „MAY“ und „OPTIONAL“ in diesem Dokument sind so zu
interpretieren, wie in [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] beschrieben – und zwar nur dann, wenn sie
wie hier in Großbuchstaben erscheinen.

<div id="overview">
  ## Überblick
</div>

MCP bietet eine standardisierte Möglichkeit für Anwendungen, um:

* Kontextinformationen mit Sprachmodellen zu teilen
* Werkzeuge und Fähigkeiten für KI-Systeme bereitzustellen
* Komponierbare Integrationen und Workflows aufzubauen

Das Protokoll verwendet [JSON-RPC](https://www.jsonrpc.org/) 2.0-Nachrichten, um die
Kommunikation zwischen folgenden Komponenten herzustellen:

* **Hosts**: LLM-Anwendungen, die Verbindungen initiieren
* **Clients**: Connectoren innerhalb der Host-Anwendung
* **Servers**: Dienste, die Kontext und Fähigkeiten bereitstellen

MCP lässt sich vom
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/) inspirieren, das
standardisiert, wie in einem gesamten Ökosystem von
Entwicklungstools Unterstützung für Programmiersprachen bereitgestellt wird. In ähnlicher Weise standardisiert MCP, wie zusätzlicher Kontext
und Werkzeuge in das Ökosystem von KI-Anwendungen integriert werden.

<div id="key-details">
  ## Wichtige Details
</div>

<div id="base-protocol">
  ### Basisprotokoll
</div>

* Nachrichtenformat [JSON-RPC](https://www.jsonrpc.org/)
* Zustandsbehaftete Verbindungen
* Fähigkeitsaushandlung zwischen Server und Client

<div id="features">
  ### Funktionen
</div>

Server können Clients beliebige der folgenden Funktionen bereitstellen:

* **Ressourcen**: Kontext und Daten zur Nutzung durch den Benutzer oder das KI-Modell
* **Prompts**: Vorlagenbasierte Nachrichten und Workflows für Benutzer
* **Werkzeuge**: Funktionen, die das KI-Modell ausführen kann

Clients können Servern die folgende Funktion bereitstellen:

* **Sampling**: Vom Server initiierte agentische Verhaltensweisen und rekursive LLM-Interaktionen

<div id="additional-utilities">
  ### Zusätzliche Hilfsfunktionen
</div>

* Konfiguration
* Fortschrittsverfolgung
* Abbruch
* Fehlerberichterstattung
* Protokollierung

## Sicherheit und Trust &amp; Safety

Das Model Context Protocol ermöglicht leistungsfähige Funktionen durch beliebigen Datenzugriff und Wege zur Codeausführung. Mit dieser Stärke gehen wichtige Sicherheits- und Vertrauensaspekte einher, die alle Implementierenden sorgfältig berücksichtigen müssen.

<div id="key-principles">
  ### Zentrale Grundsätze
</div>

1. **Zustimmung und Kontrolle der Nutzer**
   * Nutzer müssen ausdrücklich allen Datenzugriffen und -operationen zustimmen und diese verstehen
   * Nutzer müssen die Kontrolle darüber behalten, welche Daten geteilt werden und welche Aktionen durchgeführt werden
   * Implementierende sollten klare UIs für die Prüfung und Autorisierung von Aktivitäten bereitstellen

2. **Datenschutz**
   * Hosts müssen die ausdrückliche Zustimmung der Nutzer einholen, bevor sie Nutzerdaten gegenüber Servern offenlegen
   * Hosts dürfen Ressourcendaten ohne Zustimmung der Nutzer nicht anderweitig übermitteln
   * Nutzerdaten sollten mit geeigneten Zugriffskontrollen geschützt werden

3. **Tool-Sicherheit**
   * Tools stehen für die Ausführung beliebigen Codes und müssen mit entsprechender
     Vorsicht behandelt werden.
     * Insbesondere sollten Beschreibungen des Tool-Verhaltens wie Tool-Anmerkungen als
       nicht vertrauenswürdig betrachtet werden, es sei denn, sie stammen von einem vertrauenswürdigen Server.
   * Hosts müssen die ausdrückliche Zustimmung der Nutzer einholen, bevor sie ein Tool aufrufen
   * Nutzer sollten verstehen, was jedes Tool tut, bevor sie dessen Verwendung autorisieren

4. **Kontrollen für LLM-Sampling**
   * Nutzer müssen alle LLM-Sampling-Anfragen ausdrücklich genehmigen
   * Nutzer sollten steuern:
     * Ob Sampling überhaupt stattfindet
     * Den tatsächlichen Prompt, der gesendet wird
     * Welche Ergebnisse der Server sehen kann
   * Das Protokoll beschränkt bewusst die Sichtbarkeit des Servers in Bezug auf Prompts

<div id="implementation-guidelines">
  ### Richtlinien für die Implementierung
</div>

Auch wenn MCP diese Sicherheitsprinzipien nicht auf Protokollebene durchsetzen kann,
sollten Implementiererinnen und Implementierer:

1. Robuste Einwilligungs- und Autorisierungsprozesse in ihre Anwendungen integrieren
2. Eine klare Dokumentation der Sicherheitsauswirkungen bereitstellen
3. Angemessene Zugriffskontrollen und Schutzmaßnahmen für Daten implementieren
4. Sicherheitsbezogene Best Practices in ihren Integrationen befolgen
5. Datenschutzaspekte in ihren Feature-Designs berücksichtigen
