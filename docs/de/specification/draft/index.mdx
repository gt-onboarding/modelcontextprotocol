<div id="enable-section-numbers" />

[Model Context Protocol](https://modelcontextprotocol.io) (MCP) ist ein offenes Protokoll, das
eine nahtlose Integration zwischen LLM-Anwendungen und externen Datenquellen sowie
Werkzeugen ermöglicht. Ob Sie eine KI-gestützte IDE entwickeln, eine Chat-Oberfläche erweitern oder
benutzerdefinierte KI-Workflows erstellen – MCP bietet eine standardisierte Möglichkeit, LLMs mit dem Kontext zu verbinden,
den sie benötigen.

Diese Spezifikation definiert die maßgeblichen Protokollanforderungen auf Grundlage des
TypeScript-Schemas in
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts).

Für Implementierungsleitfäden und Beispiele besuchen Sie
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Die Schlüsselwörter „MUST“, „MUST NOT“, „REQUIRED“, „SHALL“, „SHALL NOT“, „SHOULD“, „SHOULD
NOT“, „RECOMMENDED“, „NOT RECOMMENDED“, „MAY“ und „OPTIONAL“ in diesem Dokument sind so zu
interpretieren, wie in [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] beschrieben – und zwar nur dann,
wenn sie in Großbuchstaben erscheinen, wie hier gezeigt.

<div id="overview">
  ## Überblick
</div>

MCP bietet Anwendungen eine standardisierte Möglichkeit, um:

* Kontextinformationen mit Sprachmodellen zu teilen
* Werkzeuge und Fähigkeiten für KI-Systeme bereitzustellen
* Komponierbare Integrationen und Workflows zu erstellen

Das Protokoll verwendet [JSON-RPC](https://www.jsonrpc.org/) 2.0-Nachrichten, um die
Kommunikation zwischen folgenden Komponenten herzustellen:

* **Hosts**: LLM-Anwendungen, die Verbindungen initiieren
* **Clients**: Connectoren innerhalb der Host-Anwendung
* **Servers**: Dienste, die Kontext und Fähigkeiten bereitstellen

MCP ist teilweise vom
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/) inspiriert, das
standardisiert, wie Unterstützung für Programmiersprachen in einem gesamten Ökosystem von
Entwicklungswerkzeugen hinzugefügt wird. In ähnlicher Weise standardisiert MCP, wie zusätzlicher Kontext
und Werkzeuge in das Ökosystem von KI-Anwendungen integriert werden.

<div id="key-details">
  ## Wichtige Eckpunkte
</div>

<div id="base-protocol">
  ### Basisprotokoll
</div>

* [JSON-RPC](https://www.jsonrpc.org/)-Nachrichtenformat
* Zustandsbehaftete Verbindungen
* Fähigkeitsaushandlung zwischen Server und Client

<div id="features">
  ### Funktionen
</div>

Server bieten Clients beliebige der folgenden Funktionen an:

* **Ressourcen**: Kontext und Daten, die vom Benutzer oder dem KI-Modell genutzt werden können
* **Prompts**: Vorlagenbasierte Nachrichten und Workflows für Benutzer
* **Werkzeuge**: Funktionen, die das KI-Modell ausführen kann

Clients können Servern die folgenden Funktionen anbieten:

* **Sampling**: Vom Server initiierte agentische Verhaltensweisen und rekursive LLM-Interaktionen
* **Wurzeln**: Vom Server initiierte Erkundungen von URI- oder Dateisystemgrenzen als Arbeitskontext
* **Elicitierung**: Vom Server initiierte Anfragen nach zusätzlichen Informationen von Benutzern

<div id="additional-utilities">
  ### Zusätzliche Hilfsfunktionen
</div>

* Konfiguration
* Fortschrittsverfolgung
* Abbruch
* Fehlerberichterstattung
* Protokollierung

## Sicherheit und Trust &amp; Safety

Das Model Context Protocol ermöglicht leistungsstarke Funktionen durch beliebigen Datenzugriff
und Pfade zur Codeausführung. Mit dieser Leistungsfähigkeit gehen wichtige Sicherheits- und Vertrauensfragen einher,
die alle Implementierenden sorgfältig adressieren müssen.

<div id="key-principles">
  ### Zentrale Grundsätze
</div>

1. **Einwilligung und Kontrolle der Nutzer**
   * Nutzer müssen allen Datenzugriffen und Vorgängen ausdrücklich zustimmen und sie verstehen
   * Nutzer müssen die Kontrolle darüber behalten, welche Daten geteilt werden und welche Aktionen ausgeführt werden
   * Implementierer sollten klare UIs für die Prüfung und Autorisierung von Aktivitäten bereitstellen

2. **Datenschutz**
   * Hosts müssen die ausdrückliche Zustimmung der Nutzer einholen, bevor sie Nutzerdaten an Server offenlegen
   * Hosts dürfen Ressourcendaten ohne Zustimmung der Nutzer nicht anderweitig übermitteln
   * Nutzerdaten sollten mit geeigneten Zugriffskontrollen geschützt werden

3. **Werkzeugsicherheit**
   * Werkzeuge ermöglichen die Ausführung beliebigen Codes und müssen mit entsprechender
     Vorsicht behandelt werden.
     * Insbesondere sollten Beschreibungen des Werkzeugverhaltens, etwa Anmerkungen, als
       nicht vertrauenswürdig betrachtet werden, sofern sie nicht von einem vertrauenswürdigen Server stammen.
   * Hosts müssen die ausdrückliche Zustimmung der Nutzer einholen, bevor ein Werkzeug aufgerufen wird
   * Nutzer sollten verstehen, was jedes Werkzeug macht, bevor sie seine Verwendung autorisieren

4. **LLM-Sampling-Kontrollen**
   * Nutzer müssen alle LLM-Sampling-Anfragen ausdrücklich genehmigen
   * Nutzer sollten steuern:
     * Ob Sampling überhaupt erfolgt
     * Den tatsächlichen Prompt, der gesendet wird
     * Welche Ergebnisse der Server sehen kann
   * Das Protokoll begrenzt bewusst die Server-Sichtbarkeit auf Prompts

<div id="implementation-guidelines">
  ### Implementierungsrichtlinien
</div>

Auch wenn MCP diese Sicherheitsprinzipien nicht auf Protokollebene erzwingen kann,
sollten Implementierende:

1. Robuste Zustimmungs- und Autorisierungsprozesse in ihre Anwendungen integrieren
2. Klare Dokumentation zu Sicherheitsauswirkungen bereitstellen
3. Angemessene Zugriffskontrollen und Datenschutzmaßnahmen implementieren
4. Sicherheitsbewährte Verfahren in ihren Integrationen befolgen
5. Datenschutzimplikationen in ihren Funktionsentwürfen berücksichtigen
