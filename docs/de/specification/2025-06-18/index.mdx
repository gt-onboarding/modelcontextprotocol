[Model Context Protocol](https://modelcontextprotocol.io) (MCP) ist ein offenes Protokoll, das
eine nahtlose Integration zwischen LLM-Anwendungen sowie externen Datenquellen und
Werkzeugen ermöglicht. Ob Sie eine KI-gestützte IDE entwickeln, eine Chat-Oberfläche verbessern oder
benutzerdefinierte KI-Workflows erstellen – MCP bietet eine standardisierte Möglichkeit, LLMs mit dem Kontext
zu verbinden, den sie benötigen.

Diese Spezifikation definiert die maßgeblichen Protokollanforderungen auf Grundlage des
TypeScript-Schemas in
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts).

Implementierungsleitfäden und Beispiele finden Sie unter
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Die Schlüsselwörter „MUST“, „MUST NOT“, „REQUIRED“, „SHALL“, „SHALL NOT“, „SHOULD“, „SHOULD
NOT“, „RECOMMENDED“, „NOT RECOMMENDED“, „MAY“ und „OPTIONAL“ in diesem Dokument sind so zu
interpretieren, wie in [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] beschrieben – und nur dann –, wenn sie
in Großbuchstaben erscheinen, wie hier gezeigt.

<div id="overview">
  ## Übersicht
</div>

MCP bietet eine standardisierte Methode für Anwendungen, um:

* Kontextinformationen mit Sprachmodellen zu teilen
* Werkzeuge und Fähigkeiten für KI-Systeme bereitzustellen
* Komponierbare Integrationen und Workflows zu erstellen

Das Protokoll verwendet [JSON-RPC](https://www.jsonrpc.org/) 2.0-Nachrichten, um die
Kommunikation herzustellen zwischen:

* **Hosts**: LLM-Anwendungen, die Verbindungen initiieren
* **Clients**: Connectoren innerhalb der Host-Anwendung
* **Servers**: Dienste, die Kontext und Fähigkeiten bereitstellen

MCP lässt sich teilweise vom
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/) inspirieren, das
standardisiert, wie Unterstützung für Programmiersprachen in einem gesamten Ökosystem von
Entwicklungswerkzeugen hinzugefügt wird. In ähnlicher Weise standardisiert MCP, wie zusätzlicher Kontext
und Werkzeuge in das Ökosystem von KI-Anwendungen integriert werden.

<div id="key-details">
  ## Wichtige Eckdaten
</div>

<div id="base-protocol">
  ### Basisprotokoll
</div>

* Nachrichtenformat [JSON-RPC](https://www.jsonrpc.org/)
* Zustandsbehaftete Verbindungen
* Fähigkeitsaushandlung zwischen Client und Server

<div id="features">
  ### Funktionen
</div>

Server stellen Clients beliebige der folgenden Funktionen bereit:

* **Ressourcen**: Kontext und Daten, die der/die Nutzer:in oder das KI-Modell verwenden kann
* **Prompts**: Vorlagenbasierte Nachrichten und Workflows für Nutzer:innen
* **Werkzeuge**: Funktionen, die das KI-Modell ausführen kann

Clients können Servern die folgenden Funktionen bereitstellen:

* **Sampling**: Vom Server initiierte agentische Verhaltensweisen und rekursive LLM-Interaktionen
* **Wurzeln**: Vom Server initiierte Erkundungen der Grenzen von URI- oder Dateisystembereichen, innerhalb derer operiert werden soll
* **Elicitation**: Vom Server initiierte Anfragen nach zusätzlichen Informationen von Nutzer:innen

<div id="additional-utilities">
  ### Zusätzliche Hilfsfunktionen
</div>

* Konfiguration
* Fortschrittsverfolgung
* Abbruch
* Fehlerberichterstattung
* Protokollierung

## Sicherheit und Trust &amp; Safety

Das Model Context Protocol ermöglicht leistungsstarke Funktionen durch beliebigen Datenzugriff
und Pfade zur Codeausführung. Mit dieser Befähigung gehen wichtige Sicherheits- und Vertrauensfragen
einher, die alle Implementierenden sorgfältig adressieren müssen.

<div id="key-principles">
  ### Zentrale Prinzipien
</div>

1. **Einwilligung und Kontrolle der Nutzenden**
   * Nutzende müssen allen Datenzugriffen und Operationen ausdrücklich zustimmen und diese verstehen
   * Nutzende müssen die Kontrolle darüber behalten, welche Daten geteilt werden und welche Aktionen ausgeführt werden
   * Implementierende sollten klare UIs zur Überprüfung und Autorisierung von Aktivitäten bereitstellen

2. **Datenschutz**
   * Hosts müssen vor der Offenlegung von Nutzendendaten gegenüber Servern die ausdrückliche Einwilligung der Nutzenden einholen
   * Hosts dürfen Ressourcendaten ohne Einwilligung der Nutzenden nicht anderweitig übertragen
   * Nutzendendaten sollten mit geeigneten Zugriffskontrollen geschützt werden

3. **Sicherheit von Werkzeugen**
   * Werkzeuge repräsentieren die Ausführung beliebigen Codes und müssen mit entsprechender Vorsicht behandelt werden.
     * Insbesondere sollten Beschreibungen des Werkzeugverhaltens wie Anmerkungen als nicht vertrauenswürdig betrachtet werden, es sei denn, sie stammen von einem vertrauenswürdigen Server.
   * Hosts müssen vor dem Aufruf eines Werkzeugs die ausdrückliche Einwilligung der Nutzenden einholen
   * Nutzende sollten verstehen, was jedes Werkzeug tut, bevor sie seine Verwendung autorisieren

4. **Kontrollen für LLM-Sampling**
   * Nutzende müssen alle LLM-Sampling-Anfragen ausdrücklich genehmigen
   * Nutzende sollten steuern:
     * Ob Sampling überhaupt stattfindet
     * Den tatsächlichen Prompt, der gesendet wird
     * Welche Ergebnisse der Server sehen kann
   * Das Protokoll begrenzt bewusst die Sichtbarkeit des Servers in Bezug auf Prompts

<div id="implementation-guidelines">
  ### Implementierungsrichtlinien
</div>

Auch wenn MCP diese Sicherheitsprinzipien nicht auf Protokollebene erzwingen kann,
sollten Implementierende:

1. Robuste Einwilligungs- und Autorisierungsprozesse in ihre Anwendungen integrieren
2. Eine klare Dokumentation der Sicherheitsauswirkungen bereitstellen
3. Angemessene Zugriffskontrollen und Maßnahmen zum Schutz von Daten implementieren
4. Sicherheitsbezogene Best Practices in ihren Integrationen befolgen
5. Datenschutzaspekte in ihren Funktionsentwürfen berücksichtigen
