---
title: "资源"
description: "向 LLM 提供来自服务器的数据与内容"
---

资源是模型上下文协议（MCP）中的核心原语，允许服务器公开可由客户端读取的数据与内容，并在与 LLM 的交互中作为上下文使用。

<Note>
  资源被设计为由**应用控制**，即由客户端应用决定何时以及如何使用它们。
  不同的 MCP 客户端可能以不同方式处理资源。例如：

  * Claude Desktop 目前要求用户在使用前显式选择资源
  * 其他客户端可能基于启发式自动选择资源
  * 某些实现甚至可能允许 AI 模型自行决定使用哪些资源

  在实现资源支持时，MCP 服务器的作者应准备好应对上述任一交互模式。若需将数据自动提供给模型，服务器作者应改用**模型控制**的原语，例如[工具](zh/./tools)。
</Note>

<div id="overview">
  ## 概览
</div>

资源指的是 MCP 服务器希望向客户端提供的各种数据，可能包括：

* 文件内容
* 数据库记录
* API 响应
* 实时系统数据
* 截图和图像
* 日志文件
* 等等

每个资源都通过唯一的 URI 进行标识，且可以包含文本或二进制数据。

<div id="resource-uris">
  ## 资源 URI
</div>

资源通过符合以下格式的 URI 进行标识：

```
[protocol]://[host]/[path]
```

例如：

* `file:///home/user/documents/report.pdf`
* `postgres://database/customers/schema`
* `screen://localhost/display1`

协议与路径结构由 MCP 服务器的实现决定。服务器可以定义自有的 URI 自定义方案。

<div id="resource-types">
  ## 资源类型
</div>

资源可以包含两类内容：

<div id="text-resources">
  ### 文本资源
</div>

文本资源包含以 UTF-8 编码的文本数据，适用于：

* 源代码
* 配置文件
* 日志文件
* JSON/XML 数据
* 纯文本

<div id="binary-resources">
  ### 二进制资源
</div>

二进制资源包含以 Base64 编码的原始二进制数据，适用于：

* 图像
* PDF
* 音频文件
* 视频文件
* 其他非文本格式

<div id="resource-discovery">
  ## 资源发现
</div>

客户端可以通过两种主要方式发现可用资源：

<div id="direct-resources">
  ### 直接资源
</div>

服务器通过 `resources/list` 请求公开一组资源。每个资源包含：

```typescript
{
  uri: string;           // 资源的唯一标识符
  name: string;          // 易读名称
  description?: string;  // 可选的描述
  mimeType?: string;     // 可选的 MIME 类型
  size?: number;         // 可选的字节数
}
```

<div id="resource-templates">
  ### 资源模板
</div>

对于动态资源，服务器可以提供 [URI 模板](https://datatracker.ietf.org/doc/html/rfc6570)，供客户端用来构造有效的资源 URI：

```typescript
{
  uriTemplate: string;   // 符合 RFC 6570 的 URI 模板
  name: string;          // 此类型的可读名称
  description?: string;  // 可选描述
  mimeType?: string;     // 适用于所有匹配资源的可选 MIME 类型
}
```

<div id="reading-resources">
  ## 读取资源
</div>

要读取资源，客户端需要使用资源 URI 发起 `resources/read` 请求。

服务器会返回一个资源内容列表：

```typescript
{
  contents: [
    {
      uri: string;        // 资源的 URI
      mimeType?: string;  // 可选的 MIME 类型

      // 二选一：
      text?: string;      // 文本资源
      blob?: string;      // 二进制资源（base64 编码）
    }
  ]
}
```

<Tip>
  服务器可能在一次 `resources/read` 请求的响应中返回多个资源。例如，读取目录时，可用于返回目录内的文件列表。
</Tip>

<div id="resource-updates">
  ## 资源更新
</div>

MCP 通过两种机制为资源提供实时更新：

<div id="list-changes">
  ### 列表变更
</div>

当可用资源列表发生变化时，服务器可以通过 `notifications/resources/list_changed` 通知告知客户端。

<div id="content-changes">
  ### 内容变更
</div>

客户端可以订阅特定资源的更新：

1. 客户端发送 `resources/subscribe`，并附带资源 URI
2. 当该资源发生变更时，服务器发送 `notifications/resources/updated`
3. 客户端可使用 `resources/read` 获取最新内容
4. 客户端可使用 `resources/unsubscribe` 取消订阅

<div id="example-implementation">
  ## 示例实现
</div>

下面是在 MCP 服务器中实现资源支持的一个简单示例：

<CodeGroup>
  ```typescript TypeScript
  const server = new Server(
    {
      name: "example-server",
      version: "1.0.0",
    },
    {
      capabilities: {
        resources: {},
      },
    },
  );

  // 列出可用资源
  server.setRequestHandler(ListResourcesRequestSchema, async () => {
    return {
      resources: [
        {
          uri: "file:///logs/app.log",
          name: "应用日志",
          mimeType: "text/plain",
        },
      ],
    };
  });

  // 读取资源内容
  server.setRequestHandler(ReadResourceRequestSchema, async (request) => {
    const uri = request.params.uri;

    if (uri === "file:///logs/app.log") {
      const logContents = await readLogFile();
      return {
        contents: [
          {
            uri,
            mimeType: "text/plain",
            text: logContents,
          },
        ],
      };
    }

    throw new Error("未找到资源");
  });
  ```

  ```python Python
  app = Server("example-server")

  @app.list_resources()
  async def list_resources() -> list[types.Resource]:
      return [
          types.Resource(
              uri="file:///logs/app.log",
              name="应用日志",
              mimeType="text/plain"
          )
      ]

  @app.read_resource()
  async def read_resource(uri: AnyUrl) -> str:
      if str(uri) == "file:///logs/app.log":
          log_contents = await read_log_file()
          return log_contents

      raise ValueError("未找到资源")

  # 启动服务器
  async with stdio_server() as streams:
      await app.run(
          streams[0],
          streams[1],
          app.create_initialization_options()
      )
  ```
</CodeGroup>

<div id="best-practices">
  ## 最佳实践
</div>

在实现资源支持时：

1. 使用清晰且具描述性的资源名称和 URI
2. 提供有助于 LLM 理解的说明
3. 在已知时设置合适的 MIME 类型
4. 为动态内容实现资源模板
5. 对于频繁变化的资源使用订阅
6. 以清晰的错误信息优雅地处理错误
7. 针对大型资源列表考虑分页
8. 在合适的情况下缓存资源内容
9. 在处理前先验证 URI
10. 为你的自定义 URI 方案撰写文档

<div id="security-considerations">
  ## 安全注意事项
</div>

在暴露资源时：

* 验证所有资源 URI
* 实施适当的访问控制
* 规范化并校验文件路径，防止目录遍历
* 谨慎处理二进制数据
* 考虑为资源读取设置速率限制
* 审计资源访问
* 在传输过程中加密敏感数据
* 验证 MIME 类型
* 为长时间运行的读取操作设置超时
* 正确进行资源清理
