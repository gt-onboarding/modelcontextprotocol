---
title: 采样
---

<Info>**协议修订**：2025-03-26</Info>

模型上下文协议（MCP）提供了一种标准化方式，使服务器能够通过客户端向语言模型请求 LLM 采样（“completion”或“generation”）。该流程在让服务器利用 AI 能力的同时，确保客户端对模型的访问、选择和权限保持控制——且服务器无需提供 API 密钥。服务器可以发起基于文本、音频或图像的交互请求，并可选择在其提示模板中包含来自 MCP 服务器的上下文。

<div id="user-interaction-model">
  ## 用户交互模型
</div>

MCP 中的采样使服务器能够实现代理式行为，因为它允许在其他 MCP 服务器功能内以_嵌套_方式发起 LLM 调用。

各实现可根据自身需求，通过任意合适的界面范式提供采样功能——协议本身并不强制规定任何特定的用户交互模型。

<Warning>
  出于信任与安全及整体安全性的考虑，**应当**始终有人参与，并且能够拒绝采样请求。

  应用程序**应当**：

  * 提供便于直观审阅采样请求的界面
  * 允许用户在发送前查看并编辑提示模板
  * 在发送前呈现生成的响应供审阅
</Warning>

<div id="capabilities">
  ## 功能
</div>

支持采样的客户端在[初始化](/zh/specification/2025-03-26/basic/lifecycle#initialization)阶段**必须**声明 `sampling` 功能：

```json
{
  "capabilities": {
    "sampling": {}
  }
}
```

  ## 协议消息
</div>

<div id="creating-messages">
  ### 创建消息
</div>

要请求语言模型生成内容，服务器会发送一个 `sampling/createMessage` 请求：

**请求：**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "What is the capital of France?"
        }
      }
    ],
    "modelPreferences": {
      "hints": [
        {
          "name": "claude-3-sonnet"
        }
      ],
      "intelligencePriority": 0.8,
      "speedPriority": 0.5
    },
    "systemPrompt": "You are a helpful assistant.",
    "maxTokens": 100
  }
}
```

**响应：**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "role": "assistant",
    "content": {
      "type": "text",
      "text": "The capital of France is Paris."
    },
    "model": "claude-3-sonnet-20240307",
    "stopReason": "endTurn"
  }
}
```

<div id="message-flow">
  ## 消息流
</div>

```mermaid
sequenceDiagram
    participant Server
    participant Client
    participant User
    participant LLM

    Note over Server,Client: 服务器发起采样
    Server->>Client: sampling/createMessage

    Note over Client,User: 人在环路审阅
    Client->>User: 展示请求以供审批
    User-->>Client: 审阅并批准/修改

    Note over Client,LLM: 模型交互
    Client->>LLM: 转发已批准的请求
    LLM-->>Client: 返回生成结果

    Note over Client,User: 响应审阅
    Client->>User: 展示响应以供审批
    User-->>Client: 审阅并批准/修改

    Note over Server,Client: 完成请求
    Client-->>Server: 返回已批准的响应
```

<div id="model-preferences">
  ### 模型偏好
</div>

在 MCP 中进行模型选择需要谨慎的抽象处理，因为服务器和客户端可能使用不同的 AI 提供商，且各自的模型供给不同。服务器无法仅凭模型名称请求某个特定模型，因为客户端可能无法访问该精确模型，或更倾向于使用来自其他提供商的等效模型。

为此，MCP 实现了一个偏好系统，将抽象的能力优先级与可选的模型提示相结合：

<div id="capability-priorities">
  #### 能力优先级
</div>

服务器通过三个归一化的优先级值（0-1）来表达其需求：

* `costPriority`: 降低成本有多重要？值越高，越倾向选择更便宜的模型。
* `speedPriority`: 低延迟有多重要？值越高，越倾向选择更快速的模型。
* `intelligencePriority`: 高级能力有多重要？值越高，越倾向选择更强大的模型。

<div id="model-hints">
  #### 模型提示
</div>

虽然优先级有助于基于特性选择模型，`hints` 允许服务器
建议特定模型或模型系列：

* 提示按子串匹配处理，可灵活匹配模型名称
* 多个提示按偏好顺序依次评估
* 客户端**可以（MAY）**将提示映射为不同提供商的等效模型
* 提示仅供参考——最终由客户端选择模型

例如：

```json
{
  "hints": [
    { "name": "claude-3-sonnet" }, // Prefer Sonnet-class models
    { "name": "claude" } // Fall back to any Claude model
  ],
  "costPriority": 0.3, // Cost is less important
  "speedPriority": 0.8, // Speed is very important
  "intelligencePriority": 0.5 // Moderate capability needs
}
```

客户端会据此处理这些偏好，从其可用选项中选取合适的模型。比如，如果客户端无法访问 Claude 模型但可用 Gemini，它可能会基于类似能力将 sonnet 提示映射到 `gemini-1.5-pro`。

<div id="error-handling">
  ## 错误处理
</div>

客户端**应**针对常见失败情形返回错误：

错误示例：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -1,
    "message": "User rejected sampling request"
  }
}
```

<div id="security-considerations">
  ## 安全注意事项
</div>

1. 客户端**应**实现用户审批控制
2. 双方**应**验证消息内容
3. 客户端**应**遵循模型偏好提示
4. 客户端**应**实施限流
5. 双方**必须**妥善处理敏感数据
