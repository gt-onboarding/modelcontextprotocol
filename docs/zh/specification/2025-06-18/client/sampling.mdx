title: 采样

<div id="enable-section-numbers" />

<Info>**协议修订**：2025-06-18</Info>

模型上下文协议（MCP）为服务器通过客户端向语言模型请求 LLM 采样（“补全”或“生成”）提供了标准化方式。该流程在使服务器能够利用 AI 能力的同时，仍由客户端掌控模型的访问、选择和权限，无需在服务器端配置任何 API 密钥。服务器可发起基于文本、音频或图像的交互请求，并可选择在提示模板中包含来自 MCP 服务器的上下文。

<div id="user-interaction-model">
  ## 用户交互模型
</div>

MCP 中的采样允许服务器实现具备代理式行为，因为它使 LLM 调用可以在其他 MCP 服务器功能内以_嵌套_方式发生。

各实现可按自身需求，通过任何合适的界面模式来暴露采样——协议本身并不强制采用特定的用户交互模型。

<Warning>
  出于信任与安全及整体安全性的考虑，**应当**始终将人纳入流程，并赋予其拒绝采样请求的能力。

  应用程序**应当**：

  * 提供便于、直观地审阅采样请求的 UI
  * 允许用户在发送前查看并编辑提示模板
  * 在交付前呈现生成的响应以供审阅
</Warning>

<div id="capabilities">
  ## 功能
</div>

支持采样的客户端在[初始化](/zh/specification/2025-06-18/basic/lifecycle#initialization)时**必须**声明 `sampling` 功能：

```json
{
  "capabilities": {
    "sampling": {}
  }
}
```

  ## 协议消息
</div>

<div id="creating-messages">
  ### 创建消息
</div>

要请求语言模型生成内容，服务器会发送一个 `sampling/createMessage` 请求：

**请求：**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "What is the capital of France?"
        }
      }
    ],
    "modelPreferences": {
      "hints": [
        {
          "name": "claude-3-sonnet"
        }
      ],
      "intelligencePriority": 0.8,
      "speedPriority": 0.5
    },
    "systemPrompt": "You are a helpful assistant.",
    "maxTokens": 100
  }
}
```

**响应：**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "role": "assistant",
    "content": {
      "type": "text",
      "text": "The capital of France is Paris."
    },
    "model": "claude-3-sonnet-20240307",
    "stopReason": "endTurn"
  }
}
```

<div id="message-flow">
  ## 消息流
</div>

```mermaid
sequenceDiagram
    participant Server
    participant Client
    participant User
    participant LLM

    Note over Server,Client: 服务器发起采样
    Server->>Client: sampling/createMessage

    Note over Client,User: 人在回路审查
    Client->>User: 提交请求以供批准
    User-->>Client: 审查并批准/修改

    Note over Client,LLM: 模型交互
    Client->>LLM: 转发已批准的请求
    LLM-->>Client: 返回生成结果

    Note over Client,User: 响应审查
    Client->>User: 提交响应以供批准
    User-->>Client: 审查并批准/修改

    Note over Server,Client: 完成请求
    Client-->>Server: 返回已批准的响应
```

<div id="model-preferences">
  ### 模型偏好
</div>

在 MCP 中进行模型选择需要进行谨慎的抽象，因为服务器和客户端可能使用
不同的 AI 提供商，且其模型供应各不相同。服务器不能仅按名称请求
某个特定模型，因为客户端可能无法使用该精确模型，或者更倾向于使用
其他提供商的等效模型。

为了解决这一问题，MCP 实现了一个偏好系统，将抽象的能力
优先级与可选的模型提示相结合：

<div id="capability-priorities">
  #### 能力优先级
</div>

服务器通过三个归一化的优先级数值（0–1）来表达其需求：

* `costPriority`：降低成本的重要性。数值越高，越偏好更便宜的模型。
* `speedPriority`：低延迟的重要性。数值越高，越偏好更快的模型。
* `intelligencePriority`：高级能力的重要性。数值越高，越偏好更强大的模型。

<div id="model-hints">
  #### 模型提示
</div>

虽然优先级有助于基于特性选择模型，`hints` 允许服务器
建议具体的模型或模型家族：

* 提示被当作子串处理，可灵活匹配模型名称
* 多个提示按偏好顺序依次评估
* 客户端**MAY**将提示映射为不同提供商的等效模型
* 提示仅供参考——最终的模型选择由客户端决定

例如：

```json
{
  "hints": [
    { "name": "claude-3-sonnet" }, // Prefer Sonnet-class models
    { "name": "claude" } // Fall back to any Claude model
  ],
  "costPriority": 0.3, // Cost is less important
  "speedPriority": 0.8, // Speed is very important
  "intelligencePriority": 0.5 // Moderate capability needs
}
```

客户端会综合这些偏好，从其可用选项中选择合适的模型。比如，如果客户端无法使用 Claude 系列但可用 Gemini，它可能会基于相近能力将 Sonnet 的提示映射到 `gemini-1.5-pro`。

<div id="error-handling">
  ## 错误处理
</div>

客户端**应**在常见失败情况下返回错误：

示例错误：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -1,
    "message": "User rejected sampling request"
  }
}
```

<div id="security-considerations">
  ## 安全注意事项
</div>

1. 客户端**应**实现用户批准控制
2. 双方**应**验证消息内容
3. 客户端**应**遵循模型偏好提示
4. 客户端**应**实施限流
5. 双方**必须**妥善处理敏感数据
