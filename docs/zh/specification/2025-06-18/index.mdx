<div id="enable-section-numbers" />

[Model Context Protocol](https://modelcontextprotocol.io)（MCP）是一种开放协议，
可在 LLM 应用与外部数据源和工具之间实现无缝集成。无论你是在构建 AI 驱动的 IDE、增强聊天界面，还是创建
自定义 AI 工作流，MCP 都提供了一种标准化方式，将 LLM 与其所需的上下文连接起来。

本规范基于 TypeScript 架构
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts)，
定义了权威的协议要求。

有关实现指南和示例，请访问
[modelcontextprotocol.io](https://modelcontextprotocol.io)。

本文档中的关键字“MUST”“MUST NOT”“REQUIRED”“SHALL”“SHALL NOT”“SHOULD”“SHOULD
NOT”“RECOMMENDED”“NOT RECOMMENDED”“MAY”“OPTIONAL”应按照
[BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)]中的说明进行解释，且仅当它们以此处所示的全大写形式出现时才适用。

<div id="overview">
  ## 概览
</div>

MCP 为应用程序提供了一种标准化方式，以：

* 与语言模型共享上下文信息
* 向 AI 系统暴露工具和能力
* 构建可组合的集成与工作流

该协议使用 [JSON-RPC](https://www.jsonrpc.org/) 2.0 消息在以下各方之间建立通信：

* **主机**：发起连接的 LLM 应用
* **客户端**：主机应用内的连接器
* **服务器**：提供上下文与能力的服务

MCP 受到
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/) 的启发。后者在整个开发工具生态中规范了如何为各种编程语言提供支持。类似地，MCP 规范了如何将额外的上下文与工具集成到 AI 应用生态中。

<div id="key-details">
  ## 关键信息
</div>

<div id="base-protocol">
  ### 基础协议
</div>

* [JSON-RPC](https://www.jsonrpc.org/) 消息格式
* 保持会话状态的连接
* 客户端与服务器的能力协商

<div id="features">
  ### 功能
</div>

服务器可以向客户端提供以下功能：

* **资源（Resources）**：供用户或 AI 模型使用的上下文与数据
* **提示模板（Prompts）**：面向用户的参数化消息与工作流
* **工具（Tools）**：供 AI 模型调用的函数

客户端可以向服务器提供以下功能：

* **采样（Sampling）**：由服务器发起的代理式行为与递归式 LLM 交互
* **根路径（Roots）**：由服务器发起的关于 URI 或文件系统边界的操作范围查询
* **信息征询（Elicitation）**：由服务器发起的向用户请求补充信息

<div id="additional-utilities">
  ### 其他辅助功能
</div>

* 配置
* 进度追踪
* 取消
* 错误报告
* 日志记录

<div id="security-and-trust-safety">
  ## 安全与信任与安全
</div>

模型上下文协议（MCP）通过对任意数据的访问和代码执行路径，带来了强大的能力。与此同时，也引出了所有实现者都必须谨慎应对的重要安全与信任方面的考量。

<div id="key-principles">
  ### 关键原则
</div>

1. **用户同意与控制**
   * 用户必须明确同意并理解所有数据访问与操作
   * 用户必须始终掌控共享哪些数据以及将执行哪些操作
   * 实施方应提供清晰的界面，用于审阅与授权各项活动

2. **数据隐私**
   * 在向服务器暴露用户数据之前，MCP 主机必须获得用户的明确同意
   * 未经用户同意，MCP 主机不得将资源数据传输到其他位置
   * 应通过适当的访问控制来保护用户数据

3. **工具安全**
   * 工具意味着可执行任意代码，必须谨慎对待。
     * 尤其是，对工具行为的描述（如注解）应视为不可信，除非来自可信的 MCP 服务器。
   * 在调用任何工具之前，MCP 主机必须获得用户的明确同意
   * 用户在授权使用前应了解每个工具的具体功能

4. **LLM 采样控制**
   * 用户必须明确批准任何 LLM 采样请求
   * 用户应能控制：
     * 是否进行采样
     * 将要发送的实际提示
     * 服务器可以看到的结果
   * 该协议有意限制服务器对提示的可见性

<div id="implementation-guidelines">
  ### 实施指南
</div>

尽管 MCP 本身无法在协议层面强制执行这些安全原则，
实现方**应当**：

1. 在应用中构建健全的同意与授权流程
2. 提供清晰的安全影响说明文档
3. 实施适当的访问控制和数据保护措施
4. 在集成中遵循安全最佳实践
5. 在功能设计中充分考虑隐私影响
