<div id="enable-section-numbers" />

[模型上下文协议（MCP）](https://modelcontextprotocol.io) 是一种开放协议，
用于在 LLM 应用与外部数据源和工具之间实现无缝集成。无论你是在构建 AI 驱动的
IDE、增强聊天界面，还是创建自定义 AI 工作流，MCP 都提供标准化方式，将
LLM 与其所需的上下文连接起来。

本规范基于 TypeScript 架构
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts)，
定义了权威的协议要求。

如需实现指南和示例，请访问
[modelcontextprotocol.io](https://modelcontextprotocol.io)。

本文件中的关键词“MUST”“MUST NOT”“REQUIRED”“SHALL”“SHALL NOT”“SHOULD”“SHOULD
NOT”“RECOMMENDED”“NOT RECOMMENDED”“MAY”“OPTIONAL”应按
[BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] 的说明进行解释，且仅当如本文所示以全大写形式出现时适用。

<div id="overview">
  ## 概览
</div>

MCP 为应用程序提供了一种标准化方式，用于：

* 与语言模型共享上下文信息
* 向 AI 系统公开工具与能力
* 构建可组合的集成与工作流

该协议使用 [JSON-RPC](https://www.jsonrpc.org/) 2.0 消息在以下组件之间建立通信：

* **主机（MCP 主机）**：发起连接的 LLM 应用
* **客户端（MCP 客户端）**：主机应用内的连接器
* **服务器（MCP 服务器）**：提供上下文与能力的服务

MCP 借鉴了
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/)，该协议
标准化了如何在整个开发工具生态中为编程语言添加支持。以类似方式，MCP 标准化了如何将额外的上下文
与工具集成到 AI 应用生态中。

<div id="key-details">
  ## 关键信息
</div>

<div id="base-protocol">
  ### 基础协议
</div>

* [JSON-RPC](https://www.jsonrpc.org/) 消息格式
* 有状态连接
* 服务端与客户端的能力协商

<div id="features">
  ### 功能
</div>

服务器可向客户端提供以下功能：

* **资源**：供用户或 AI 模型使用的上下文与数据
* **提示模板**：面向用户的可复用消息与工作流模板
* **工具**：供 AI 模型调用执行的功能

客户端可向服务器提供以下功能：

* **采样**：由服务器发起的智能体式行为与递归 LLM 交互
* **根路径**：由服务器发起、用于确定其可操作范围的 URI 或文件系统边界查询
* **信息征询**：由服务器发起、向用户请求补充信息

<div id="additional-utilities">
  ### 其他辅助功能
</div>

* 配置
* 进度跟踪
* 取消
* 错误报告
* 日志记录

<div id="security-and-trust-safety">
  ## 安全与信任
</div>

模型上下文协议（MCP）通过对任意数据的访问和代码执行路径提供强大功能。能力越大，越需要重视安全与信任方面的考量，所有实现者都必须谨慎应对。

<div id="key-principles">
  ### 关键原则
</div>

1. **用户同意与控制**
   * 用户必须对所有数据访问与操作明确同意并充分理解
   * 用户必须始终掌控共享哪些数据以及将执行哪些操作
   * 实施者应提供清晰的界面用于审查与授权各项活动

2. **数据隐私**
   * MCP 主机在向 MCP 服务器暴露用户数据之前必须获得用户的明确同意
   * MCP 主机未经用户同意不得将资源数据传输至其他位置
   * 应通过适当的访问控制来保护用户数据

3. **工具安全**
   * 工具意味着可执行任意代码，必须以适当的谨慎对待。
     * 尤其是，对工具行为的描述（如注释）应视为不可信，除非来自可信的服务器。
   * MCP 主机在调用任何工具之前必须获得用户的明确同意
   * 用户在授权使用之前应了解每个工具的功能与影响

4. **LLM 采样控制**
   * 任何 LLM 采样请求都必须由用户明确批准
   * 用户应能控制：
     * 是否进行采样
     * 将要发送的实际提示
     * 服务器可以看到的结果
   * 该协议有意限制服务器对提示的可见性

<div id="implementation-guidelines">
  ### 实施指南
</div>

尽管 MCP 本身无法在协议层面强制执行这些安全原则，
实现者**应**：

1. 在应用中构建健全的同意与授权流程
2. 清晰说明相关的安全影响并提供文档
3. 实施适当的访问控制与数据保护
4. 在集成中遵循安全最佳实践
5. 在功能设计时考虑隐私影响
