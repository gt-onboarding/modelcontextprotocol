<div id="enable-section-numbers" />

模型上下文协议（MCP）采用客户端—主机—服务器架构，其中每个 MCP 主机可运行多个 MCP 客户端实例。该架构使用户能够在各类应用中集成 AI 能力，同时保持清晰的安全边界并实现关注点隔离。基于 JSON-RPC 2.0，MCP 提供了一个有状态的会话协议，专注于在客户端与服务器之间进行上下文交换与采样协调。

<div id="host">
  ### 主机
</div>

主机进程充当容器和协调者：

* 创建并管理多个客户端实例
* 控制客户端的连接权限与生命周期
* 执行安全策略与用户同意要求
* 处理用户授权决策
* 协调 AI/LLM 集成与采样
* 跨客户端汇聚与管理上下文

<div id="clients">
  ### 客户端
</div>

每个客户端由主机创建，并与服务器保持一条相互隔离的连接：

* 为每个服务器建立一个有状态会话
* 处理协议协商与能力交换
* 双向路由协议消息
* 管理订阅与通知
* 在不同服务器之间维持安全边界

主机应用会创建并管理多个客户端，每个客户端与特定服务器保持 1:1 的对应关系。

<div id="servers">
  ### 服务器
</div>

服务器提供专门的上下文与能力：

* 通过 MCP 基元公开资源、工具和提示模板
* 独立运行，职责聚焦
* 通过客户端接口请求采样
* 必须遵守安全约束
* 可以是本地进程或远程服务

<div id="design-principles">
  ## 设计原则
</div>

MCP 构建于若干关键设计原则之上，这些原则指导其架构与实现：

1. **服务器应当极易构建**
   * 主机应用负责处理复杂的编排职责
   * 服务器专注于具体且定义清晰的能力
   * 简洁接口将实现开销降至最低
   * 明确分层有助于代码可维护

2. **服务器应当高度可组合**
   * 每个服务器在隔离状态下提供聚焦功能
   * 可将多个服务器无缝组合
   * 共享协议实现互操作性
   * 模块化设计支持扩展性

3. **服务器不应读取完整对话，也不应“窥视”其他服务器**
   * 服务器仅接收必要的上下文信息
   * 完整对话历史保留在主机端
   * 每个服务器连接保持隔离
   * 跨服务器交互由主机控制
   * 主机进程强制执行安全边界

4. **可在服务器与客户端上渐进式添加功能**
   * 核心协议提供最低限度的必要功能
   * 额外能力可按需协商启用
   * 服务器与客户端可独立演进
   * 协议为未来的可扩展性而设计
   * 保持向后兼容

<div id="capability-negotiation">
  ## 能力协商
</div>

模型上下文协议（MCP）采用基于能力的协商机制，客户端和服务器会在初始化阶段明确声明各自支持的功能。能力决定会话期间可用的协议特性与原语。

* 服务器声明的能力包括资源订阅、工具支持和提示模板等
* 客户端声明的能力包括采样支持和通知处理等
* 双方在整个会话期间必须遵守已声明的能力
* 还可通过协议扩展协商额外能力

```mermaid
sequenceDiagram
    participant Host
    participant Client
    participant Server

    Host->>+Client: Initialize client
    Client->>+Server: Initialize session with capabilities
    Server-->>Client: Respond with supported capabilities

    Note over Host,Server: Active Session with Negotiated Features

    loop Client Requests
        Host->>Client: User- or model-initiated action
        Client->>Server: Request (tools/resources)
        Server-->>Client: Response
        Client-->>Host: Update UI or respond to model
    end

    loop Server Requests
        Server->>Client: Request (sampling)
        Client->>Host: Forward to AI
        Host-->>Client: AI response
        Client-->>Server: Response
    end

    loop Notifications
        Server--)Client: Resource updates
        Client--)Server: Status changes
    end

    Host->>Client: Terminate
    Client->>-Server: End session
    deactivate Server
```

每项能力都会在会话期间启用对应的协议功能。例如：

* 已实现的[服务器功能](/zh/specification/draft/server)必须在服务器的能力中予以声明
* 发送资源订阅通知要求服务器声明对订阅的支持
* 调用工具要求服务器声明工具相关能力
* [采样](/zh/specification/draft/client)要求客户端在其能力中声明支持

能力协商确保在保持协议可扩展性的同时，客户端与服务器对受支持功能有清晰一致的共识。
