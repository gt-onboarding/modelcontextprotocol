本文档阐述如何在模型上下文协议（MCP）项目中开展沟通与协作。

<div id="communication-channels">
  ## 沟通渠道
</div>

简而言之：

* **[Discord][discord-join]**：用于实时或临时讨论。
* **[GitHub Discussions](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions)**：用于结构化、深入的讨论。
* **[GitHub Issues](https://github.com/modelcontextprotocol/modelcontextprotocol/issues)**：用于可执行的任务、缺陷报告和功能请求。
* **涉及安全敏感问题**：请遵循 [SECURITY.md](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/SECURITY.md) 中的流程。

所有沟通均受我们的 [行为准则](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/CODE_OF_CONDUCT.md) 约束。我们期望所有参与者在各个渠道中保持尊重、专业且包容的交流。

<div id="discord">
  ### Discord
</div>

用于贡献者的实时讨论与协作。该服务器专为**MCP 贡献者**设计，并非用于提供一般性的 MCP 支持。

Discord 服务器将同时设有公共和私有频道。

[点击此处加入 Discord 服务器][discord-join]。

<div id="public-channels-default">
  #### 公共频道（默认）
</div>

* **目的**：开放的社区参与、协作开发与透明的项目协调。
* 主要使用场景：
  * **公共 SDK 与工具开发**：从构思到发布规划的所有开发都在公共频道进行（如 `#typescript-sdk-dev`、`#inspector-dev`）。
  * **工作组与兴趣组讨论**（`#client-implementors`、`#agents-wg` 等）
    * **工作组**：围绕明确的目标或项目（如某个 SDK、inspector、registry、server-identity、load-balancing 等）。
    * **兴趣组**：更为松散的交流群体，可能讨论多种话题。部分议题会一次性处理，另一些可能演变为工作组。
  * **社区入门**与贡献指引。
  * **社区反馈**与协作式头脑风暴。
  * 公开的**office hours**与**维护者在线时段**。
* 避免：
  * MCP 用户支持：参与者应先查阅官方文档，并在 GitHub Discussions 发起新讨论以提出问题或寻求支持。
  * 服务或产品营销：本 Discord 的交流应保持供应商中立，不用于品牌建设或销售。除用于举例或回应以规范为中心的讨论外，不鼓励提及品牌或产品。

<div id="private-channels-exceptions">
  #### 私有频道（例外）
</div>

* **目的**：用于无法公开讨论的保密协调与敏感事项。仅指定的维护者可访问。
* **严格的私用标准**：
  * **安全事件**（CVE、协议漏洞）。
  * **人员相关事项**（维护者相关讨论、行为准则政策）。
  * 部分频道将配置为**只读**。这在维护者作出决策时很有帮助。
  * 需要在小范围内获得**即时**或**集中响应**的协调。
* **透明度**：
  * 影响社区的**所有技术与治理决策**都**必须**在 GitHub Discussions 和/或 Issues 中记录，并标注 `notes`。
  * **与个别贡献者相关的某些事项**在适当情况下可保持私密（如个人情况、纪律处分或其他敏感个人事项）。
  * 私有频道应仅作为**临时“事件处理室”**使用，而非日常开发之用。

任何在 Discord 上的重大讨论，如将导致潜在的决定或提案，必须转移至 GitHub Discussion 或 GitHub Issue，以创建可持久、可搜索的记录。随后将视需要把提案提升为完整的 PR，并附带相关工作项（GitHub Issues）。

<div id="github-discussions">
  ### GitHub Discussions
</div>

适用于就项目方向、功能、改进以及社区议题开展结构化、长篇讨论与辩论。

使用场景：

* 项目路线图规划与里程碑讨论
* 公告与版本发布沟通
* 社区投票与共识建立流程
* 具备上下文与论证的功能请求
  * 如果某个仓库未启用 GitHub Discussions，可改为提交 GitHub Issue。

<div id="github-issues">
  ### GitHub Issues
</div>

用于提交缺陷报告、功能跟踪，以及可执行的开发任务。

适用场景：

* 提交 SEP 提案（遵循[SEP 指南](zh/./sep-guidelines)）
* 带有可复现步骤的缺陷报告
* 明确范围的文档改进
* CI/CD 相关问题与基础设施故障
* 发布任务与里程碑跟踪

<div id="security-issues">
  ### 安全问题
</div>

**请勿公开发布安全问题。** 请改为：

1. 使用私密的安全报告流程。对于协议层面的安全问题，请遵循 [modelcontextprotocol GitHub 仓库中的 SECURITY.md](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/SECURITY.md) 所述流程。
2. 直接联系项目负责人和/或[核心维护者](zh/./governance#current-core-maintainers)。
3. 遵循负责任披露准则。

<div id="decision-records">
  ## 决策记录
</div>

所有 MCP 决策都会记录并发布在公开渠道。

* **技术决策**：[GitHub Issues](https://github.com/modelcontextprotocol/modelcontextprotocol/issues) 和 SEP。
* **规范变更**：[Model Context Protocol 网站](https://modelcontextprotocol.io/specification/draft/changelog)。
* **流程变更**：[社区文档](https://modelcontextprotocol.io/community/governance)。
* **治理决策与更新**：[GitHub Issues](https://github.com/modelcontextprotocol/modelcontextprotocol/issues) 和 SEP。

在记录决策时，我们将尽可能保留完整上下文：

* 决策者
* 背景与动机
* 备选方案
* 选择该做法的理由
* 实施步骤
