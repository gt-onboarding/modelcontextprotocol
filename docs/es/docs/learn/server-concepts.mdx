title: Comprender los servidores MCP
sidebarTitle: Servidores

Los servidores MCP son programas que exponen capacidades específicas a las aplicaciones de IA mediante interfaces de protocolo estandarizadas.

Algunos ejemplos comunes incluyen servidores de sistema de archivos para el acceso a documentos, servidores de bases de datos para consultas de datos, servidores de GitHub para la gestión de código, servidores de Slack para la comunicación de equipos y servidores de calendario para la programación.

<div id="core-server-features">
  ## Funciones principales del servidor
</div>

Los servidores ofrecen funcionalidad mediante tres componentes básicos:

| Función       | Explicación                                                                                                                                                                               | Ejemplos                                                            | Quién lo controla |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ----------------- |
| **Herramientas**     | Funciones que tu LLM puede invocar activamente y decide cuándo usar según las solicitudes del usuario. Las Herramientas pueden escribir en bases de datos, llamar a APIs externas, modificar archivos o activar otra lógica. | Buscar vuelos<br />Enviar mensajes<br />Crear eventos de calendario | Modelo            |
| **Recursos** | Fuentes de datos pasivas que brindan acceso de solo lectura a información para proporcionar contexto, como contenido de archivos, esquemas de bases de datos o documentación de APIs.     | Recuperar documentos<br />Acceder a bases de conocimiento<br />Leer calendarios | Aplicación        |
| **Indicaciones**   | Plantillas de instrucciones predefinidas que indican al modelo cómo trabajar con Herramientas y Recursos específicos.                                                                     | Planificar unas vacaciones<br />Resumir mis reuniones<br />Redactar un correo | Usuario           |

Usaremos un escenario hipotético para demostrar el papel de cada una de estas funciones y mostrar cómo pueden trabajar juntas.

<div id="tools">
  ### Herramientas
</div>

Las herramientas permiten que los modelos de IA realicen acciones. Cada herramienta define una operación específica con entradas y salidas tipadas. El modelo solicita la ejecución de la herramienta según el contexto.

<div id="how-tools-work">
  #### Cómo funcionan las Herramientas
</div>

Las Herramientas son interfaces definidas por esquemas que los LLM pueden invocar. MCP utiliza JSON Schema para la validación. Cada herramienta realiza una única operación con entradas y salidas claramente definidas. Las Herramientas pueden requerir el consentimiento del usuario antes de ejecutarse, lo que ayuda a garantizar que las personas mantengan el control sobre las acciones que realiza un modelo.

**Operaciones del protocolo:**

| Method       | Purpose                           | Returns                                       |
| ------------ | --------------------------------- | --------------------------------------------- |
| `tools/list` | Descubrir herramientas disponibles | Lista de definiciones de herramientas con sus esquemas |
| `tools/call` | Ejecutar una herramienta específica | Resultado de la ejecución de la herramienta   |

**Ejemplo de definición de herramienta:**

```typescript
{
  name: "searchFlights",
  description: "Buscar vuelos disponibles",
  inputSchema: {
    type: "object",
    properties: {
      origin: { type: "string", description: "Ciudad de salida" },
      destination: { type: "string", description: "Ciudad de llegada" },
      date: { type: "string", format: "date", description: "Fecha de viaje" }
    },
    required: ["origin", "destination", "date"]
  }
}
```

<div id="example-travel-booking">
  #### Ejemplo: Reserva de viajes
</div>

Las Herramientas permiten que las aplicaciones de IA realicen acciones en nombre de los usuarios. En un escenario de planificación de viajes, la aplicación de IA podría usar varias Herramientas para ayudar a reservar unas vacaciones:

**Búsqueda de vuelos**

```
searchFlights(origin: "NYC", destination: "Barcelona", date: "2024-06-15")
```

Consulta varias aerolíneas y devuelve opciones de vuelo estructuradas.

**Bloqueo en el calendario**

```
createCalendarEvent(title: "Barcelona Trip", startDate: "2024-06-15", endDate: "2024-06-22")
```

Bloquea las fechas del viaje en el calendario del usuario.

**Notificación por correo electrónico**

```
sendEmail(to: "team@work.com", subject: "Out of Office", body: "...")
```

Envía un mensaje automático de “fuera de la oficina” a los colegas.

<div id="user-interaction-model">
  #### Modelo de interacción con el usuario
</div>

Las Herramientas están controladas por el modelo, lo que significa que los modelos de IA pueden descubrirlas e invocarlas automáticamente. Sin embargo, el MCP hace hincapié en la supervisión humana mediante varios mecanismos.

Por motivos de confianza y seguridad, las aplicaciones pueden implementar control del usuario a través de diversos mecanismos, como:

- Mostrar las Herramientas disponibles en la interfaz de usuario, permitiendo que los usuarios definan si una Herramienta debe estar disponible en interacciones específicas
- Diálogos de aprobación para ejecuciones individuales de Herramientas
- Configuraciones de permisos para preaprobar ciertas operaciones seguras
- Registros de actividad que muestren todas las ejecuciones de Herramientas con sus resultados

<div id="resources">
  ### Recursos
</div>

Los recursos brindan acceso estructurado a información que la aplicación de IA puede recuperar y suministrar a los modelos como contexto.

<div id="how-resources-work">
  #### Cómo funcionan los Recursos
</div>

Los Recursos exponen datos provenientes de archivos, APIs, bases de datos o cualquier otra fuente que una IA necesite para comprender el contexto. Las aplicaciones pueden acceder a esta información directamente y decidir cómo usarla: ya sea seleccionando partes relevantes, buscando con embeddings o pasándola íntegra al modelo.

Cada recurso tiene un URI único (como `file:///path/to/document.md`) y declara su tipo MIME para un manejo adecuado del contenido. Admiten dos patrones de descubrimiento:

- **Recursos directos**: URIs fijos que apuntan a datos específicos. Ejemplo: `calendar://events/2024` — devuelve la disponibilidad del calendario para 2024
- **Plantillas de recursos**: URIs dinámicos con parámetros para consultas flexibles. Ejemplo:
  - `travel://activities/{city}/{category}` — devuelve actividades por ciudad y categoría
  - `travel://activities/barcelona/museums` — devuelve todos los museos en Barcelona

Las plantillas de recursos incluyen metadatos como título, descripción y tipo MIME esperado, lo que las hace detectables y autodescriptivas.

**Operaciones del protocolo:**

| Método                     | Propósito                             | Devuelve                                 |
| -------------------------- | -------------------------------------- | ---------------------------------------- |
| `resources/list`           | Listar recursos directos disponibles   | Lista de descriptores de recursos        |
| `resources/templates/list` | Descubrir plantillas de recursos       | Lista de definiciones de plantillas      |
| `resources/read`           | Recuperar el contenido de un recurso   | Datos del recurso con metadatos          |
| `resources/subscribe`      | Supervisar cambios de un recurso       | Confirmación de suscripción              |

<div id="example-getting-travel-planning-context">
  #### Ejemplo: obtención de contexto para la planificación de viajes
</div>

Continuando con el ejemplo de planificación de viajes, los Recursos proporcionan a la aplicación de IA acceso a información relevante:

- **Datos del calendario** (`calendar://events/2024`) - Verifica la disponibilidad del usuario
- **Documentos de viaje** (`file:///Documents/Travel/passport.pdf`) - Accede a documentos importantes
- **Itinerarios anteriores** (`trips://history/barcelona-2023`) - Hace referencia a viajes y preferencias previas

La aplicación de IA obtiene estos Recursos y decide cómo procesarlos, ya sea seleccionando un subconjunto de datos mediante embeddings o búsqueda por palabras clave, o pasando datos en bruto directamente al modelo.

En este caso, proporciona al modelo datos del calendario, información meteorológica y preferencias de viaje, lo que le permite comprobar disponibilidad, consultar patrones meteorológicos y tener en cuenta preferencias de viaje anteriores.

**Ejemplos de plantillas de Recursos:**

```json
{
  "uriTemplate": "weather://forecast/{city}/{date}",
  "name": "weather-forecast",
  "title": "Weather Forecast",
  "description": "Get weather forecast for any city and date",
  "mimeType": "application/json"
}

{
  "uriTemplate": "travel://flights/{origin}/{destination}",
  "name": "flight-search",
  "title": "Flight Search",
  "description": "Search available flights between cities",
  "mimeType": "application/json"
}
```

Estas plantillas permiten consultas flexibles. Para los datos meteorológicos, los usuarios pueden acceder a pronósticos para cualquier combinación de ciudad y fecha. Para los vuelos, pueden buscar rutas entre cualesquiera dos aeropuertos. Cuando un usuario ha introducido "NYC" como aeropuerto de `origin` y comienza a introducir "Bar" como aeropuerto de `destination`, el sistema puede sugerir "Barcelona (BCN)" o "Barbados (BGI)".

<div id="parameter-completion">
  #### Autocompletado de parámetros
</div>

Los recursos dinámicos admiten el autocompletado de parámetros. Por ejemplo:

- Escribir «Par» como entrada para `weather://forecast/{city}` podría sugerir «Paris» o «Park City»
- Escribir «JFK» para `flights://search/{airport}` podría sugerir «JFK - John F. Kennedy International»

El sistema ayuda a descubrir valores válidos sin requerir conocer el formato exacto.

<div id="user-interaction-model">
  #### Modelo de interacción del usuario
</div>

Los Recursos están dirigidos por la aplicación, lo que les da flexibilidad a la hora de obtener, procesar y presentar el contexto disponible. Entre los patrones de interacción comunes se encuentran:

- Vistas en árbol o en lista para explorar recursos en estructuras tipo carpeta
- Interfaces de búsqueda y filtrado para encontrar recursos específicos
- Inclusión automática de contexto o sugerencias inteligentes basadas en heurísticas o selección de IA
- Interfaces de selección manual o masiva para incluir uno o varios recursos

Las aplicaciones pueden implementar el descubrimiento de recursos con cualquier patrón de interfaz que se ajuste a sus necesidades. El protocolo no impone patrones de interfaz específicos, lo que permite selectores de recursos con vista previa, sugerencias inteligentes basadas en el contexto de la conversación actual, selección masiva para incluir varios recursos o integración con exploradores de archivos y de datos existentes.

<div id="prompts">
  ### Indicaciones
</div>

Las indicaciones proporcionan plantillas reutilizables. Permiten a los autores de servidores MCP ofrecer indicaciones parametrizadas para un dominio o mostrar la mejor manera de usar el servidor MCP.

<div id="how-prompts-work">
  #### Cómo funcionan las Indicaciones
</div>

Las Indicaciones son plantillas estructuradas que definen las entradas esperadas y los patrones de interacción. Están bajo control del usuario: requieren invocación explícita en lugar de activarse automáticamente. Pueden ser contextuales, haciendo referencia a los Recursos y Herramientas disponibles para crear flujos de trabajo completos. Al igual que los Recursos, las Indicaciones admiten el autocompletado de parámetros para ayudar a los usuarios a descubrir valores de argumentos válidos.

**Operaciones del protocolo:**

| Method         | Purpose                             | Returns                                        |
| -------------- | ----------------------------------- | ---------------------------------------------- |
| `prompts/list` | Descubrir Indicaciones disponibles  | Matriz de descriptores de Indicaciones         |
| `prompts/get`  | Obtener detalles de la Indicación   | Definición completa de la Indicación con argumentos |

<div id="example-streamlined-workflows">
  #### Ejemplo: flujos de trabajo agilizados
</div>

Las Indicaciones proporcionan plantillas estructuradas para tareas comunes. En el contexto de la planificación de viajes:

Indicaciones «Plan a vacation»:

```json
{
  "name": "plan-vacation",
  "title": "Plan a vacation",
  "description": "Guide through vacation planning process",
  "arguments": [
    { "name": "destination", "type": "string", "required": true },
    { "name": "duration", "type": "number", "description": "days" },
    { "name": "budget", "type": "number", "required": false },
    { "name": "interests", "type": "array", "items": { "type": "string" } }
  ]
}
```

En lugar de entradas en lenguaje natural no estructuradas, el sistema de Indicaciones permite:

1. Seleccionar la plantilla «Plan a vacation»
2. Proporcionar entrada estructurada: Barcelona, 7 días, $3000, ["beaches", "architecture", "food"]
3. Ejecutar flujos de trabajo de forma consistente basados en la plantilla

<div id="user-interaction-model">
  #### Modelo de interacción con el usuario
</div>

Las Indicaciones están controladas por el usuario y requieren una invocación explícita. El protocolo ofrece a los implementadores libertad para diseñar interfaces que se sientan naturales dentro de su aplicación. Los principios clave incluyen:

- Descubrimiento sencillo de las Indicaciones disponibles
- Descripciones claras de lo que hace cada Indicación
- Entrada natural de argumentos con validación
- Visualización transparente de la plantilla subyacente de la Indicación

Las aplicaciones suelen exponer las Indicaciones mediante varios patrones de UI, tales como:

- Comandos de barra (escribir “/” para ver Indicaciones disponibles como /plan-vacation)
- Paletas de comandos para acceso con búsqueda
- Botones de UI dedicados para Indicaciones usadas con frecuencia
- Menús contextuales que sugieren Indicaciones relevantes

<div id="bringing-servers-together">
  ## Uniendo servidores
</div>

El verdadero poder del MCP surge cuando varios servidores trabajan juntos y combinan sus capacidades especializadas a través de una interfaz unificada.

<div id="example-multi-server-travel-planning">
  ### Ejemplo: planificación de viajes con varios servidores
</div>

Considera una aplicación de planificación de viajes con IA personalizada, con tres servidores conectados:

- **Servidor de viajes** - Gestiona vuelos, hoteles e itinerarios
- **Servidor meteorológico** - Proporciona datos climáticos y pronósticos
- **Servidor de calendario/correo electrónico** - Administra agendas y comunicaciones

<div id="the-complete-flow">
  #### El flujo completo
</div>

1. **El usuario invoca una indicación con parámetros:**

   ```json
   {
     "prompt": "plan-vacation",
     "arguments": {
       "destination": "Barcelona",
       "departure_date": "2024-06-15",
       "return_date": "2024-06-22",
       "budget": 3000,
       "travelers": 2
     }
   }
   ```

2. **El usuario selecciona recursos para incluir:**
   - `calendar://my-calendar/June-2024` (del Servidor de Calendario)
   - `travel://preferences/europe` (del Servidor de Viajes)
   - `travel://past-trips/Spain-2023` (del Servidor de Viajes)

3. **La IA procesa la solicitud usando herramientas:**

   La IA primero lee todos los recursos seleccionados para reunir contexto: identifica fechas disponibles en el calendario, aprende aerolíneas y tipos de hotel preferidos a partir de las preferencias de viaje, y descubre lugares anteriormente disfrutados según viajes pasados.

   Con este contexto, la IA ejecuta una serie de Herramientas:
   - `searchFlights()` - Consulta a aerolíneas por vuelos de NYC a Barcelona
   - `checkWeather()` - Obtiene pronósticos del tiempo para las fechas del viaje

   Luego, la IA usa esta información para crear la reserva y los pasos siguientes, solicitando aprobación del usuario cuando sea necesario:
   - `bookHotel()` - Encuentra hoteles dentro del presupuesto especificado
   - `createCalendarEvent()` - Agrega el viaje al calendario del usuario
   - `sendEmail()` - Envía una confirmación con los detalles del viaje

**El resultado:** A través de múltiples servidores MCP, el usuario investigó y reservó un viaje a Barcelona adaptado a su agenda. La indicación "Plan a Vacation" guió a la IA para combinar Recursos (disponibilidad del calendario e historial de viajes) con Herramientas (búsqueda de vuelos, reserva de hoteles, actualización de calendarios) en diferentes servidores, reuniendo contexto y ejecutando la reserva. Una tarea que podría haber tomado horas se completó en minutos usando MCP.
