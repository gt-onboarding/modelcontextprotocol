<div id="server-features">
  ## Funciones del servidor
</div>

El Servidor MCP es un componente fundamental en la arquitectura del Protocolo de Contexto de Modelo (MCP) que proporciona herramientas, recursos y capacidades a los clientes. Implementa el lado del servidor del protocolo y se encarga de:

* Exponer herramientas que los clientes pueden descubrir y ejecutar.
  Admite esquemas de entrada y salida y devuelve tipos de contenido estructurados y no estructurados.
* Gestionar recursos con patrones de acceso basados en URI
* Proporcionar plantillas de indicaciones y atender solicitudes de indicaciones
* Admitir la negociación de capacidades con los clientes
* Implementar operaciones del protocolo del lado del servidor
* Gestionar conexiones concurrentes de clientes
* Ofrecer registro estructurado, seguimiento del progreso y notificaciones
* Proporcionar implementaciones de transporte de servidor `STDIO`, `Streamable-HTTP` y `SSE` sin requerir frameworks web externos.
* **Opcional**, dependencias de transporte específicas de Spring `io.modelcontextprotocol.sdk:mcp-spring-webflux`, `io.modelcontextprotocol.sdk:mcp-spring-webmvc` para usuarios de [Spring AI](https://docs.spring.io/spring-ai/reference/api/mcp/mcp-server-boot-starter-docs.html).

<Tip>
  Esta [guía de inicio rápido](/es/quickstart/server), basada en Spring AI MCP, te mostrará
  cómo crear un servidor MCP.
</Tip>

El servidor admite API síncronas y asíncronas, lo que permite una integración flexible en distintos contextos de aplicación.

<Tabs>
  <Tab title="Sync API">
    ```java
    // Create a server with custom configuration
    McpSyncServer syncServer = McpServer.sync(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .capabilities(ServerCapabilities.builder()
            .resources(false, true)  // Enable resource support
            .tools(true)             // Enable tool support
            .prompts(true)           // Enable prompt support
            .logging()               // Enable logging support
            .completions()           // Enable completions support
            .build())
        .build();

    // Register tools, resources, and prompts
    syncServer.addTool(syncToolSpecification);
    syncServer.addResource(syncResourceSpecification);
    syncServer.addPrompt(syncPromptSpecification);

    // Close the server when done
    syncServer.close();
    ```

    ### Conservar los thread-locals en los controladores

    `McpSyncServer` delega la ejecución a un `McpAsyncServer` subyacente.
    La ejecución de los controladores de herramientas, recursos, etc., podría no ocurrir en el hilo que invoca el método.
    En ese caso, se pierden los thread-locals, lo que puede interrumpir ciertas funcionalidades en frameworks que dependen de trabajo ligado al hilo.
    Para asegurar que la ejecución ocurra en el hilo llamador y que los thread-locals sigan disponibles, establece `McpServer.sync(...).immediateExecution(true)`.

    <Note>
      Esto solo es relevante para servidores Sync. Los servidores Async usan la pila reactiva y no deben depender de thread-locals.
    </Note>
  </Tab>

  <Tab title="Async API">
    ```java
    // Create an async server with custom configuration
    McpAsyncServer asyncServer = McpServer.async(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .capabilities(ServerCapabilities.builder()
            .resources(false, true)     // Enable resource support
            .tools(true)                // Enable tool support
            .prompts(true)              // Enable prompt support
            .logging()                  // Enable logging support
            .completions()              // Enable completions support
            .build())
        .build();

    // Register tools, resources, and prompts
    asyncServer.addTool(asyncToolSpecification)
        .doOnSuccess(v -> logger.info("Tool registered"))
        .subscribe();

    asyncServer.addResource(asyncResourceSpecification)
        .doOnSuccess(v -> logger.info("Resource registered"))
        .subscribe();

    asyncServer.addPrompt(asyncPromptSpecification)
        .doOnSuccess(v -> logger.info("Prompt registered"))
        .subscribe();

    // Close the server when done
    asyncServer.close()
        .doOnSuccess(v -> logger.info("Server closed"))
        .subscribe();
    ```
  </Tab>
</Tabs>

<div id="server-transport-providers">
  ## Proveedores de transporte del servidor
</div>

La capa de transporte del SDK de MCP se encarga de gestionar la comunicación entre clientes y servidores.
Ofrece distintas implementaciones para admitir diversos protocolos y patrones de comunicación.
El SDK incluye varias implementaciones de proveedores de transporte integradas:

<Tabs>
  <Tab title="STDIO">
    Cree un transporte basado en procesos:

    ```java
    StdioServerTransportProvider transportProvider = new StdioServerTransportProvider(new ObjectMapper());
    ```

    Ofrece manejo bidireccional de mensajes JSON-RPC a través de flujos estándar de entrada/salida, con procesamiento de mensajes no bloqueante, serialización/deserialización y compatibilidad con apagado controlado.

    Características clave:

    <ul>
      <li>Comunicación bidireccional a través de stdin/stdout</li>
      <li>Compatibilidad con integración basada en procesos</li>
      <li>Configuración sencilla</li>
      <li>Implementación ligera</li>
    </ul>
  </Tab>

  <Tab title="WebFlux">
    <p>Transporte de servidor SSE y HTTP transmitible basado en WebFlux. [Requiere la dependencia `mcp-spring-webflux`](/es/sdk/java/mcp-overview#dependencies).</p>

    <Tabs>
      <Tab title="HTTP transmitible (WebFlux)">
        ```java
        @Configuration
        class McpConfig {
            @Bean
            WebFluxStreamableServerTransportProvider webFluxStreamableServerTransportProvider(ObjectMapper mapper) {
              	return this.mcpStreamableServerTransportProvider = WebFluxStreamableServerTransportProvider.builder()
                        .objectMapper(mapper)
                        .messageEndpoint("/mcp")
                        .build();
            }

            @Bean
            RouterFunction<?> mcpRouterFunction(WebFluxStreamableServerTransportProvider transportProvider) {
                return transportProvider.getRouterFunction();
            }
        }
        ```

        <p>Implementa la especificación de transporte HTTP transmitible y ofrece:</p>

        <ul>
          <li>Comunicación bidireccional mediante HTTP transmitible</li>
          <li>Transmisión HTTP reactiva con WebFlux</li>
          <li>Conexiones concurrentes de clientes a través de endpoints HTTP transmitible</li>
          <li>Enrutamiento de mensajes y gestión de sesiones</li>
          <li>Apagado gradual</li>
        </ul>

        <Note>
          La implementación actual no permite reanudar sesiones por falta de almacenamiento de sesión.
        </Note>

        <Tip>
          En entornos distribuidos con múltiples instancias de Servidor MCP se requiere un enrutamiento adecuado de mensajes.
          Para lograr escalabilidad sin una sobrecarga de infraestructura significativa, considere la implementación de transporte de servidor `Stateless Streamable-HTTP`.
        </Tip>
      </Tab>

<div id="server-capabilities">
  ## Capacidades del servidor
</div>

El servidor se puede configurar con distintas capacidades:

```java
var capabilities = ServerCapabilities.builder()
    .resources(false, true)  // Compatibilidad con Recursos con notificaciones de cambios en la lista
    .tools(true)            // Compatibilidad con Herramientas con notificaciones de cambios en la lista
    .prompts(true)          // Compatibilidad con Indicaciones con notificaciones de cambios en la lista
    .logging()              // Habilitar el registro (activado de forma predeterminada con nivel INFO)
    .build();
```

<div id="tool-specification">
  ### Especificación de Herramientas
</div>

El Protocolo de Contexto de Modelo (MCP) permite que los servidores [expongan herramientas](/es/specification/2024-11-05/server/tools/) que pueden ser invocadas por modelos de lenguaje.
El SDK de Java permite implementar especificaciones de herramientas con sus funciones manejadoras.
Las herramientas permiten que los modelos de IA realicen cálculos, accedan a API externas, consulten bases de datos y manipulen archivos:

<Tabs>
  <Tab title="Sync">
    ```java
    // Sync tool specification
    var schema = """
                {
                  "type" : "object",
                  "id" : "urn:jsonschema:Operation",
                  "properties" : {
                    "operation" : {
                      "type" : "string"
                    },
                    "a" : {
                      "type" : "number"
                    },
                    "b" : {
                      "type" : "number"
                    }
                  }
                }
                """;
    var syncToolSpecification = new McpServerFeatures.SyncToolSpecification(
        new Tool("calculator", "Basic calculator", schema),
        (exchange, arguments) -> {
            // Tool implementation
            return new CallToolResult(result, false);
        }
    );
    ```
  </Tab>

  <Tab title="Async">
    ```java
    // Async tool specification
    var schema = """
                {
                  "type" : "object",
                  "id" : "urn:jsonschema:Operation",
                  "properties" : {
                    "operation" : {
                      "type" : "string"
                    },
                    "a" : {
                      "type" : "number"
                    },
                    "b" : {
                      "type" : "number"
                    }
                  }
                }
                """;
    var asyncToolSpecification = new McpServerFeatures.AsyncToolSpecification(
        new Tool("calculator", "Basic calculator", schema),
        (exchange, arguments) -> {
            // Tool implementation
            return Mono.just(new CallToolResult(result, false));
        }
    );
    ```
  </Tab>
</Tabs>

La especificación de herramientas incluye una definición de herramienta con `name`, `description` y `parameter schema`, seguida de un controlador de llamadas que implementa la lógica de la herramienta.
El primer argumento de la función es `McpAsyncServerExchange` para la interacción con el cliente, y el segundo es un mapa de argumentos de la herramienta.

<div id="resource-specification">
  ### Especificación de Recursos
</div>

Especificación de un recurso con su función controladora.
Los recursos proporcionan contexto a los modelos de IA exponiendo datos como: contenido de archivos, registros de bases de datos, respuestas de API, información del sistema, estado de la aplicación.
Ejemplo de especificación de recurso:

<Tabs>
  <Tab title="Sync">
    ```java
    // Especificación de recurso síncrono
    var syncResourceSpecification = new McpServerFeatures.SyncResourceSpecification(
        new Resource("custom://resource", "name", "description", "mime-type", null),
        (exchange, request) -> {
            // Implementación de lectura del recurso
            return new ReadResourceResult(contents);
        }
    );
    ```
  </Tab>

  <Tab title="Async">
    ```java
    // Especificación de recurso asíncrono
    var asyncResourceSpecification = new McpServerFeatures.AsyncResourceSpecification(
        new Resource("custom://resource", "name", "description", "mime-type", null),
        (exchange, request) -> {
            // Implementación de lectura del recurso
            return Mono.just(new ReadResourceResult(contents));
        }
    );
    ```
  </Tab>
</Tabs>

La especificación del recurso se compone de definiciones de recursos y un controlador de lectura de recursos.
La definición del recurso incluye `name`, `description` y `MIME type`.
El primer argumento de la función que maneja las solicitudes de lectura de recursos es un `McpAsyncServerExchange`, con el cual el servidor puede interactuar con el cliente conectado.
El segundo argumento es un `McpSchema.ReadResourceRequest`.

<div id="prompt-specification">
  ### Especificación de Indicaciones
</div>

Como parte de las [capacidades de Indicaciones](/es/specification/2024-11-05/server/prompts/), MCP proporciona una forma estandarizada para que los servidores expongan plantillas de indicaciones a los clientes.
La Especificación de Indicaciones es una plantilla estructurada para interacciones con modelos de IA que permite un formato de mensajes consistente, sustitución de parámetros, inyección de contexto, formateo de respuestas y plantillas de instrucciones.

<Tabs>
  <Tab title="Sync">
    ```java
    // Especificación de indicación síncrona
    var syncPromptSpecification = new McpServerFeatures.SyncPromptSpecification(
        new Prompt("greeting", "description", List.of(
            new PromptArgument("name", "description", true)
        )),
        (exchange, request) -> {
            // Implementación de la indicación
            return new GetPromptResult(description, messages);
        }
    );
    ```
  </Tab>

  <Tab title="Async">
    ```java
    // Especificación de indicación asíncrona
    var asyncPromptSpecification = new McpServerFeatures.AsyncPromptSpecification(
        new Prompt("greeting", "description", List.of(
            new PromptArgument("name", "description", true)
        )),
    ```

La definición de la indicación incluye el nombre (identificador de la indicación), la descripción (propósito de la indicación) y la lista de argumentos (parámetros para la creación de plantillas).
La función controladora procesa las solicitudes y devuelve plantillas formateadas.
El primer argumento es `McpAsyncServerExchange` para la interacción con el cliente y el segundo argumento es una instancia de `GetPromptRequest`.

    ```java
    var mcpServer = McpServer.sync(mcpServerTransportProvider)

        .build())

      .completions(new McpServerFeatures.SyncCompletionSpecification( // register completion specification
    ```

<div id="using-sampling-from-a-server">
  ### Uso de muestreo desde un servidor
</div>

Para usar las [capacidades de muestreo](/es/specification/2024-11-05/client/sampling/), necesitas un cliente compatible que lo admita.
No hace falta una configuración especial del servidor, pero verifica la compatibilidad de muestreo del cliente antes de hacer solicitudes.
Más información sobre la [compatibilidad de muestreo en el cliente](es/./mcp-client#sampling-support).

Cuando un cliente compatible se conecta a un servidor con estado, el servidor puede solicitar generaciones del modelo de lenguaje:

<Tabs>
  <Tab title="Sync API">
    ```java
    // Create a server
    McpSyncServer server = McpServer.sync(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .build();

    // Define a tool that uses sampling
    var calculatorTool = new McpServerFeatures.SyncToolSpecification(
        new Tool("ai-calculator", "Performs calculations using AI", schema),
        (exchange, arguments) -> {
            // Check if client supports sampling
            if (exchange.getClientCapabilities().sampling() == null) {
                return new CallToolResult("Client does not support AI capabilities", false);
            }

            // Create a sampling request
            McpSchema.CreateMessageRequest request = McpSchema.CreateMessageRequest.builder()
                .messages(List.of(new McpSchema.SamplingMessage(McpSchema.Role.USER,
                    new McpSchema.TextContent("Calculate: " + arguments.get("expression")))
                .modelPreferences(McpSchema.ModelPreferences.builder()
                    .hints(List.of(
                        McpSchema.ModelHint.of("claude-3-sonnet"),
                        McpSchema.ModelHint.of("claude")
                    ))
                    .intelligencePriority(0.8)  // Prioritize intelligence
                    .speedPriority(0.5)         // Moderate speed importance
                    .build())
                .systemPrompt("You are a helpful calculator assistant. Provide only the numerical answer.")
                .maxTokens(100)
                .build();

            // Request sampling from the client
            McpSchema.CreateMessageResult result = exchange.createMessage(request);

            // Process the result
            String answer = result.content().text();
            return new CallToolResult(answer, false);
        }
    );

    // Add the tool to the server
    server.addTool(calculatorTool);
    ```
  </Tab>

  <Tab title="Async API">
    ```java

    McpAsyncServer server = McpServer.async(transportProvider)
        .serverInfo("my-server", "1.0.0")
        .build();

        new Tool("ai-calculator", "Performs calculations using AI", schema),
        (exchange, arguments) -> {

                return Mono.just(new CallToolResult("Client does not support AI capabilities", false));
            }

            // Create a sampling request
            McpSchema.CreateMessageRequest request = McpSchema.CreateMessageRequest.builder()
                .content(new McpSchema.TextContent("Calculate: " + arguments.get("expression")))
                .modelPreferences(McpSchema.ModelPreferences.builder()
                    .hints(List.of(
                        McpSchema.ModelHint.of("claude-3-sonnet"),
                        McpSchema.ModelHint.of("claude")
                    ))
                    .intelligencePriority(0.8)  // Prioritize intelligence
                    .speedPriority(0.5)         // Moderate speed importance
                    .build())
                .systemPrompt("You are a helpful calculator assistant. Provide only the numerical answer.")

            return exchange.createMessage(request)
                .map(result -> {
                    // Process the result
    ```

El objeto `CreateMessageRequest` te permite especificar: `Content` (el texto o la imagen de entrada para el modelo),
`Model Preferences` (sugerencias y prioridades para la selección del modelo), `System Prompt` (instrucciones para el comportamiento del modelo) y
`Max Tokens` (longitud máxima de la respuesta generada).

<div id="logging-support">
  ### Compatibilidad con registro
</div>

El servidor ofrece capacidades de registro estructurado que permiten enviar mensajes de registro a los clientes con distintos niveles de gravedad. Las notificaciones de registro solo pueden enviarse dentro de una sesión de cliente existente, como en llamadas a Herramientas, Recursos e Indicaciones.

Por ejemplo, podemos enviar un mensaje de registro desde una función controladora de herramienta.
En el lado del cliente, puedes registrar un consumidor de registro para recibir mensajes del servidor y establecer el nivel mínimo de registro para filtrar mensajes.

```java
var mcpClient = McpClient.sync(transport)
        .loggingConsumer(notification -> {
            System.out.println("Received log message: " + notification.data());
        })
        .build();

mcpClient.initialize();

mcpClient.setLoggingLevel(McpSchema.LoggingLevel.INFO);

// Llamar a la herramienta que envía notificaciones de registro
CallToolResult result = mcpClient.callTool(new McpSchema.CallToolRequest("logging-test", Map.of()));
```

El servidor puede enviar mensajes de registro usando el objeto `McpAsyncServerExchange`/`McpSyncServerExchange` en la función controladora de herramienta/recurso/indicación:

```java
var tool = new McpServerFeatures.AsyncToolSpecification(
    new McpSchema.Tool("logging-test", "Test logging notifications", emptyJsonSchema),
    (exchange, request) -> {

      exchange.loggingNotification( // Usar el exchange para enviar mensajes de registro
          McpSchema.LoggingMessageNotification.builder()
            .level(McpSchema.LoggingLevel.DEBUG)
            .logger("test-logger")
            .data("Debug message")
            .build())
        .block();

      return Mono.just(new CallToolResult("Logging test completed", false));
    });

var mcpServer = McpServer.async(mcpServerTransportProvider)
  .serverInfo("test-server", "1.0.0")
  .capabilities(
    ServerCapabilities.builder()
      .logging() // Habilitar compatibilidad con registro
      .tools(true)
      .build())
  .tools(tool)
  .build();
```

Los clientes pueden controlar el nivel mínimo de registro que reciben mediante la solicitud `mcpClient.setLoggingLevel(level)`. Los mensajes por debajo del nivel establecido se filtrarán.
Niveles de registro compatibles (en orden de gravedad creciente): DEBUG (0), INFO (1), NOTICE (2), WARNING (3), ERROR (4), CRITICAL (5), ALERT (6), EMERGENCY (7)

<div id="error-handling">
  ## Gestión de errores
</div>

El SDK ofrece una gestión de errores integral mediante la clase McpError, que abarca compatibilidad del protocolo, comunicación de transporte, mensajería JSON-RPC, ejecución de Herramientas, gestión de Recursos, manejo de Indicaciones, tiempos de espera y problemas de conexión. Este enfoque unificado garantiza un tratamiento coherente y fiable de los errores tanto en operaciones síncronas como asíncronas.
