<div id="client-features">
  ## Funciones del cliente
</div>

El Cliente MCP es un componente clave en la arquitectura del Protocolo de Contexto de Modelo (MCP), responsable de establecer y gestionar conexiones con Servidores MCP. Implementa el lado del cliente del protocolo y se encarga de:

* La negociación de versiones del protocolo para garantizar la compatibilidad con los servidores
* La negociación de capacidades para determinar las funciones disponibles
* El transporte de mensajes y la comunicación JSON-RPC
* El descubrimiento y la ejecución de Herramientas
* El acceso y la gestión de Recursos
* Las interacciones con el sistema de Indicaciones
* Funciones opcionales como la gestión de Raíces y el soporte de Muestreo

<Tip>
  El módulo principal `io.modelcontextprotocol.sdk:mcp` proporciona implementaciones de transporte de cliente para STDIO, HTTP transmitible y SSE sin requerir frameworks web externos.

  Las implementaciones de transporte específicas de Spring están disponibles como una dependencia **opcional**: `io.modelcontextprotocol.sdk:mcp-spring-webflux`, para usuarios de [Spring Framework](https://docs.spring.io/spring-ai/reference/api/mcp/mcp-client-boot-starter-docs.html).
</Tip>

<Tip>
  Esta [guía de inicio rápido](/es/quickstart/client), basada en Spring AI MCP, te mostrará
  cómo crear un cliente de IA que se conecte a Servidores MCP.
</Tip>

El cliente ofrece APIs síncronas y asíncronas para adaptarse a distintos contextos de aplicación.

<Tabs>
  <Tab title="API sincrónica">
    ```java
    // Crea un cliente sincrónico con configuración personalizada
    McpSyncClient client = McpClient.sync(transport)
        .requestTimeout(Duration.ofSeconds(10))
        .capabilities(ClientCapabilities.builder()
            .roots(true)      // Habilitar la capacidad de Raíces
            .sampling()       // Habilitar la capacidad de Muestreo
            .elicitation()    // Habilitar la capacidad de Elicitación
            .build())
        .sampling(request -> CreateMessageResult.builder()...build())
        .elicitation(elicitRequest -> ElicitResult.builder()...build())
        .toolsChangeConsumer((List<McpSchema.Tool> tools) -> ...)
        .resourcesChangeConsumer((List<McpSchema.Resource> resources) -> ...)
        .promptsChangeConsumer((List<McpSchema.Prompt> prompts) -> ...)
        .loggingConsumer((LoggingMessageNotification logging) -> ...)
        .progressConsumer((ProgressNotification progress) -> ...)
        .build();

    // Inicializar la conexión
    client.initialize();

    // Listar las Herramientas disponibles
    ListToolsResult tools = client.listTools();

    // Llamar a una Herramienta
    CallToolResult result = client.callTool(
        new CallToolRequest("calculator",
            Map.of("operation", "add", "a", 2, "b", 3))
    );

    // Listar y leer Recursos
    ListResourcesResult resources = client.listResources();
    ReadResourceResult resource = client.readResource(
        new ReadResourceRequest("resource://uri")
    );

    // Listar y usar Indicaciones
    ListPromptsResult prompts = client.listPrompts();
    GetPromptResult prompt = client.getPrompt(
        new GetPromptRequest("greeting", Map.of("name", "Spring"))
    );

    // Agregar/eliminar Raíces
    client.addRoot(new Root("file:///path", "description"));
    client.removeRoot("file:///path");

    // Cerrar el cliente
    client.closeGracefully();
    ```
  </Tab>

  <Tab title="API asíncrona">
    ```java
    // Crea un cliente asíncrono con configuración personalizada
    McpAsyncClient client = McpClient.async(transport)
        .requestTimeout(Duration.ofSeconds(10))
        .capabilities(ClientCapabilities.builder()
            .roots(true)      // Habilitar la capacidad de Raíces
            .sampling()       // Habilitar la capacidad de Muestreo
            .elicitation()    // Habilitar la capacidad de Elicitación
            .build())
        .sampling(request -> Mono.just(new CreateMessageResult(response)))
        .elicitation(elicitRequest -> Mono.just(ElicitResult.builder()...build()))
        .toolsChangeConsumer(tools -> Mono.fromRunnable(() -> logger.info("Herramientas actualizadas: {}", tools)))
        .resourcesChangeConsumer(resources -> Mono.fromRunnable(() -> logger.info("Recursos actualizados: {}", resources)))
        .promptsChangeConsumer(prompts -> Mono.fromRunnable(() -> logger.info("Indicaciones actualizadas: {}", prompts)))
        .loggingConsumer(notification -> Mono.fromRunnable(() -> logger.info("Registro: {}", notification.data())))
        .progressConsumer(progress -> Mono.fromRunnable(() -> logger.info("Actualización de progreso: {}", progress.data())))
        .build();

    // Inicializar la conexión y usar las funciones
    client.initialize()
        .flatMap(initResult -> client.listTools())
        .flatMap(tools -> {
            return client.callTool(new CallToolRequest(
                "calculator",
                Map.of("operation", "add", "a", 2, "b", 3)
            ));
        })
        .flatMap(result -> {
            return client.listResources()
                .flatMap(resources ->
                    client.readResource(new ReadResourceRequest("resource://uri"))
                );
        })
        .flatMap(resource -> {
            return client.listPrompts()
                .flatMap(prompts ->
                    client.getPrompt(new GetPromptRequest(
                        "greeting",
                        Map.of("name", "Spring")
                    ))
                );
        })
        .flatMap(prompt -> {
            return client.addRoot(new Root("file:///path", "description"))
                .then(client.removeRoot("file:///path"));
        })
        .doFinally(signalType -> {
            client.closeGracefully().subscribe();
        })
        .subscribe();
    ```
  </Tab>
</Tabs>

<div id="client-transport">
  ## Transporte del cliente
</div>

La capa de transporte gestiona la comunicación entre el Cliente MCP y el Servidor MCP, ofreciendo distintas implementaciones para diversos casos de uso. El transporte del cliente se encarga de la serialización de mensajes, el establecimiento de conexiones y los patrones de comunicación específicos del protocolo.

<Tabs>
  <Tab title="STDIO">
    Crea un transporte para la comunicación dentro del mismo proceso

    ```java
    ServerParameters params = ServerParameters.builder("npx")
        .args("-y", "@modelcontextprotocol/server-everything", "dir")
        .build();
    McpTransport transport = new StdioClientTransport(params);
    ```
  </Tab>

  <Tab title="HttpClient">
    <Tabs>
      <Tab title="Streamable-HTTP (HttpClient)">
        Transporte de cliente Streamable-HTTP independiente del framework (solo con APIs del JDK)

        ```java
        McpTransport transport = HttpClientStreamableHttpTransport
                                    .builder("http://your-mcp-server")
                                    .build();
        ```
      </Tab>

      <Tab title="SSE (HttpClient)">
        Transporte de cliente SSE independiente del framework (solo con APIs del JDK)

        ```java
        McpTransport transport = HttpClientSseClientTransport
                                    .builder("http://your-mcp-server")
                                    .build();
        ```
      </Tab>
    </Tabs>

    **HttpClient: Personalización de solicitudes HTTP**

    Para personalizar el generador base de solicitudes HTTP utilizado en cada solicitud, proporciona un `HttpRequestBuilder` personalizado.
    Esto está disponible tanto en los transportes HTTP transmitible como SSE.
    Al usar un `HttpRequest.Builder` personalizado, cada solicitud HTTP emitida por el transporte
    usará la configuración fija del generador.
    Por ejemplo, esto es útil para incluir en un encabezado un token de seguridad que no expire.
    Para agregar un encabezado `X-Custom-Header` a cada solicitud, usa:

    ```java
    var requestBuilder = HttpRequest
        .newBuilder()
        .header("X-Custom-Header", "some header value");

    HttpClientStreamableHttpTransport
        .builder("https://mcp.example.com")
        .requestBuilder(requestBuilder)
        .build();
    ```

    Para modificar dinámicamente la solicitud HTTP antes de emitirla, implementa `McpSyncHttpClientRequestCustomizer` o `McpAsyncHttpClientRequestCustomizer`.
    Elige el personalizador de solicitudes que coincida con el tipo de Cliente MCP, síncrono o asíncrono.
    Ten en cuenta que los thread-locals pueden no estar disponibles en los personalizadores; la información relacionada con el contexto
    debe obtenerse a través de `McpTransportContext` (consulta [adding context information](#adding-context-information)).
    Implementaciones de ejemplo:

    ```java
    // Sync
    class MyRequestCustomizer implements McpSyncHttpClientRequestCustomizer {
        @Override
        public void customize(HttpRequest.Builder builder, String method,
            URI endpoint, String body, McpTransportContext context) {
            // ... custom logic ...
            var token = obtainAccessToken(context);
            builder.header("Authorization", "Bearer " + token);
        }
    }

    // Async
    class MyAsyncRequestCustomizer implements McpAsyncHttpClientRequestCustomizer {
        @Override
        public Publisher<HttpRequest.Builder> customize(HttpRequest.Builder builder,
            String method, URI endpoint, String body, McpTransportContext context) {
            // ... custom logic ...
            Mono<String> token = obtainAccessToken(context);
            return token.map(t ->
                builder.copy()
                    .header("Authorization", "Bearer " + t)
            );
        }
    }
    ```

    Los transportes, tanto HTTP transmitible como SSE, se pueden configurar para usar un único personalizador.

    ```java
    HttpClientStreamableHttpTransport
        .builder("https://mcp.example.com")
        .httpRequestCustomizer(new MyRequestCustomizer()) // sync
        .asyncHttpRequestCustomizer(new MyAsyncRequestCustomizer()) // OR async
        .build();
    ```

    Para componer varios personalizadores, usa `DelegatingMcpSyncHttpClientRequestCustomizer` o `DelegatingMcpAsyncHttpClientRequestCustomizer`.
  </Tab>

    <Tabs>
      <Tab title="HTTP transmitible (WebClient)">

      <Tab title="SSE (WebClient)">

    **WebClient: Personalización de solicitudes HTTP**

    Para personalizar las solicitudes HTTP salientes, proporcione un `WebClient.Builder` personalizado.
    Al usar un builder personalizado, cada solicitud enviada por el transporte se personalizará.
    Por ejemplo, esto es útil para incluir en un encabezado un token de seguridad que no expire.
    Para agregar el encabezado `X-Custom-Header` a cada solicitud, use:

    ```java
    var webClientBuilder = WebClient.builder()
        .defaultHeader("X-Custom-Header", "some header value");

    McpTransport transport = WebClientStreamableHttpTransport
                                .builder(webClientBuilder)
                                .build();
    ```

    Para modificar dinámicamente la solicitud HTTP, el builder cuenta con una API específica: `ExchangeFilterFunction`.
    En esa función, la nueva solicitud puede calcularse en tiempo de ejecución y se puede obtener información adicional
    del [contexto de Reactor](https://projectreactor.io/docs/core/release/reference/advancedFeatures/context.html).
    Es común almacenar información de contexto en un `McpTransportContext` dentro del contexto de Reactor,
    típicamente en McpSyncClient, aunque se puede usar cualquier clave de contexto. Por ejemplo:

    ```java
    WebClient.builder()
        .filter((request, next) -> {
            return Mono.deferContextual(ctx -> {
                var transportContext = ctx.get(McpTransportContext.KEY);
                var otherInfo = ctx.get("your-context-key");
                Mono<String> token = obtainAccessToken(transportContext, otherInfo);
                return token.map(t -> {
                    var newRequest = ClientRequest.from(request)
                        .header("Authorization", "Bearer " + t)
                        .build();
                    // Asegúrate de llamar a next.exchange para ejecutar la solicitud HTTP
                    return next.exchange(newRequest);
                });
            });
        });
    ```

    Para saber cómo poblar la información de contexto, consulte [agregar información de contexto](#adding-context-information).
  </Tab>
</Tabs>

<div id="client-capabilities">
  ## Capacidades del cliente
</div>

El cliente se puede configurar con varias capacidades:

```java
var capacidades = ClientCapabilities.builder()
    .roots(true)      // Habilita el soporte de Raíces del sistema de archivos con notificaciones de cambios en la lista
    .sampling()       // Habilita el soporte de Muestreo de LLM
    .elicitation()    // Habilita la capacidad de Elicitación
    .build();
```

<div id="roots-support">
  ### Compatibilidad con Raíces
</div>

Las Raíces definen los límites dentro de los cuales los servidores pueden operar en el sistema de archivos:

```java
// Agregar una raíz dinámicamente
client.addRoot(new Root("file:///path", "description"));

// Eliminar una raíz
client.removeRoot("file:///path");

// Notificar al servidor sobre cambios en las raíces
client.rootsListChangedNotification();
```

La capacidad de Raíces permite a los servidores:

* Solicitar la lista de raíces del sistema de archivos a las que tienen acceso
* Recibir notificaciones cuando cambia la lista de raíces
* Comprender a qué directorios y archivos tienen acceso

<div id="sampling-support">
  ### Compatibilidad con muestreo
</div>

El muestreo permite que los servidores soliciten interacciones con LLM («completions» o «generations») a través del cliente:

```java
// Configure sampling handler
Function<CreateMessageRequest, CreateMessageResult> samplingHandler = request -> {
    // Sampling implementation that interfaces with LLM
    return new CreateMessageResult(response);
};

// Create client with sampling support
var client = McpClient.sync(transport)
    .capabilities(ClientCapabilities.builder()
        .sampling()
        .build())
    .sampling(samplingHandler)
    .build();
```

Esta capacidad permite:

* Que los servidores aprovechen capacidades de IA sin requerir claves de API
* Que los clientes mantengan el control sobre el acceso al modelo y los permisos
* Compatibilidad con interacciones tanto de texto como basadas en imágenes
* Inclusión opcional del contexto del Servidor MCP en las indicaciones

<div id="elicitation-support">
  ### Compatibilidad con Elicitación
</div>

Elicitación permite que los servidores soliciten información específica o aclaraciones al cliente:

```java
// Configurar el manejador de elicitación
Function<ElicitRequest, ElicitResult> elicitationHandler = request -> {
    // Implementación de elicitación que interactúa con el LLM
    return ElicitResult.builder()...build();
};

// Crear un cliente con compatibilidad de elicitación
var client = McpClient.sync(transport)
    .capabilities(ClientCapabilities.builder()
        .elicitation() // habilitar la capacidad de elicitación
        .build())
    .elicitation(elicitationHandler) // registrar el manejador de elicitación
    .build();
```

<div id="logging-support">
  ### Compatibilidad con el registro (Logging)
</div>

El cliente puede registrar un consumidor de registros para recibir mensajes de registro del servidor y establecer el nivel mínimo de registro para filtrar mensajes:

```java
var mcpClient = McpClient.sync(transport)
        .loggingConsumer((LoggingMessageNotification notification) -> {
            System.out.println("Received log message: " + notification.data());
        })
        .build();

mcpClient.initialize();

mcpClient.setLoggingLevel(McpSchema.LoggingLevel.INFO);

// Llamar a la herramienta que puede enviar notificaciones de registro
CallToolResult result = mcpClient.callTool(new McpSchema.CallToolRequest("logging-test", Map.of()));
```

Los clientes pueden controlar el nivel mínimo de registro que reciben mediante la llamada `mcpClient.setLoggingLevel(level)`. Los mensajes por debajo del nivel establecido se filtrarán.
Niveles de registro admitidos (en orden de gravedad ascendente): DEBUG (0), INFO (1), NOTICE (2), WARNING (3), ERROR (4), CRITICAL (5), ALERT (6), EMERGENCY (7)

<div id="progress-support">
  ### Compatibilidad con progreso
</div>

El cliente puede registrar un consumidor de progreso para recibir actualizaciones de progreso del servidor:

```java
var mcpClient = McpClient.sync(transport)
        .progressConsumer((ProgressNotification progress) -> {
            System.out.println("Actualización de progreso recibida: " + progress.data());
        })
        .build();

mcpClient.initialize();

// Llamar a la herramienta que puede enviar notificaciones de progreso
CallToolResult result = mcpClient.callTool(new McpSchema.CallToolRequest("progress-test", Map.of()));
```

<div id="change-notifications">
  ### Notificaciones de cambios
</div>

El cliente puede registrar un consumidor de cambios para recibir notificaciones del servidor sobre actualizaciones de Herramientas, Recursos o Indicaciones:

```java

var spec = McpClient.sync(transport);

// Adds a consumer to be notified when the available tools change, such as tools
// being added or removed.
spec.toolsChangeConsumer((List<McpSchema.Tool> tools) -> {
    // Manejar cambios en las herramientas
});

// Adds a consumer to be notified when the available resources change, such as resources
// being added or removed.
spec.resourcesChangeConsumer((List<McpSchema.Resource> resources) -> {
    // Manejar cambios en los recursos
});

// Adds a consumer to be notified when the available prompts change, such as prompts
// being added or removed.
spec.promptsChangeConsumer((List<McpSchema.Prompt> prompts) -> {
    // Manejar cambios en las indicaciones

});
```

<div id="tool-execution">
  ### Ejecución de Herramientas
</div>

Las Herramientas son funciones del lado del servidor que los clientes pueden descubrir y ejecutar. El Cliente MCP proporciona métodos para listar las herramientas disponibles y ejecutarlas con parámetros específicos. Cada herramienta tiene un nombre único y acepta un mapa de parámetros.

<Tabs>
  <Tab title="API síncrona">
    ```java
    // Listar las herramientas disponibles y sus nombres
    var tools = client.listTools();
    tools.forEach(tool -> System.out.println(tool.getName()));

    // Ejecutar una herramienta con parámetros
    var result = client.callTool("calculator", Map.of(
        "operation", "add",
        "a", 1,
        "b", 2
    ));
    ```
  </Tab>

  <Tab title="API asíncrona">
    ```java
    // Listar las herramientas disponibles de forma asíncrona
    client.listTools()
        .doOnNext(tools -> tools.forEach(tool ->
            System.out.println(tool.getName())))
        .subscribe();

    // Ejecutar una herramienta de forma asíncrona
    client.callTool("calculator", Map.of(
            "operation", "add",
            "a", 1,
            "b", 2
        ))
        .subscribe();
    ```
  </Tab>
</Tabs>

<div id="resource-access">
  ### Acceso a recursos
</div>

Los recursos representan fuentes de datos del lado del servidor a las que los clientes pueden acceder mediante plantillas de URI. El Cliente MCP ofrece métodos para descubrir los recursos disponibles y obtener su contenido a través de una interfaz estandarizada.

<Tabs>
  <Tab title="API síncrona">
    ```java
    // Listar los recursos disponibles y sus nombres
    var resources = client.listResources();
    resources.forEach(resource -> System.out.println(resource.getName()));

    // Obtener el contenido de un recurso usando una plantilla de URI
    var content = client.getResource("file", Map.of(
        "path", "/path/to/file.txt"
    ));
    ```
  </Tab>

  <Tab title="API asíncrona">
    ```java
    // Listar los recursos disponibles de forma asíncrona
    client.listResources()
        .doOnNext(resources -> resources.forEach(resource ->
            System.out.println(resource.getName())))
        .subscribe();

    // Obtener el contenido de un recurso de forma asíncrona
    client.getResource("file", Map.of(
            "path", "/path/to/file.txt"
        ))
        .subscribe();
    ```
  </Tab>
</Tabs>

<div id="prompt-system">
  ### Sistema de indicaciones

El sistema de indicaciones permite interactuar con plantillas de indicaciones del lado del servidor. Estas plantillas pueden descubrirse y ejecutarse con parámetros personalizados, lo que posibilita la generación dinámica de texto basada en patrones predefinidos.

  <Tab title="Sync API">

    ```java
        "text", "Hello, World!"
    ```

  <Tab title="Async API">
    ```java

    client.listPrompts()

            System.out.println(prompt.getName())))
        .subscribe();

    // Ejecutar una plantilla de indicaciones de forma asíncrona
    ```

<div id="using-completion">
  ### Uso de Completion
</div>

Como parte de las [capacidades de Completion](/es/specification/2025-03-26/server/utilities/completion), MCP ofrece una forma estandarizada para que los servidores proporcionen sugerencias de autocompletado de argumentos para Indicaciones y URI de Recursos.

Consulta las [capacidades de Completion del servidor](/es/sdk/java/mcp-server#completion-specification) para aprender a habilitar y configurar las completions en el servidor.

En el lado del cliente, el Cliente MCP proporciona métodos para solicitar autocompletado:

<Tabs>
  <Tab title="Sync API">
    ```java

    CompleteRequest request = new CompleteRequest(
            new PromptReference("code_review"),
            new CompleteRequest.CompleteArgument("language", "py"));

    CompleteResult result = syncMcpClient.completeCompletion(request);

    ```
  </Tab>

  <Tab title="Async API">
    ```java

    CompleteRequest request = new CompleteRequest(
            new PromptReference("code_review"),
            new CompleteRequest.CompleteArgument("language", "py"));

    Mono<CompleteResult> result = mcpClient.completeCompletion(request);

    ```
  </Tab>
</Tabs>

<div id="adding-context-information">
  ### Agregar información de contexto
</div>

La solicitud HTTP enviada a través del transporte SSE o HTTP transmitible se puede personalizar con
APIs específicas (ver [client transport](#client-transport)). Estos personalizadores pueden necesitar
información adicional específica del contexto que debe inyectarse a nivel del cliente.

<Tabs>
  <Tab title="Sync API">
    El `McpSyncClient` se usa en un entorno bloqueante y puede apoyarse en variables locales de hilo para compartir información.
    Por ejemplo, algunos frameworks almacenan la solicitud actual del servidor o tokens de seguridad en una variable local de hilo.
    Para poner este tipo de información a disposición de los transportes subyacentes, use `SyncSpec#transportContextProvider`:

    ```java
    McpClient.sync(transport)
        .transportContextProvider(() -> {
            var data = obtainDataFromThreadLocals();
            return McpTransportContext.create(
                    Map.of("some-data", data)
            );
        })
        .build();
    ```

    Este `McpTransportContext` estará disponible en `McpSyncHttpClientRequestCustomizer` basado en HttpClient
    y en `ExchangeFilterFunction` basado en WebClient.
  </Tab>

  <Tab title="Async API">
    El `McpAsyncClient` se usa en un entorno reactivo.
    La información se transmite a través de la cadena reactiva usando el [contexto de Reactor](https://projectreactor.io/docs/core/release/reference/advancedFeatures/context.html).
    Para insertar datos en el contexto de Reactor, use `contextWrite` al invocar operaciones del cliente:

    ```java
    var client = McpClient.async(transport)
        // ...
        .build();

    Mono<McpSchema.ListToolsResult> toolList = client
        .listTools()
        .contextWrite(ctx -> {
            var transportContext = McpTransportContext.create(
                Map.of("some-key", "some value")
            );
            return ctx.put(McpTransportContext.KEY, transportContext)
                .put("reactor-context-key", "some reactor value");
        });
    ```

    Con esto, el `McpTransportContext` estará disponible en `McpAsyncHttpClientRequestCustomizer` basado en HttpClient.
    Todo el contexto de Reactor también está disponible tanto en `McpAsyncHttpClientRequestCustomizer`
    como en el `ExchangeFilterFunction` de WebClient, mediante `Mono.deferContextual`:

    ```java
    WebClient.builder()
        .filter((request, next) -> {
            return Mono.deferContextual(ctx -> {
                var transportContext = ctx.get(McpTransportContext.KEY);
                var someReactorValue = ctx.get("reactor-context-key");
                // ... use context values ...
            });
        });
    ```
  </Tab>
</Tabs>
