title: "Herramientas"
description: "Permite que los LLM realicen acciones a través de tu servidor"

Las herramientas son una primitiva potente en el Protocolo de Contexto del Modelo (MCP) que permite a los servidores exponer funcionalidad ejecutable a los clientes. A través de las herramientas, los LLM pueden interactuar con sistemas externos, realizar cálculos y llevar a cabo acciones en el mundo real.

<Note>
  Las herramientas están diseñadas para ser **controladas por el modelo**, lo que significa que se exponen desde los servidores a los clientes con la intención de que el modelo de IA pueda invocarlas automáticamente (con una persona en el ciclo para otorgar aprobación).
</Note>

<div id="overview">
  ## Descripción general
</div>

Las herramientas en MCP permiten que los servidores expongan funciones ejecutables que los clientes pueden invocar y que los LLM pueden utilizar para realizar acciones. Los aspectos clave de las herramientas incluyen:

* **Descubrimiento**: Los clientes pueden obtener una lista de herramientas disponibles enviando una solicitud `tools/list`
* **Invocación**: Las herramientas se invocan mediante la solicitud `tools/call`, donde los servidores realizan la operación solicitada y devuelven resultados
* **Flexibilidad**: Las herramientas pueden abarcar desde cálculos simples hasta interacciones complejas con API

Al igual que los [recursos](/es/legacy/concepts/resources), las herramientas se identifican con nombres únicos y pueden incluir descripciones para orientar su uso. Sin embargo, a diferencia de los recursos, las herramientas representan operaciones dinámicas que pueden modificar el estado o interactuar con sistemas externos.

<div id="tool-definition-structure">
  ## Estructura de definición de herramientas
</div>

Cada herramienta se define con la siguiente estructura:

```typescript
{
  name: string;          // Identificador único de la herramienta
  description?: string;  // Descripción en lenguaje natural
  inputSchema: {         // JSON Schema para los parámetros de la herramienta
    type: "object",
    properties: { ... }  // Parámetros específicos de la herramienta
  },
  annotations?: {        // Pistas opcionales sobre el comportamiento de la herramienta
    title?: string;      // Título en lenguaje natural de la herramienta
    readOnlyHint?: boolean;    // Si es true, la herramienta no modifica su entorno
    destructiveHint?: boolean; // Si es true, la herramienta puede realizar actualizaciones destructivas
    idempotentHint?: boolean;  // Si es true, las llamadas repetidas con los mismos argumentos no tienen efecto adicional
    openWorldHint?: boolean;   // Si es true, la herramienta interactúa con entidades externas
  }
}
```

<div id="implementing-tools">
  ## Implementación de Herramientas
</div>

Aquí tienes un ejemplo de cómo implementar una herramienta básica en un Servidor MCP:

<CodeGroup>
  ```typescript TypeScript
  const server = new Server(
    {
      name: "example-server",
      version: "1.0.0",
    },
    {
      capabilities: {
        tools: {},
      },
    },
  );

  // Definir las herramientas disponibles
  server.setRequestHandler(ListToolsRequestSchema, async () => {
    return {
      tools: [
        {
          name: "calculate_sum",
          description: "Suma dos números",
          inputSchema: {
            type: "object",
            properties: {
              a: { type: "number" },
              b: { type: "number" },
            },
            required: ["a", "b"],
          },
        },
      ],
    };
  });

  // Gestionar la ejecución de herramientas
  server.setRequestHandler(CallToolRequestSchema, async (request) => {
    if (request.params.name === "calculate_sum") {
      const { a, b } = request.params.arguments;
      return {
        content: [
          {
            type: "text",
            text: String(a + b),
          },
        ],
      };
    }
    throw new Error("Tool not found");
  });
  ```

  ```python Python
  app = Server("example-server")

  @app.list_tools()
  async def list_tools() -> list[types.Tool]:
      return [
          types.Tool(
              name="calculate_sum",
              description="Suma dos números",
              inputSchema={
                  "type": "object",
                  "properties": {
                      "a": {"type": "number"},
                      "b": {"type": "number"}
                  },
                  "required": ["a", "b"]
              }
          )
      ]

  @app.call_tool()
  async def call_tool(
      name: str,
      arguments: dict
  ) -> list[types.TextContent | types.ImageContent | types.EmbeddedResource]:
      if name == "calculate_sum":
          a = arguments["a"]
          b = arguments["b"]
          result = a + b
          return [types.TextContent(type="text", text=str(result))]
      raise ValueError(f"Tool not found: {name}")
  ```
</CodeGroup>

<div id="best-practices">
  ## Mejores prácticas
</div>

Al implementar Herramientas:

1. Proporcione nombres y descripciones claras y precisas
2. Use definiciones detalladas de JSON Schema para los parámetros
3. Incluya ejemplos en las descripciones de las Herramientas para mostrar cómo debe usarlas el modelo
4. Implemente un manejo de errores y validaciones adecuados
5. Use reportes de progreso para operaciones prolongadas
6. Mantenga las operaciones de las Herramientas enfocadas y atómicas
7. Documente las estructuras de los valores de retorno esperados
8. Implemente tiempos de espera adecuados
9. Considere la limitación de frecuencia para operaciones con uso intensivo de Recursos
10. Registre el uso de las Herramientas para depuración y monitoreo

<div id="tool-name-conflicts">
  ### Conflictos de nombres de herramientas
</div>

Las aplicaciones cliente MCP y los proxies de servidor MCP pueden encontrarse con conflictos de nombres de herramientas al construir sus propias listas de herramientas. Por ejemplo, dos servidores MCP conectados, `web1` y `web2`, pueden exponer ambos una herramienta llamada `search_web`.

Las aplicaciones pueden desambiguar las herramientas con una de las siguientes estrategias (entre otras; no es una lista exhaustiva):

* Concatenar un nombre de servidor único definido por el usuario con el nombre de la herramienta, p. ej., `web1___search_web` y `web2___search_web`. Esta estrategia puede ser preferible cuando el usuario ya proporciona nombres de servidor únicos en un archivo de configuración.
* Generar un prefijo aleatorio para el nombre de la herramienta, p. ej., `jrwxs___search_web` y `6cq52___search_web`. Esta estrategia puede ser preferible en proxies de servidor donde no hay nombres únicos definidos por el usuario.
* Usar el URI del servidor como prefijo del nombre de la herramienta, p. ej., `web1.example.com:search_web` y `web2.example.com:search_web`. Esta estrategia puede ser adecuada al trabajar con servidores MCP remotos.

Tenga en cuenta que el nombre proporcionado por el servidor durante el flujo de inicialización no tiene garantía de ser único y, por lo general, no es adecuado para fines de desambiguación.

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

Al exponer Herramientas:

<div id="input-validation">
  ### Validación de entradas
</div>

* Validar todos los parámetros según el esquema
* Sanitizar/depurar rutas de archivos y comandos del sistema
* Validar URL e identificadores externos
* Verificar los tamaños y rangos de los parámetros
* Evitar la inyección de comandos

<div id="access-control">
  ### Control de acceso
</div>

* Implementar autenticación cuando sea necesario
* Usar comprobaciones de autorización adecuadas
* Auditar el uso de herramientas
* Limitar la frecuencia de las solicitudes
* Supervisar para detectar abusos

<div id="error-handling">
  ### Manejo de errores
</div>

* No expongas errores internos a los clientes
* Registra los errores relevantes para la seguridad
* Gestiona adecuadamente los tiempos de espera
* Libera/limpia los recursos después de errores
* Valida los valores de retorno

<div id="tool-discovery-and-updates">
  ## Descubrimiento y actualizaciones de Herramientas
</div>

MCP admite el descubrimiento dinámico de Herramientas:

1. Los clientes pueden enumerar las Herramientas disponibles en cualquier momento
2. Los servidores pueden notificar a los clientes cuando las Herramientas cambian usando `notifications/tools/list_changed`
3. Las Herramientas pueden agregarse o eliminarse durante la ejecución
4. Las definiciones de Herramientas pueden actualizarse (aunque debe hacerse con cuidado)

<div id="error-handling">
  ## Manejo de errores
</div>

Los errores de las Herramientas deben informarse dentro del objeto de resultado, no como errores a nivel del protocolo MCP. Esto permite que el LLM vea el error y, potencialmente, lo maneje. Cuando una herramienta encuentra un error:

1. Establece `isError` en `true` en el resultado
2. Incluye los detalles del error en el array `content`

Aquí tienes un ejemplo de manejo de errores adecuado para las herramientas:

<CodeGroup>
  ```typescript TypeScript
  try {
    // Tool operation
    const result = performOperation();
    return {
      content: [
        {
          type: "text",
          text: `Operation successful: ${result}`,
        },
      ],
    };
  } catch (error) {
    return {
      isError: true,
      content: [
        {
          type: "text",
          text: `Error: ${error.message}`,
        },
      ],
    };
  }
  ```

  ```python Python
  try:
      # Tool operation
      result = perform_operation()
      return types.CallToolResult(
          content=[
              types.TextContent(
                  type="text",
                  text=f"Operation successful: {result}"
              )
          ]
      )
  except Exception as error:
      return types.CallToolResult(
          isError=True,
          content=[
              types.TextContent(
                  type="text",
                  text=f"Error: {str(error)}"
              )
          ]
      )
  ```
</CodeGroup>

Este enfoque permite que el LLM vea que se produjo un error y, potencialmente, tome medidas correctivas o solicite intervención humana.

<div id="tool-annotations">
  ## Anotaciones de herramientas
</div>

Las anotaciones de herramientas proporcionan metadatos adicionales sobre el comportamiento de una herramienta, ayudando a los clientes a comprender cómo presentar y gestionar las herramientas. Estas anotaciones son indicaciones que describen la naturaleza e impacto de una herramienta, pero no deben usarse como base para decisiones de seguridad.

<div id="purpose-of-tool-annotations">
  ### Propósito de las anotaciones de herramientas
</div>

Las anotaciones de herramientas cumplen varios propósitos clave:

1. Proporcionar información específica de UX sin afectar el contexto del modelo
2. Ayudar a los clientes a clasificar y presentar las herramientas de forma adecuada
3. Comunicar información sobre los posibles efectos secundarios de una herramienta
4. Facilitar el desarrollo de interfaces intuitivas para la aprobación de herramientas

<div id="available-tool-annotations">
  ### Anotaciones de herramientas disponibles
</div>

La especificación de MCP define las siguientes anotaciones para Herramientas:

| Anotación         | Tipo    | Predeterminado | Descripción                                                                                                                          |
| ----------------- | ------- | -------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `title`           | string  | -              | Un título legible por humanos para la herramienta, útil para mostrarse en la interfaz de usuario                                    |
| `readOnlyHint`    | boolean | false          | Si es true, indica que la herramienta no modifica su entorno                                                                         |
| `destructiveHint` | boolean | true           | Si es true, la herramienta puede realizar actualizaciones destructivas (solo aplica cuando `readOnlyHint` es false)                 |
| `idempotentHint`  | boolean | false          | Si es true, invocar repetidamente la herramienta con los mismos argumentos no tiene efectos adicionales (solo aplica cuando `readOnlyHint` es false) |
| `openWorldHint`   | boolean | true           | Si es true, la herramienta puede interactuar con un “mundo abierto” de entidades externas                                           |

<div id="example-usage">
  ### Ejemplo de uso
</div>

Así es como definir Herramientas con Anotaciones de herramientas para distintos escenarios:

```typescript
// Una herramienta de búsqueda de solo lectura
{
  name: "web_search",
  description: "Buscar información en la web",
  inputSchema: {
    type: "object",
    properties: {
      query: { type: "string" }
    },
    required: ["query"]
  },
  annotations: {
    title: "Búsqueda web",
    readOnlyHint: true,
    openWorldHint: true
  }
}

// Una herramienta destructiva para eliminar archivos
{
  name: "delete_file",
  description: "Eliminar un archivo del sistema de archivos",
  inputSchema: {
    type: "object",
    properties: {
      path: { type: "string" }
    },
    required: ["path"]
  },
  annotations: {
    title: "Eliminar archivo",
    readOnlyHint: false,
    destructiveHint: true,
    idempotentHint: true,
    openWorldHint: false
  }
}

// Una herramienta no destructiva para crear registros en la base de datos
{
  name: "create_record",
  description: "Crear un nuevo registro en la base de datos",
  inputSchema: {
    type: "object",
    properties: {
      table: { type: "string" },
      data: { type: "object" }
    },
    required: ["table", "data"]
  },
  annotations: {
    title: "Crear registro en la base de datos",
    readOnlyHint: false,
    destructiveHint: false,
    idempotentHint: false,
    openWorldHint: false
  }
}
```

<div id="best-practices-for-tool-annotations">
  ### Mejores prácticas para las Anotaciones de herramientas
</div>

1. **Sé preciso respecto a los efectos secundarios**: Indica claramente si una herramienta modifica su entorno y si esas modificaciones son destructivas.

2. **Usa títulos descriptivos**: Proporciona títulos fáciles de entender que describan claramente el propósito de la herramienta.

3. **Indica la idempotencia correctamente**: Marca las herramientas como idempotentes solo si las llamadas repetidas con los mismos argumentos realmente no tienen ningún efecto adicional.

4. **Establece pistas adecuadas de mundo abierto/cerrado**: Indica si una herramienta interactúa con un sistema cerrado (como una base de datos) o un sistema abierto (como la web).

5. **Recuerda que las anotaciones son pistas**: Todas las propiedades en Anotaciones de herramientas son pistas y no garantizan ofrecer una descripción fiel del comportamiento de la herramienta. Los clientes nunca deben tomar decisiones de seguridad críticas basándose únicamente en anotaciones.

<div id="testing-tools">
  ## Pruebas de herramientas
</div>

Una estrategia de pruebas integral para las Herramientas de MCP debe abarcar:

* **Pruebas funcionales**: Verificar que las herramientas se ejecuten correctamente con entradas válidas y que gestionen adecuadamente las entradas no válidas
* **Pruebas de integración**: Probar la interacción de las herramientas con sistemas externos usando dependencias reales y simuladas
* **Pruebas de seguridad**: Validar autenticación, autorización, depuración/saneamiento de entradas y limitación de velocidad
* **Pruebas de rendimiento**: Comprobar el comportamiento bajo carga, el manejo de tiempos de espera y la liberación/limpieza de recursos
* **Manejo de errores**: Garantizar que las herramientas informen correctamente los errores a través del Protocolo de Contexto del Modelo (MCP) y liberen/limpien los recursos
