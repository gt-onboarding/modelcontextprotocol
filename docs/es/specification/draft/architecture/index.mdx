<div id="enable-section-numbers" />

El Protocolo de Contexto del Modelo (MCP) sigue una arquitectura cliente-anfitrión-servidor en la que cada
anfitrión puede ejecutar múltiples instancias de Cliente MCP. Esta arquitectura permite a los usuarios integrar capacidades de IA
en distintas aplicaciones, manteniendo límites de seguridad claros y
separando las responsabilidades. Basado en JSON-RPC 2.0, MCP proporciona un protocolo de sesión con estado centrado
en el intercambio de contexto y la coordinación de muestreo entre clientes y servidores.

<div id="host">
  ### Anfitrión
</div>

El proceso anfitrión actúa como contenedor y coordinador:

* Crea y gestiona múltiples instancias de clientes
* Controla los permisos de conexión de los clientes y su ciclo de vida
* Aplica políticas de seguridad y requisitos de consentimiento
* Gestiona las decisiones de autorización del usuario
* Coordina la integración con IA/LLM y el muestreo
* Administra la agregación de contexto entre clientes

<div id="clients">
  ### Clientes
</div>

Cada cliente lo crea el anfitrión y mantiene una conexión aislada con un servidor:

* Establece una sesión con estado por servidor
* Gestiona la negociación del protocolo y el intercambio de capacidades
* Enruta los mensajes del protocolo en ambos sentidos
* Gestiona suscripciones y notificaciones
* Mantiene límites de seguridad entre servidores

Una aplicación anfitriona crea y administra múltiples clientes, y cada cliente tiene una relación 1:1 con un servidor específico.

<div id="servers">
  ### Servidores
</div>

Los servidores proporcionan contexto y capacidades especializadas:

* Exponen Recursos, Herramientas e Indicaciones mediante primitivas de MCP
* Operan de forma independiente con responsabilidades específicas
* Solicitan Muestreo a través de interfaces del cliente
* Deben respetar las restricciones de seguridad
* Pueden ser procesos locales o servicios remotos

<div id="design-principles">
  ## Principios de diseño
</div>

MCP se basa en varios principios clave de diseño que guían su arquitectura e
implementación:

1. **Los servidores deben ser extremadamente fáciles de crear**
   * Las aplicaciones anfitrionas se encargan de la orquestación compleja
   * Los servidores se centran en capacidades específicas y bien definidas
   * Interfaces simples minimizan el esfuerzo de implementación
   * Una separación clara permite mantener el código

2. **Los servidores deben ser altamente componibles**
   * Cada servidor ofrece funcionalidad enfocada de forma aislada
   * Varios servidores pueden combinarse sin fricciones
   * Un protocolo compartido habilita la interoperabilidad
   * Un diseño modular facilita la extensibilidad

3. **Los servidores no deben poder leer toda la conversación ni “ver” otros
   servidores**
   * Los servidores reciben solo la información contextual necesaria
   * El historial completo de la conversación permanece con el anfitrión
   * Cada conexión de servidor mantiene el aislamiento
   * Las interacciones entre servidores están controladas por el anfitrión
   * El proceso del anfitrión impone límites de seguridad

4. **Las funciones pueden añadirse a servidores y clientes de forma progresiva**
   * El protocolo núcleo proporciona la funcionalidad mínima necesaria
   * Capacidades adicionales pueden negociarse según sea necesario
   * Servidores y clientes evolucionan de forma independiente
   * El protocolo está diseñado para una futura extensibilidad
   * Se mantiene la compatibilidad con versiones anteriores

<div id="capability-negotiation">
  ## Negociación de capacidades
</div>

El Protocolo de Contexto del Modelo (MCP) utiliza un sistema de negociación basado en capacidades, donde los Clientes MCP y los Servidores MCP declaran explícitamente las funciones que admiten durante la inicialización. Las capacidades determinan qué funciones y primitivas del protocolo están disponibles durante una sesión.

* Los servidores declaran capacidades como suscripciones a Recursos, compatibilidad con Herramientas e Indicaciones (plantillas)
* Los clientes declaran capacidades como compatibilidad con Muestreo y manejo de notificaciones
* Ambas partes deben respetar las capacidades declaradas durante toda la sesión
* Pueden negociarse capacidades adicionales mediante extensiones del protocolo

```mermaid
sequenceDiagram
    participant Host
    participant Client
    participant Server

    Host->>+Client: Initialize client
    Client->>+Server: Initialize session with capabilities
    Server-->>Client: Respond with supported capabilities

    Note over Host,Server: Active Session with Negotiated Features

    loop Client Requests
        Host->>Client: User- or model-initiated action
        Client->>Server: Request (tools/resources)
        Server-->>Client: Response
        Client-->>Host: Update UI or respond to model
    end

    loop Server Requests
        Server->>Client: Request (sampling)
        Client->>Host: Forward to AI
        Host-->>Client: AI response
        Client-->>Server: Response
    end

    loop Notifications
        Server--)Client: Resource updates
        Client--)Server: Status changes
    end

    Host->>Client: Terminate
    Client->>-Server: End session
    deactivate Server
```

Cada capacidad habilita funciones específicas del protocolo para usarse durante la sesión. Por ejemplo:

* Las [funciones del servidor](/es/specification/draft/server) implementadas deben anunciarse en las capacidades del servidor
* Emitir notificaciones de suscripción a Recursos requiere que el servidor declare compatibilidad con suscripciones
* La invocación de Herramientas requiere que el servidor declare capacidades de herramientas
* El [Muestreo](/es/specification/draft/client) requiere que el cliente declare compatibilidad en sus capacidades

Esta negociación de capacidades garantiza que los clientes y los servidores tengan una comprensión clara de la funcionalidad admitida, a la vez que mantiene la extensibilidad del protocolo.
