<div id="enable-section-numbers" />

<Info>**Revisión del protocolo**: borrador</Info>

El Protocolo de Contexto de Modelo (MCP) ofrece una manera estandarizada para que los servidores envíen mensajes de registro estructurados a los clientes. Los clientes pueden controlar la verbosidad del registro estableciendo niveles mínimos, mientras que los servidores envían notificaciones con niveles de severidad, nombres de registrador opcionales y datos arbitrarios serializables en JSON.

<div id="user-interaction-model">
  ## Modelo de interacción con el usuario
</div>

Las implementaciones pueden exponer el registro mediante cualquier patrón de interfaz que se ajuste a sus necesidades: el propio protocolo no impone ningún modelo específico de interacción con el usuario.

<div id="capabilities">
  ## Capacidades
</div>

Los servidores que emiten notificaciones de registros **DEBEN** declarar la capacidad `logging`:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

<div id="log-levels">
  ## Niveles de registro
</div>

El protocolo sigue los niveles de gravedad estándar de syslog especificados en
[RFC 5424](https://datatracker.ietf.org/doc/html/rfc5424#section-6.2.1):

| Nivel     | Descripción                       | Caso de uso de ejemplo     |
| --------- | --------------------------------- | -------------------------- |
| debug     | Información detallada de depuración | Puntos de entrada/salida de funciones |
| info      | Mensajes informativos generales   | Actualizaciones del progreso de la operación |
| notice    | Eventos normales pero significativos | Cambios de configuración   |
| warning   | Condiciones de advertencia        | Uso de funcionalidades obsoletas |
| error     | Condiciones de error              | Fallos de operación        |
| critical  | Condiciones críticas              | Fallos de componentes del sistema |
| alert     | Debe actuarse de inmediato        | Corrupción de datos detectada |
| emergency | El sistema es inutilizable        | Fallo total del sistema    |

<div id="protocol-messages">
  ## Mensajes del protocolo
</div>

<div id="setting-log-level">
  ### Configuración del nivel de registros
</div>

Para establecer el nivel mínimo de registros, los clientes **PUEDEN** enviar una solicitud `logging/setLevel`:

**Solicitud:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "logging/setLevel",
  "params": {
    "level": "info"
  }
}
```

<div id="log-message-notifications">
  ### Notificaciones de mensajes de registro
</div>

Los servidores envían mensajes de registro mediante notificaciones `notifications/message`:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/message",
  "params": {
    "level": "error",
    "logger": "database",
    "data": {
      "error": "Connection failed",
      "details": {
        "host": "localhost",
        "port": 5432
      }
    }
  }
}
```

<div id="message-flow">
  ## Flujo de mensajes
</div>

```mermaid
sequenceDiagram
    participant Client
    participant Server

    Note over Client,Server: Configurar el registro
    Client->>Server: logging/setLevel (info)
    Server-->>Client: Resultado vacío

    Note over Client,Server: Actividad del servidor
    Server--)Client: notifications/message (info)
    Server--)Client: notifications/message (warning)
    Server--)Client: notifications/message (error)

    Note over Client,Server: Cambio de nivel
    Client->>Server: logging/setLevel (error)
    Server-->>Client: Resultado vacío
    Note over Server: Solo envía errores<br/>y niveles superiores
```

<div id="error-handling">
  ## Manejo de errores
</div>

Los servidores **DEBERÍAN** devolver errores estándar de JSON-RPC para casos comunes de error:

- Nivel de registro no válido: `-32602` (Parámetros no válidos)
- Errores de configuración: `-32603` (Error interno)

<div id="implementation-considerations">
  ## Consideraciones de implementación
</div>

1. Los servidores **DEBERÍAN**:
   - Limitar la frecuencia de los mensajes de registro
   - Incluir el contexto relevante en el campo de datos
   - Usar nombres de registrador coherentes
   - Eliminar información confidencial

2. Los clientes **PUEDEN**:
   - Mostrar mensajes de registro en la interfaz de usuario
   - Implementar filtrado/búsqueda de registros
   - Representar visualmente la gravedad
   - Conservar mensajes de registro

<div id="security">
  ## Seguridad
</div>

1. Los mensajes de los registros **NO DEBEN** contener:
   - Credenciales o secretos
   - Información de identificación personal
   - Detalles internos del sistema que puedan facilitar ataques

2. Las implementaciones **DEBERÍAN**:
   - Limitar la frecuencia de los mensajes
   - Validar todos los campos de datos
   - Controlar el acceso a los registros
   - Supervisar la presencia de contenido sensible
