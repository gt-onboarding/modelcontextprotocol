<div id="enable-section-numbers" />

<Info>**Revisión del protocolo**: borrador</Info>

El Protocolo de Contexto del Modelo (MCP) establece una forma estandarizada para que los servidores soliciten muestreo de LLM («completions» o «generations») a través de los clientes. Este flujo permite que los clientes mantengan el control sobre el acceso al modelo, su selección y los permisos, a la vez que permite a los servidores aprovechar capacidades de IA, sin necesidad de claves de API en el servidor. Los servidores pueden solicitar interacciones basadas en texto, audio o imágenes y, opcionalmente, incluir contexto de Servidores MCP en sus indicaciones.

<div id="user-interaction-model">
  ## Modelo de interacción con el usuario
</div>

El Muestreo en MCP permite que los servidores implementen comportamientos de agente al habilitar que las llamadas al LLM ocurran de forma *anidada* dentro de otras funciones del Servidor MCP.

Las implementaciones pueden exponer el Muestreo mediante cualquier patrón de interfaz que se adapte a sus necesidades; el protocolo en sí no exige ningún modelo específico de interacción con el usuario.

<Warning>
  Por motivos de confianza y seguridad, **DEBERÍA** haber siempre una persona en el circuito con la capacidad de rechazar solicitudes de Muestreo.

  Las aplicaciones **DEBERÍAN**:

  * Proporcionar una interfaz que haga fácil e intuitiva la revisión de solicitudes de Muestreo
  * Permitir que los usuarios vean y editen las Indicaciones antes de enviarlas
  * Presentar las respuestas generadas para su revisión antes de su entrega
</Warning>

<div id="capabilities">
  ## Capacidades
</div>

Los clientes que admitan Muestreo **DEBEN** declarar la capacidad `sampling` durante la
[inicialización](/es/specification/draft/basic/lifecycle#initialization):

```json
{
  "capabilities": {
    "sampling": {}
  }
}
```

  ## Mensajes del protocolo

<div id="creating-messages">
  ### Creación de mensajes
</div>

Para solicitar la generación por parte de un modelo de lenguaje, los servidores envían una solicitud `sampling/createMessage`:

**Solicitud:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "sampling/createMessage",
  "params": {
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "What is the capital of France?"
        }
      }
    ],
    "modelPreferences": {
      "hints": [
        {
          "name": "claude-3-sonnet"
        }
      ],
      "intelligencePriority": 0.8,
      "speedPriority": 0.5
    },
    "systemPrompt": "You are a helpful assistant.",
    "maxTokens": 100
  }
}
```

**Respuesta:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "role": "assistant",
    "content": {
      "type": "text",
      "text": "The capital of France is Paris."
    },
    "model": "claude-3-sonnet-20240307",
    "stopReason": "endTurn"
  }
}
```

<div id="message-flow">
  ## Flujo de mensajes
</div>

```mermaid
sequenceDiagram
    participant Server
    participant Client
    participant User
    participant LLM

    Note over Server,Client: El servidor inicia el muestreo
    Server->>Client: sampling/createMessage

    Note over Client,User: Revisión con participación humana
    Client->>User: Presentar la solicitud para su aprobación
    User-->>Client: Revisar y aprobar/modificar

    Note over Client,LLM: Interacción con el modelo
    Client->>LLM: Reenviar la solicitud aprobada
    LLM-->>Client: Devolver la generación

    Note over Client,User: Revisión de la respuesta
    Client->>User: Presentar la respuesta para su aprobación
    User-->>Client: Revisar y aprobar/modificar

    Note over Server,Client: Completar la solicitud
    Client-->>Server: Devolver la respuesta aprobada
```

<div id="model-preferences">
  ### Preferencias de modelos
</div>

La selección de modelos en MCP requiere una abstracción cuidadosa, ya que servidores y clientes pueden usar distintos proveedores de IA con catálogos de modelos diferentes. Un servidor no puede simplemente solicitar un modelo específico por nombre, ya que el cliente podría no tener acceso a ese modelo exacto o preferir usar un modelo equivalente de otro proveedor.

Para resolver esto, MCP implementa un sistema de preferencias que combina prioridades de capacidades abstractas con indicaciones opcionales de modelos:

<div id="capability-priorities">
  #### Prioridades de capacidades
</div>

Los servidores expresan sus necesidades mediante tres valores de prioridad normalizados (0-1):

* `costPriority`: ¿Qué tan importante es minimizar los costos? Valores más altos favorecen modelos más económicos.
* `speedPriority`: ¿Qué tan importante es la baja latencia? Valores más altos favorecen modelos más rápidos.
* `intelligencePriority`: ¿Qué tan importantes son las capacidades avanzadas? Valores más altos favorecen modelos más capaces.

<div id="model-hints">
  #### Sugerencias de modelo
</div>

Si bien las prioridades ayudan a seleccionar modelos según sus características, `hints` permiten que los servidores
propongan modelos específicos o familias de modelos:

* Las sugerencias se tratan como subcadenas que pueden coincidir de forma flexible con nombres de modelos
* Se evalúan múltiples sugerencias en orden de preferencia
* Los clientes **PUEDEN** asignar sugerencias a modelos equivalentes de diferentes proveedores
* Las sugerencias son orientativas—la selección final del modelo la realizan los clientes

Por ejemplo:

```json
{
  "hints": [
    { "name": "claude-3-sonnet" }, // Preferir modelos de la clase Sonnet
    { "name": "claude" } // En su defecto, cualquier modelo Claude
  ],
  "costPriority": 0.3, // El costo es menos importante
  "speedPriority": 0.8, // La velocidad es muy importante
  "intelligencePriority": 0.5, // Se necesitan capacidades moderadas
}
```

El cliente procesa estas preferencias para seleccionar un modelo adecuado de entre sus opciones disponibles.
Por ejemplo, si el cliente no tiene acceso a modelos Claude pero sí a Gemini, podría asignar la sugerencia de sonnet a `gemini-1.5-pro` según capacidades similares.

<div id="error-handling">
  ## Manejo de errores
</div>

Los clientes **DEBERÍAN** devolver errores para casos de falla comunes:

Ejemplo de error:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -1,
    "message": "User rejected sampling request"
  }
}
```

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

1. Los clientes **DEBERÍAN** implementar controles de aprobación del usuario
2. Ambas partes **DEBERÍAN** validar el contenido de los mensajes
3. Los clientes **DEBERÍAN** respetar las indicaciones de preferencia del modelo
4. Los clientes **DEBERÍAN** implementar límites de tasa
5. Ambas partes **DEBEN** manejar los datos sensibles de manera adecuada
