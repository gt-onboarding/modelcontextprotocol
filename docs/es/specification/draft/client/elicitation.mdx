<Info>**Revisión del protocolo**: borrador</Info>

<Note>
  Elicitación se introduce por primera vez en esta versión de la especificación del Protocolo de Contexto de Modelo (MCP) y su diseño puede evolucionar en futuras versiones del protocolo.
</Note>

El Protocolo de Contexto de Modelo (MCP) proporciona una forma estandarizada para que los servidores soliciten
información adicional a los usuarios a través del cliente durante las interacciones. Este flujo permite que los clientes
mantengan el control sobre las interacciones del usuario y el intercambio de datos, al tiempo que permite a los servidores recopilar
la información necesaria de forma dinámica.
Los servidores solicitan datos estructurados a los usuarios mediante esquemas JSON para validar las respuestas.

<div id="user-interaction-model">
  ## Modelo de interacción con el usuario
</div>

La Elicitación en MCP permite que los servidores implementen flujos de trabajo interactivos al habilitar que las solicitudes de entrada del usuario ocurran *anidadas* dentro de otras funciones del Servidor MCP.

Las implementaciones pueden exponer la elicitación mediante cualquier patrón de interfaz que se adapte a sus necesidades; el protocolo en sí no impone ningún modelo específico de interacción con el usuario.

<Warning>
  Por motivos de confianza, seguridad y protección:

  * Los servidores **NO DEBEN** usar la elicitación para solicitar información sensible.

  Las aplicaciones **DEBERÍAN**:

  * Proporcionar una interfaz que deje claro qué servidor está solicitando información
  * Permitir que los usuarios revisen y modifiquen sus respuestas antes de enviarlas
  * Respetar la privacidad del usuario y proporcionar opciones claras para rechazar y cancelar
</Warning>

<div id="capabilities">
  ## Capacidades
</div>

Los clientes que admitan Elicitación **DEBEN** declarar la capacidad `elicitation` durante la
[inicialización](/es/specification/draft/basic/lifecycle#initialization):

```json
{
  "capabilities": {
    "elicitation": {}
  }
}
```

  ## Mensajes del protocolo

<div id="creating-elicitation-requests">
  ### Creación de solicitudes de Elicitación
</div>

Para solicitar información a un usuario, los servidores envían una solicitud `elicitation/create`:

<div id="simple-text-request">
  #### Solicitud de texto simple
</div>

**Solicitud:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "elicitation/create",
  "params": {
    "message": "Por favor, indica tu nombre de usuario de GitHub",
    "requestedSchema": {
      "type": "object",
      "properties": {
        "name": {
          "type": "string"
        }
      },
      "required": ["name"]
    }
  }
}
```

**Respuesta:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "action": "accept",
    "content": {
      "name": "octocat"
    }
  }
}
```

<div id="structured-data-request">
  #### Solicitud de datos estructurados
</div>

**Solicitud:**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "elicitation/create",
  "params": {
    "message": "Please provide your contact information",
    "requestedSchema": {
      "type": "object",
      "properties": {
        "name": {
          "type": "string",
          "description": "Tu nombre completo"
        },
        "email": {
          "type": "string",
          "format": "email",
          "description": "Tu dirección de correo electrónico"
        },
        "age": {
          "type": "number",
          "minimum": 18,
          "description": "Tu edad"
        }
      },
      "required": ["name", "email"]
    }
  }
}
```

**Respuesta:**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "action": "accept",
    "content": {
      "name": "Monalisa Octocat",
      "email": "octocat@github.com",
      "age": 30
    }
  }
}
```

**Ejemplo de respuesta de rechazo:**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "action": "decline"
  }
}
```

**Ejemplo de respuesta de cancelación:**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "action": "cancel"
  }
}
```

<div id="message-flow">
  ## Flujo de mensajes
</div>

```mermaid
sequenceDiagram
    participant User
    participant Client
    participant Server

    Note over Server,Client: El servidor inicia la elicitación
    Server->>Client: elicitation/create

    Note over Client,User: Interacción con el usuario
    Client->>User: Mostrar la interfaz de elicitación
    User-->>Client: Proporcionar la información solicitada

    Note over Server,Client: Completar la solicitud
    Client-->>Server: Devolver la respuesta del usuario

    Note over Server: Continuar el procesamiento con la nueva información
```

<div id="request-schema">
  ## Esquema de solicitud
</div>

El campo `requestedSchema` permite a los servidores definir la estructura de la respuesta esperada utilizando un subconjunto restringido de JSON Schema. Para simplificar la experiencia del usuario del cliente, los esquemas de elicitación se limitan a objetos planos con propiedades primitivas únicamente:

```json
"requestedSchema": {
  "type": "object",
  "properties": {
    "propertyName": {
      "type": "string",
      "title": "Nombre para mostrar",
      "description": "Descripción de la propiedad"
    },
    "anotherProperty": {
      "type": "number",
      "minimum": 0,
      "maximum": 100
    }
  },
  "required": ["propertyName"]
}
```

<div id="supported-schema-types">
  ### Tipos de esquemas compatibles
</div>

El esquema se limita a estos tipos primitivos:

1. **Esquema de cadena**

   ```json
   {
     "type": "string",
     "title": "Display Name",
     "description": "Description text",
     "minLength": 3,
     "maxLength": 50,
     "pattern": "^[A-Za-z]+$",
     "format": "email",
     "default": "user@example.com"
   }
   ```

   Formatos admitidos: `email`, `uri`, `date`, `date-time`

2. **Esquema numérico**

   ```json
   {
     "type": "number", // or "integer"
     "title": "Display Name",
     "description": "Description text",
     "minimum": 0,
     "maximum": 100,
     "default": 50
   }
   ```

3. **Esquema booleano**

   ```json
   {
     "type": "boolean",
     "title": "Display Name",
     "description": "Description text",
     "default": false
   }
   ```

4. **Esquema de enumeración**
   ```json
   {
     "type": "string",
     "title": "Display Name",
     "description": "Description text",
     "enum": ["option1", "option2", "option3"],
     "enumNames": ["Option 1", "Option 2", "Option 3"],
     "default": "option1"
   }
   ```

Los clientes pueden usar este esquema para:

1. Generar formularios de entrada adecuados
2. Validar la entrada del usuario antes de enviar
3. Ofrecer una mejor orientación a los usuarios

Todos los tipos primitivos admiten valores predeterminados opcionales para proporcionar puntos de partida razonables. Los clientes que admitan valores predeterminados DEBEN rellenar previamente los campos del formulario con estos valores.

Tenga en cuenta que las estructuras anidadas complejas, los arreglos de objetos y otras funciones avanzadas de JSON Schema no son compatibles intencionalmente para simplificar la experiencia del usuario en el cliente.

<div id="response-actions">
  ## Acciones de respuesta
</div>

Las respuestas de Elicitación usan un modelo de tres acciones para distinguir claramente entre diferentes acciones del usuario:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "action": "accept", // or "decline" or "cancel"
    "content": {
      "propertyName": "value",
      "anotherProperty": 42
    }
  }
}
```

Las tres acciones de respuesta son:

1. **Aceptar** (`action: "accept"`): El usuario aprobó explícitamente y envió datos
   * El campo `content` contiene los datos enviados que cumplen con el esquema solicitado
   * Ejemplo: El usuario hizo clic en “Enviar”, “OK”, “Confirmar”, etc.

2. **Rechazar** (`action: "decline"`): El usuario rechazó explícitamente la solicitud
   * El campo `content` suele omitirse
   * Ejemplo: El usuario hizo clic en “Rechazar”, “Declinar”, “No”, etc.

3. **Cancelar** (`action: "cancel"`): El usuario salió sin tomar una decisión explícita
   * El campo `content` suele omitirse
   * Ejemplo: El usuario cerró el cuadro de diálogo, hizo clic fuera, presionó Escape, etc.

Los servidores deben manejar cada estado de manera adecuada:

* **Aceptar**: Procesar los datos enviados
* **Rechazar**: Manejar el rechazo explícito (p. ej., ofrecer alternativas)
* **Cancelar**: Manejar el descarte (p. ej., solicitar nuevamente más tarde)

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

1. Los servidores **NO DEBEN** solicitar información sensible mediante Elicitación
2. Los clientes **DEBERÍAN** implementar controles de aprobación del usuario
3. Ambas partes **DEBERÍAN** validar el contenido de la Elicitación con respecto al esquema proporcionado
4. Los clientes **DEBERÍAN** indicar claramente qué servidor está solicitando la información
5. Los clientes **DEBERÍAN** permitir que los usuarios rechacen solicitudes de Elicitación en cualquier momento
6. Los clientes **DEBERÍAN** implementar limitación de frecuencia
7. Los clientes **DEBERÍAN** presentar las solicitudes de Elicitación de manera que quede claro qué información se solicita y por qué
