<Info>**Revisión del protocolo**: borrador</Info>

El Protocolo de Contexto del Modelo (MCP) consta de varios componentes clave que funcionan en conjunto:

* **Protocolo base**: tipos de mensajes JSON-RPC fundamentales
* **Gestión del ciclo de vida**: inicialización de la conexión, negociación de capacidades y
  control de la sesión
* **Autorización**: marco de autenticación y autorización para transportes basados en HTTP
* **Funciones del servidor**: Recursos, Indicaciones y Herramientas expuestos por los servidores
* **Funciones del cliente**: Muestreo y listas de directorios raíz proporcionadas por los clientes
* **Utilidades**: aspectos transversales como el registro y el autocompletado de argumentos

Todas las implementaciones **DEBEN** admitir el protocolo base y los componentes de gestión del ciclo de vida. Los demás componentes **PUEDEN** implementarse según las necesidades específicas de la
aplicación.

Estas capas del protocolo establecen una separación clara de responsabilidades y, al mismo tiempo, habilitan interacciones ricas entre clientes y servidores. El diseño modular permite a las implementaciones
admitir exactamente las funciones que necesitan.

<div id="messages">
  ## Mensajes
</div>

Todos los mensajes entre clientes y servidores de MCP **DEBEN** cumplir la especificación
[JSON-RPC 2.0](https://www.jsonrpc.org/specification). El protocolo define
estos tipos de mensajes:

<div id="requests">
  ### Solicitudes
</div>

Las solicitudes se envían del cliente al servidor o viceversa para iniciar una operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las solicitudes **DEBEN** incluir un identificador de tipo cadena o entero.
* A diferencia de JSON-RPC base, el identificador **NO DEBE** ser `null`.
* El identificador de la solicitud **NO DEBE** haberse usado previamente por el solicitante dentro de la misma sesión.

<div id="responses">
  ### Respuestas
</div>

Las respuestas se envían a las solicitudes y contienen el resultado o el error de la operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* Las respuestas **DEBEN** incluir el mismo ID que la solicitud a la que corresponden.
* Las **respuestas** se clasifican además en **resultados correctos** o
  **errores**. Se **DEBE** establecer un `result` o un `error`. Una respuesta **NO DEBE**
  establecer ambos.
* Los resultados **PUEDEN** tener cualquier estructura de objeto JSON, mientras que los errores **DEBEN** incluir como mínimo un
  código de error y un mensaje.
* Los códigos de error **DEBEN** ser enteros.

<div id="notifications">
  ### Notificaciones
</div>

Las notificaciones se envían del cliente al servidor o viceversa, como mensajes unidireccionales.
El receptor **NO DEBE** enviar una respuesta.

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las notificaciones **NO DEBEN** incluir un ID.

<div id="auth">
  ## Autenticación
</div>

MCP proporciona un marco de [autorización](/es/specification/draft/basic/authorization) para usar con HTTP.
Las implementaciones que utilicen un transporte basado en HTTP **DEBERÍAN** ajustarse a esta especificación,
mientras que las implementaciones que usen el transporte STDIO **NO DEBERÍAN** seguirla,
sino obtener las credenciales del entorno.

Además, clientes y servidores **PUEDEN** negociar sus propias estrategias personalizadas de autenticación y
autorización.

Para participar en debates y contribuir a la evolución de los mecanismos de autenticación de MCP, únete
a
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
para ayudar a dar forma al futuro del protocolo.

<div id="schema">
  ## Esquema
</div>

La especificación completa del protocolo se define como un
[esquema de TypeScript](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts).
Este es el referente oficial para todos los mensajes y estructuras del protocolo.

También hay un
[JSON Schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.json),
que se genera automáticamente a partir de la referencia en TypeScript, para usar con
diversas herramientas automatizadas.

<div id="general-fields">
  ### Campos generales
</div>

<div id="_meta">
  #### `_meta`
</div>

La propiedad/parámetro `_meta` está reservada por el MCP para permitir que clientes y servidores
adjunten metadatos adicionales a sus interacciones.

Ciertos nombres de clave están reservados por el MCP para metadatos a nivel de protocolo, como se especifica a continuación;
las implementaciones NO DEBEN hacer suposiciones sobre los valores en esas claves.

Además, las definiciones en el [esquema](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts)
pueden reservar nombres concretos para metadatos con fines específicos, según se declare en dichas definiciones.

Nombre de clave: los nombres de clave `_meta` válidos tienen dos segmentos: un **prefijo** opcional y un **nombre**.

Prefijo:

* Si se especifica, DEBE ser una serie de etiquetas separadas por puntos (`.`), seguida de una barra inclinada (`/`).
  * Las etiquetas DEBEN comenzar con una letra y terminar con una letra o un dígito; los caracteres intermedios pueden ser letras, dígitos o guiones (`-`).
* Cualquier prefijo que comience con cero o más etiquetas válidas, seguido de `modelcontextprotocol` o `mcp`, y luego cualquier etiqueta válida,
  está reservado para uso del MCP.
  * Por ejemplo: `modelcontextprotocol.io/`, `mcp.dev/`, `api.modelcontextprotocol.org/` y `tools.mcp.com/` están todos reservados.

Nombre:

* A menos que esté vacío, DEBE comenzar y terminar con un carácter alfanumérico (`[a-z0-9A-Z]`).
* PUEDE contener guiones (`-`), guiones bajos (`_`), puntos (`.`) y caracteres alfanuméricos en medio.

<div id="icons">
  #### `icons`
</div>

La propiedad `icons` proporciona una forma estandarizada para que los servidores expongan identificadores visuales de sus Recursos, Herramientas, Indicaciones e implementaciones. Los iconos mejoran las interfaces de usuario al aportar contexto visual y facilitar el descubrimiento de la funcionalidad disponible.

Los iconos se representan como una matriz de objetos `Icon`, donde cada icono incluye:

* `src`: Un URI que apunta al recurso del icono (obligatorio). Puede ser:
  * Una URL HTTP/HTTPS que apunte a un archivo de imagen
  * Un URI de datos con la imagen codificada en base64
* `mimeType`: Tipo MIME opcional si el tipo declarado por el servidor falta o es genérico
* `sizes`: Especificación de tamaño opcional (p. ej., &quot;48x48&quot;; &quot;any&quot; para formatos escalables como SVG; o &quot;48x48 96x96&quot; para múltiples tamaños)

**Compatibilidad de tipos MIME requerida:**

Los clientes que admitan el renderizado de iconos **DEBEN** admitir al menos los siguientes tipos MIME:

* `image/png` - Imágenes PNG (seguras, compatibilidad universal)
* `image/jpeg` (y `image/jpg`) - Imágenes JPEG (seguras, compatibilidad universal)

Los clientes que admitan el renderizado de iconos **DEBERÍAN** también admitir:

* `image/svg+xml` - Imágenes SVG (escalables, pero requieren precauciones de seguridad como se indica a continuación)
* `image/webp` - Imágenes WebP (formato moderno y eficiente)

**Consideraciones de seguridad:**

Los consumidores de metadatos de iconos **DEBEN** tomar las precauciones de seguridad correspondientes al manejar iconos para evitar compromisos:

* Tratar los metadatos y los bytes de los iconos como entradas no confiables y protegerse frente a riesgos de red, privacidad y análisis.
* Asegurarse de que el URI del icono sea HTTPS o un URI `data:`. Los clientes **DEBEN** rechazar URIs de iconos que usen esquemas y redirecciones no seguros, como `javascript:`, `file:`, `ftp:`, `ws:` o esquemas de URI de aplicaciones locales.
  * No permitir cambios de esquema ni redirecciones a hosts de orígenes distintos.
* Ser resilientes frente a ataques de agotamiento de recursos derivados de imágenes de gran tamaño, dimensiones excesivas o demasiados fotogramas (p. ej., en GIF).
  * Los consumidores **PUEDEN** establecer límites para el tamaño de la imagen y del contenido.
* Obtener iconos sin credenciales. No enviar cookies, encabezados `Authorization` ni credenciales del cliente.
* Verificar que los URIs de los iconos provengan del mismo origen que el servidor. Esto minimiza el riesgo de exponer datos o información de seguimiento a terceros.
* Actuar con cautela al obtener y renderizar iconos, ya que la carga útil **PUEDE** contener contenido ejecutable (p. ej., SVG con [JavaScript incrustado](https://www.w3.org/TR/SVG11/script.html) o [capacidades ampliadas](https://www.w3.org/TR/SVG11/extend.html)).
  * Los consumidores **PUEDEN** optar por deshabilitar tipos de archivo específicos o, de lo contrario, sanitizar los archivos de iconos antes de renderizarlos.
* Validar los tipos MIME y el contenido del archivo antes de renderizar. Tratar la información del tipo MIME como orientativa. Detectar el tipo de contenido mediante magic bytes; rechazar en caso de discrepancia o tipos desconocidos.
  * Mantener una lista estricta de tipos de imagen permitidos.

**Uso:**

Los iconos se pueden adjuntar a:

* `Implementation`: Identificador visual de la implementación del Servidor/Cliente MCP
* `Tool`: Representación visual de la funcionalidad de la Herramienta
* `Prompt`: Icono para mostrar junto a las plantillas de Indicaciones
* `Resource`: Indicador visual para diferentes tipos de Recursos

Se pueden proporcionar varios iconos para admitir diferentes contextos de visualización y resoluciones. Los clientes deben seleccionar el icono más adecuado según los requisitos de su interfaz.
