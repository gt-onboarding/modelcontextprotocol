title: Raíces

<div id="enable-section-numbers" />

<Info>**Revisión del protocolo**: 2025-06-18</Info>

El Protocolo de Contexto de Modelo (MCP) ofrece una forma estandarizada para que los clientes expongan
“raíces” del sistema de archivos a los servidores. Las raíces definen los límites dentro de los cuales pueden operar los servidores
en el sistema de archivos, lo que les permite entender a qué directorios y archivos tienen
acceso. Los servidores pueden solicitar la lista de raíces a los clientes compatibles y recibir
notificaciones cuando esa lista cambie.

<div id="user-interaction-model">
  ## Modelo de interacción con el usuario
</div>

Las Raíces en MCP normalmente se exponen a través de interfaces de configuración del espacio de trabajo o del proyecto.

Por ejemplo, las implementaciones podrían ofrecer un selector de espacio de trabajo/proyecto que permita a los usuarios
elegir los directorios y archivos a los que el servidor debería tener acceso. Esto se puede combinar con
la detección automática del espacio de trabajo a partir de sistemas de control de versiones o archivos de proyecto.

Sin embargo, las implementaciones pueden exponer las Raíces mediante cualquier patrón de interfaz que
se ajuste a sus necesidades; el protocolo en sí no impone ningún modelo específico de
interacción con el usuario.

<div id="capabilities">
  ## Capacidades
</div>

Los clientes que admiten Raíces **DEBEN** declarar la capacidad `roots` durante la
[inicialización](/es/specification/2025-06-18/basic/lifecycle#initialization):

```json
{
  "capabilities": {
    "roots": {
      "listChanged": true
    }
  }
}
```

`listChanged` indica si el cliente enviará notificaciones cuando cambie la lista de Raíces.

<div id="protocol-messages">
  ## Mensajes del protocolo
</div>

<div id="listing-roots">
  ### Listar Raíces
</div>

Para obtener las Raíces, los servidores envían una solicitud `roots/list`:

**Solicitud:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "roots/list"
}
```

**Respuesta:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "roots": [
      {
        "uri": "file:///home/user/projects/myproject",
        "name": "My Project"
      }
    ]
  }
}
```

<div id="root-list-changes">
  ### Cambios en la lista de Raíces
</div>

Cuando cambian las Raíces, los clientes que admiten `listChanged` **DEBEN** enviar una notificación:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/roots/list_changed"
}
```

<div id="data-types">
  ## Tipos de datos
</div>

<div id="root">
  ### Raíz
</div>

Una definición de raíz incluye:

* `uri`: Identificador único de la raíz. Según la especificación actual, **DEBE** ser un URI `file://`.
* `name`: Nombre opcional, legible por personas, para fines de visualización.

Ejemplos de raíces para distintos casos de uso:

<div id="project-directory">
  #### Directorio del proyecto
</div>

```json
{
  "uri": "file:///home/user/projects/myproject",
  "name": "My Project"
}
```

<div id="multiple-repositories">
  #### Varios repositorios
</div>

```json
[
  {
    "uri": "file:///home/user/repos/frontend",
    "name": "Repositorio de frontend"
  },
  {
    "uri": "file:///home/user/repos/backend",
    "name": "Repositorio de backend"
  }
]
```

<div id="error-handling">
  ## Manejo de errores
</div>

Los clientes **DEBERÍAN** devolver errores estándar de JSON-RPC para casos comunes de fallo:

* El cliente no admite Raíces: `-32601` (Method not found)
* Errores internos: `-32603`

Ejemplo de error:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Roots not supported",
    "data": {
      "reason": "Client does not have roots capability"
    }
  }
}
```

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

1. Los clientes **DEBEN**:
   * Exponer únicamente Raíces con los permisos adecuados
   * Validar todos los URI de Raíces para prevenir recorridos de ruta
   * Implementar controles de acceso adecuados
   * Supervisar la accesibilidad de las Raíces

2. Los servidores **DEBERÍAN**:
   * Manejar casos en los que las Raíces dejen de estar disponibles
   * Respetar los límites de las Raíces durante las operaciones
   * Validar todas las rutas con respecto a las Raíces proporcionadas

<div id="implementation-guidelines">
  ## Directrices de implementación
</div>

1. Los clientes **DEBERÍAN**:
   * Solicitar el consentimiento de los usuarios antes de exponer las Raíces a los servidores
   * Ofrecer interfaces de usuario claras para la gestión de las Raíces
   * Validar la accesibilidad de las Raíces antes de exponerlas
   * Supervisar los cambios en las Raíces

2. Los servidores **DEBERÍAN**:
   * Comprobar la capacidad de Raíces antes de utilizarlas
   * Gestionar adecuadamente los cambios en la lista de Raíces
   * Respetar los límites de las Raíces durante las operaciones
   * Cachear la información de las Raíces de forma adecuada
