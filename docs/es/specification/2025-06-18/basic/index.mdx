<Info>**Revisión del protocolo**: 2025-06-18</Info>

El Protocolo de Contexto de Modelo (MCP) consta de varios componentes clave que funcionan en conjunto:

* **Protocolo base**: Tipos de mensajes principales de JSON-RPC
* **Gestión del ciclo de vida**: Inicialización de la conexión, negociación de capacidades y
  control de la sesión
* **Autorización**: Marco de autenticación y autorización para transportes basados en HTTP
* **Funciones del servidor**: Recursos, Indicaciones y Herramientas expuestos por los servidores
* **Funciones del cliente**: Muestreo y listas de directorios raíz proporcionadas por los clientes
* **Utilidades**: Aspectos transversales como el registro y la autocompletación de argumentos

Todas las implementaciones **DEBEN** admitir el protocolo base y los componentes de gestión del ciclo de vida.
Otros componentes **PUEDEN** implementarse según las necesidades específicas de la
aplicación.

Estas capas del protocolo establecen una separación clara de responsabilidades y permiten interacciones
ricas entre clientes y servidores. El diseño modular permite que las implementaciones
admitan exactamente las funciones que necesitan.

<div id="messages">
  ## Mensajes
</div>

Todos los mensajes entre Clientes MCP y Servidores MCP **DEBEN** seguir la especificación de
[JSON-RPC 2.0](https://www.jsonrpc.org/specification). El protocolo define
estos tipos de mensajes:

<div id="requests">
  ### Solicitudes
</div>

Las solicitudes se envían desde el cliente al servidor o viceversa para iniciar una operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las solicitudes **DEBEN** incluir un identificador (ID) de tipo cadena o entero.
* A diferencia de JSON-RPC básico, el ID **NO DEBE** ser `null`.
* El ID de la solicitud **NO DEBE** haberse usado previamente por el solicitante dentro de la misma
  sesión.

<div id="responses">
  ### Respuestas
</div>

Las respuestas se envían como contestación a las solicitudes y contienen el resultado o el error de la operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* Las respuestas **DEBEN** incluir el mismo ID que la solicitud a la que correspondan.
* **Las respuestas** se clasifican además como **resultados correctos** o
  **errores**. Se **DEBE** establecer un `result` o un `error`. Una respuesta **NO DEBE**
  establecer ambos.
* Los resultados **PUEDEN** seguir cualquier estructura de objeto JSON, mientras que los errores **DEBEN** incluir, como mínimo, un
  código de error y un mensaje.
* Los códigos de error **DEBEN** ser enteros.

<div id="notifications">
  ### Notificaciones
</div>

Las notificaciones se envían del cliente al servidor o viceversa, como mensajes unidireccionales.
El receptor **NO DEBE** enviar una respuesta.

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las notificaciones **NO DEBEN** incluir un ID.

<div id="auth">
  ## Autenticación
</div>

MCP proporciona un marco de [Autorización](/es/specification/2025-06-18/basic/authorization) para su uso con HTTP.
Las implementaciones que utilicen un transporte basado en HTTP **DEBERÍAN** ajustarse a esta especificación,
mientras que las implementaciones que utilicen el transporte STDIO **NO DEBERÍAN** seguir esta especificación,
sino obtener las credenciales del entorno.

Además, los clientes y servidores **PUEDEN** negociar sus propias estrategias personalizadas de autenticación y
autorización.

Para más debates y para contribuir a la evolución de los mecanismos de autenticación de MCP, únete
a la conversación en
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
y ayuda a dar forma al futuro del protocolo.

<div id="schema">
  ## Esquema
</div>

La especificación completa del protocolo se define como un
[esquema de TypeScript](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts).
Esta es la fuente de referencia para todos los mensajes y estructuras del protocolo.

También hay un
[esquema JSON](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.json),
que se genera automáticamente a partir de la fuente de referencia en TypeScript, para su uso con
diversas herramientas automatizadas.

<div id="general-fields">
  ### Campos generales
</div>

<div id="_meta">
  #### `_meta`
</div>

La propiedad/parámetro `_meta` está reservada por MCP para permitir que clientes y servidores
adjunten metadatos adicionales a sus interacciones.

Ciertos nombres de clave están reservados por MCP para metadatos a nivel de protocolo, como se especifica a continuación;
las implementaciones NO DEBEN hacer suposiciones sobre los valores en estas claves.

Además, las definiciones en el [schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts)
pueden reservar nombres concretos para metadatos con un propósito específico, tal como se declara en esas definiciones.

Formato del nombre de clave: los nombres de clave `_meta` válidos tienen dos segmentos: un **prefijo** opcional y un **nombre**.

Prefijo:

* Si se especifica, DEBE ser una serie de etiquetas separadas por puntos (`.`), seguida de una barra (`/`).
  * Las etiquetas DEBEN comenzar con una letra y terminar con una letra o dígito; los caracteres intermedios pueden ser letras, dígitos o guiones (`-`).
* Cualquier prefijo que comience con cero o más etiquetas válidas, seguido de `modelcontextprotocol` o `mcp`, y luego de cualquier etiqueta válida,
  está **reservado** para uso de MCP.
  * Por ejemplo: `modelcontextprotocol.io/`, `mcp.dev/`, `api.modelcontextprotocol.org/` y `tools.mcp.com/` están todos reservados.

Nombre:

* A menos que esté vacío, DEBE comenzar y terminar con un carácter alfanumérico (`[a-z0-9A-Z]`).
* PUEDE contener guiones (`-`), guiones bajos (`_`), puntos (`.`) y caracteres alfanuméricos entre medias.
