[Model Context Protocol](https://modelcontextprotocol.io) (MCP) es un protocolo abierto que
permite la integración fluida entre aplicaciones de LLM y fuentes de datos externas y
herramientas. Ya sea que estés creando un IDE con IA, mejorando una interfaz de chat o diseñando
flujos de trabajo de IA personalizados, MCP proporciona una forma estandarizada de conectar los LLM con el contexto
que necesitan.

Esta especificación define los requisitos definitivos del protocolo, basados en el
esquema de TypeScript en
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2024-11-05/schema.ts).

Para consultar guías de implementación y ejemplos, visita
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Las palabras clave «MUST», «MUST NOT», «REQUIRED», «SHALL», «SHALL NOT», «SHOULD», «SHOULD
NOT», «RECOMMENDED», «NOT RECOMMENDED», «MAY» y «OPTIONAL» en este documento deben
interpretarse como se describe en [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] cuando, y solo cuando,
aparezcan en mayúsculas, como se muestra aquí.

<div id="overview">
  ## Descripción general
</div>

MCP proporciona una forma estandarizada para que las aplicaciones:

* Compartan información contextual con modelos de lenguaje
* Expongan Herramientas y capacidades a los sistemas de IA
* Construyan integraciones y flujos de trabajo componibles

El protocolo utiliza mensajes [JSON-RPC](https://www.jsonrpc.org/) 2.0 para establecer
la comunicación entre:

* **Hosts**: aplicaciones LLM que inician conexiones
* **Clientes**: conectores dentro de la aplicación host
* **Servidores**: servicios que proporcionan contexto y capacidades

MCP se inspira en parte en el
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), que
estandariza cómo agregar compatibilidad con lenguajes de programación en todo un ecosistema de
herramientas de desarrollo. De manera similar, MCP estandariza cómo integrar contexto adicional
y Herramientas en el ecosistema de aplicaciones de IA.

  ## Información clave
</div>

<div id="base-protocol">
  ### Protocolo base
</div>

* Formato de mensaje [JSON-RPC](https://www.jsonrpc.org/)
* Conexiones con estado
* Negociación de capacidades entre servidor y cliente

<div id="features">
  ### Funciones
</div>

Los servidores ofrecen a los clientes cualquiera de las siguientes funciones:

* **Recursos**: Contexto y datos para que los use el usuario o el modelo de IA
* **Indicaciones**: Mensajes y flujos de trabajo con plantillas para usuarios
* **Herramientas**: Funciones que puede ejecutar el modelo de IA

Los clientes pueden ofrecer la siguiente función a los servidores:

* **Muestreo**: Comportamientos con agencia iniciados por el servidor e interacciones recursivas con LLM

<div id="additional-utilities">
  ### Utilidades adicionales
</div>

* Configuración
* Seguimiento del progreso
* Cancelación
* Reporte de errores
* Registro

<div id="security-and-trust-safety">
  ## Seguridad y Confianza y Seguridad
</div>

El Protocolo de Contexto de Modelo (MCP) habilita potentes capacidades mediante el acceso arbitrario a datos
y rutas de ejecución de código. Con este poder surgen importantes consideraciones de seguridad y confianza
que todos los implementadores deben abordar con cuidado.

<div id="key-principles">
  ### Principios clave
</div>

1. **Consentimiento y control del usuario**
   * Los usuarios deben dar su consentimiento explícito y comprender todo acceso a datos y operaciones
   * Los usuarios deben mantener el control sobre qué datos se comparten y qué acciones se realizan
   * Los implementadores deben proporcionar interfaces claras para revisar y autorizar actividades

2. **Privacidad de los datos**
   * Los hosts deben obtener el consentimiento explícito del usuario antes de exponer datos del usuario a los servidores
   * Los hosts no deben transmitir datos de Recursos a ningún otro lugar sin el consentimiento del usuario
   * Los datos del usuario deben protegerse con controles de acceso adecuados

3. **Seguridad de las Herramientas**
   * Las Herramientas representan ejecución de código arbitrario y deben tratarse con la debida precaución
   * Los hosts deben obtener el consentimiento explícito del usuario antes de invocar cualquier Herramienta
   * Los usuarios deben comprender qué hace cada Herramienta antes de autorizar su uso

4. **Controles de Muestreo de LLM**
   * Los usuarios deben aprobar explícitamente cualquier solicitud de Muestreo de LLM
   * Los usuarios deben controlar:
     * Si se realiza el Muestreo o no
     * La Indicacion real que se enviará
     * Qué resultados puede ver el servidor
   * El protocolo limita intencionalmente la visibilidad del servidor sobre las Indicaciones

<div id="implementation-guidelines">
  ### Directrices de implementación
</div>

Si bien el propio MCP no puede hacer cumplir estos principios de seguridad a nivel de protocolo,
los implementadores **DEBERÍAN**:

1. Integrar flujos sólidos de consentimiento y autorización en sus aplicaciones
2. Proporcionar documentación clara sobre las implicaciones de seguridad
3. Implementar controles de acceso y protecciones de datos adecuados
4. Seguir las mejores prácticas de seguridad en sus integraciones
5. Considerar las implicaciones de privacidad en el diseño de sus características
