<Info>**Revisión del protocolo**: 2025-03-26</Info>

El Protocolo de Contexto de Modelo (MCP) consta de varios componentes clave que funcionan en conjunto:

* **Protocolo base**: Tipos de mensajes centrales de JSON-RPC
* **Gestión del ciclo de vida**: Inicialización de la conexión, negociación de capacidades y
  control de sesiones
* **Funciones del servidor**: Recursos, Indicaciones y Herramientas expuestos por los servidores
* **Funciones del cliente**: Muestreo y listas de directorios raíz proporcionadas por los clientes
* **Utilidades**: Aspectos transversales como el registro y la finalización de argumentos

Todas las implementaciones **DEBEN** admitir el protocolo base y los componentes de gestión del ciclo de vida.
Los demás componentes **PUEDEN** implementarse según las necesidades específicas de la aplicación.

Estas capas del protocolo establecen una clara separación de responsabilidades, a la vez que habilitan
interacciones enriquecidas entre clientes y servidores. El diseño modular permite que las implementaciones
admitan exactamente las funciones que necesitan.

<div id="messages">
  ## Mensajes
</div>

Todos los mensajes entre clientes y servidores MCP **DEBEN** ajustarse a la especificación
[JSON-RPC 2.0](https://www.jsonrpc.org/specification). El protocolo define
estos tipos de mensajes:

<div id="requests">
  ### Solicitudes
</div>

Las solicitudes se envían del cliente al servidor o viceversa para iniciar una operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las solicitudes **DEBEN** incluir un ID de tipo cadena o entero.
* A diferencia de JSON-RPC básico, el ID **NO DEBE** ser `null`.
* El ID de la solicitud **NO DEBE** haberse usado anteriormente por el solicitante en la misma sesión.

<div id="responses">
  ### Respuestas
</div>

Las respuestas se envían en contestación a las solicitudes y contienen el resultado o el error de la operación.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* Las respuestas **DEBEN** incluir el mismo ID que la solicitud correspondiente.
* Las **respuestas** se clasifican además como **resultados correctos** o **errores**. Se **DEBE** establecer `result` o `error`. Una respuesta **NO DEBE** establecer ambos.
* Los resultados **PUEDEN** tener cualquier estructura de objeto JSON, mientras que los errores **DEBEN** incluir, como mínimo, un código de error y un mensaje.
* Los códigos de error **DEBEN** ser enteros.

<div id="notifications">
  ### Notificaciones
</div>

Las notificaciones se envían del cliente al servidor o viceversa, como mensajes unidireccionales.
El receptor **NO DEBE** enviar una respuesta.

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Las notificaciones **NO DEBEN** incluir un ID.

<div id="batching">
  ### Procesamiento por lotes
</div>

JSON-RPC también define una forma de
[agrupar varias solicitudes y notificaciones](https://www.jsonrpc.org/specification#batch),
enviándolas en una matriz. Las implementaciones de MCP **PUEDEN** admitir el envío de lotes de JSON-RPC,
pero **DEBEN** admitir la recepción de lotes de JSON-RPC.

<div id="auth">
  ## Autenticación
</div>

MCP proporciona un marco de [Autorización](/es/specification/2025-03-26/basic/authorization) para usar con HTTP.
Las implementaciones que usen un transporte basado en HTTP **DEBERÍAN** ajustarse a esta especificación,
mientras que las implementaciones que usen el transporte STDIO **NO DEBERÍAN** seguirla
y, en su lugar, obtener las credenciales del entorno.

Además, los clientes y servidores **PUEDEN** negociar sus propias estrategias personalizadas de autenticación y
autorización.

Para más debates y para contribuir a la evolución de los mecanismos de autenticación de MCP, únete
a
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
para ayudar a dar forma al futuro del protocolo.

<div id="schema">
  ## Esquema
</div>

La especificación completa del protocolo está definida como un
[esquema de TypeScript](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.ts).
Este es la referencia autorizada para todos los mensajes y estructuras del protocolo.

También hay un
[JSON Schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.json),
que se genera automáticamente a partir de la referencia autorizada en TypeScript, para su uso con
diversas herramientas automatizadas.
