<Info>**プロトコル改訂**: draft</Info>

Model Context Protocol（MCP）は、連携して動作するいくつかの主要コンポーネントから構成されます：

* **ベースプロトコル**: 中核となるJSON-RPCメッセージ型
* **ライフサイクル管理**: 接続の初期化、機能交渉、セッション制御
* **認可**: HTTPベースのトランスポート向けの認証・認可フレームワーク
* **サーバー機能**: サーバーが公開するリソース、プロンプト、ツール
* **クライアント機能**: クライアントが提供するサンプリングとルーツ一覧
* **ユーティリティ**: ロギングや引数補完などの横断的関心事

すべての実装は、ベースプロトコルとライフサイクル管理コンポーネントを**必ず**サポートする必要があります。その他のコンポーネントは、アプリケーションの特定のニーズに応じて**任意で**実装できます。

これらのプロトコル層は、クライアントとサーバー間の豊かな相互作用を可能にしつつ、関心の分離を明確にします。モジュール式の設計により、実装は必要な機能だけを的確にサポートできます。

<div id="messages">
  ## メッセージ
</div>

MCPクライアントとサーバー間のすべてのメッセージは、[JSON-RPC 2.0](https://www.jsonrpc.org/specification) の仕様に**準拠しなければなりません**。このプロトコルでは、次の種類のメッセージが定義されています。

<div id="requests">
  ### リクエスト
</div>

リクエストは、オペレーションを開始するためにクライアントからサーバー、またはその逆に送信されます。

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* リクエストには、文字列または整数のIDを必ず含めなければなりません（MUST）。
* ベースのJSON-RPCとは異なり、IDを`null`にしてはなりません（MUST NOT）。
* 同一セッション内で、そのリクエスト発行者が以前に使用したリクエストIDを再使用してはなりません（MUST NOT）。

<div id="responses">
  ### レスポンス
</div>

レスポンスはリクエストへの返信として送信され、操作の結果またはエラーを含みます。

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* レスポンスは、対応するリクエストと同一のIDを含めることが必須です（MUST）。
* レスポンスは、さらに「成功結果」または「エラー」のいずれかに分類されます。`result` または `error` のどちらか一方を必ず設定し、両方を同時に設定してはいけません（MUST NOT）。
* 結果は任意のJSONオブジェクト構造を取っても構いません（MAY）が、エラーは少なくともエラーコードとメッセージを含める必要があります（MUST）。
* エラーコードは整数でなければなりません（MUST）。

<div id="notifications">
  ### 通知
</div>

通知は、クライアントからサーバー、またはその逆方向へ送信される一方向メッセージです。
受信側は応答を送ってはなりません（MUST NOT）。

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* 通知は ID を含めてはなりません（MUST NOT）。

<div id="auth">
  ## 認証
</div>

MCP は、HTTP で使用するための [Authorization](/ja/specification/draft/basic/authorization) フレームワークを提供します。
HTTP ベースのトランスポートを使用する実装は、この仕様に準拠することが **推奨されます（SHOULD）**。
一方、STDIO トランスポートを使用する実装はこの仕様に **従うべきではありません（SHOULD NOT）**。その代わり、環境から認証情報を取得してください。

また、クライアントとサーバーは、独自のカスタムの認証および認可戦略を交渉して採用しても **かまいません（MAY）**。

MCP の認証メカニズムの発展に関する議論や貢献のために、
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
に参加して、このプロトコルの将来をともに形作りましょう。

<div id="schema">
  ## スキーマ
</div>

このプロトコルの完全な仕様は
[TypeScript スキーマ](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts)
として定義されています。これは、すべてのプロトコルメッセージと構造に関する唯一の信頼できる情報源（ソース・オブ・トゥルース）です。

また、
[JSON Schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.json)
もあり、上記の TypeScript のソース・オブ・トゥルースから自動生成され、各種の自動化ツールで利用できます。

<div id="general-fields">
  ### 共通フィールド
</div>

<div id="_meta">
  #### `_meta`
</div>

`_meta` プロパティ／パラメータは MCP により予約されており、クライアントとサーバーが
やり取りに追加のメタデータを付与できるようにするためのものです。

以下に示すように、特定のキー名はプロトコルレベルのメタデータ用として MCP によって予約されています。
実装はこれらのキーの値について推測や前提を置いてはなりません。

さらに、[schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/draft/schema.ts) の定義において、
特定の目的のメタデータのために特定の名前が予約される場合があります。これは各定義内で宣言されます。

**キー名の形式:** 有効な `_meta` のキー名は、オプションの**プレフィックス**と**名前**の2つのセグメントで構成されます。

**プレフィックス:**

* 指定する場合は、ドット（`.`）で区切られたラベルの並びに続いてスラッシュ（`/`）で終わらなければなりません。
  * ラベルは文字で始まり、文字または数字で終わる必要があります。内部の文字は、文字、数字、またはハイフン（`-`）を使用できます。
* 先頭に有効なラベルが0個以上続き、その後に `modelcontextprotocol` または `mcp` が続き、さらに有効なラベルが続く任意のプレフィックスは、
  MCP 用に**予約**されています。
  * 例: `modelcontextprotocol.io/`、`mcp.dev/`、`api.modelcontextprotocol.org/`、`tools.mcp.com/` はいずれも予約済みです。

**名前:**

* 空でない限り、英数字（`[a-z0-9A-Z]`）で始まり、英数字で終わる必要があります。
* 中間にはハイフン（`-`）、アンダースコア（`_`）、ドット（`.`）、および英数字を含めることができます。

<div id="icons">
  #### `icons`
</div>

`icons` プロパティは、サーバーが自らのリソース、ツール、プロンプト、実装に対する視覚的識別子を公開するための標準化された手段を提供します。アイコンは、視覚的な文脈を与え、利用可能な機能の発見性を高めることで、ユーザーインターフェースを向上させます。

アイコンは `Icon` オブジェクトの配列として表され、各アイコンには次が含まれます:

* `src`: アイコンリソースを指す URI（必須）。次のいずれか:
  * 画像ファイルを指す HTTP/HTTPS の URL
  * base64 エンコード済み画像データを含む data URI
* `mimeType`: サーバー側のタイプ情報が欠落または汎用的な場合に用いる任意の MIME タイプ
* `sizes`: 任意のサイズ指定（例: &quot;48x48&quot;、SVG のようなスケーラブル形式には &quot;any&quot;、複数サイズには &quot;48x48 96x96&quot;）

**必須の MIME タイプサポート:**

アイコンのレンダリングをサポートするクライアントは、少なくとも次の MIME タイプをサポートしなければなりません（MUST）:

* `image/png` - PNG 画像（安全で広く互換性あり）
* `image/jpeg`（および `image/jpg`）- JPEG 画像（安全で広く互換性あり）

アイコンのレンダリングをサポートするクライアントは、次のサポートも推奨されます（SHOULD）:

* `image/svg+xml` - SVG 画像（スケーラブルだが、後述のとおりセキュリティ上の注意が必要）
* `image/webp` - WebP 画像（モダンで効率的な形式）

**セキュリティ上の考慮事項:**

アイコンメタデータの利用者は、侵害を防ぐため、アイコンの取り扱い時に適切なセキュリティ対策を講じなければなりません（MUST）:

* アイコンのメタデータとアイコンのバイト列は信頼できない入力として扱い、ネットワーク、プライバシー、パースに関するリスクに備える。
* アイコンの URI が HTTPS か `data:` であることを確認する。クライアントは `javascript:`、`file:`、`ftp:`、`ws:`、ローカルアプリの URI スキームなど、安全でないスキームやリダイレクトを用いるアイコン URI を拒否しなければならない（MUST）。
  * オリジンの異なるホストへのスキーム変更やリダイレクトを不許可にする。
* 過大な画像サイズ、極端な寸法、過剰なフレーム数（例: GIF）によるリソース枯渇攻撃に対して耐性を持つ。
  * 利用者は画像やコンテンツサイズに上限を設けてもよい（MAY）。
* 資格情報なしでアイコンを取得する。Cookie、`Authorization` ヘッダー、クライアント資格情報は送信しない。
* アイコンの URI がサーバーと同一オリジンであることを検証する。これにより第三者へのデータやトラッキング情報の漏えいリスクを最小化できる。
* ペイロードに実行可能コンテンツが含まれる可能性があるため（MAY）（例: [埋め込み JavaScript](https://www.w3.org/TR/SVG11/script.html) や [拡張機能](https://www.w3.org/TR/SVG11/extend.html) を含む SVG）、アイコンの取得およびレンダリング時には注意する。
  * 利用者は特定のファイルタイプを不許可にする、またはレンダリング前にアイコンファイルをサニタイズしてもよい（MAY）。
* レンダリング前に MIME タイプおよびファイル内容を検証する。MIME タイプ情報は参考情報として扱い、マジックバイトでコンテンツタイプを判別し、不一致または不明なタイプは拒否する。
  * 画像タイプの厳格な許可リストを維持する。

**使用法:**

アイコンは次に添付できます:

* `Implementation`: MCP サーバー/クライアント実装の視覚的識別子
* `Tool`: ツール機能の視覚的表現
* `Prompt`: プロンプトテンプレートの横に表示するアイコン
* `Resource`: リソース種別の違いを示す視覚的インジケーター

異なる表示コンテキストや解像度に対応するため、複数のアイコンを提供できます。クライアントは UI 要件に基づき、最も適切なアイコンを選択してください。
