title: ログ記録
---

<Info>**プロトコル改訂**: 2025-03-26</Info>

Model Context Protocol（MCP）は、サーバーがクライアントに構造化されたログメッセージを送信するための標準化された方法を提供します。クライアントは最小ログレベルを設定してログの詳細度を制御でき、サーバーは重大度レベル、任意のロガー名、任意のJSONシリアライズ可能なデータを含む通知を送信します。

<div id="user-interaction-model">
  ## ユーザーインタラクションモデル
</div>

実装は、ニーズに合った任意のインターフェースパターンでログ出力を提供できます。プロトコル自体は、特定のユーザーインタラクションモデルを規定しません。

<div id="capabilities">
  ## 機能
</div>

ログメッセージ通知を送出するサーバーは、`logging` 機能を宣言することが**必須**です:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

<div id="log-levels">
  ## ログレベル
</div>

本プロトコルは、
[RFC 5424](https://datatracker.ietf.org/doc/html/rfc5424#section-6.2.1) で規定された標準的な syslog の重大度レベルに従います。

| レベル     | 説明                         | 使用例                       |
| --------- | ---------------------------- | ---------------------------- |
| debug     | 詳細なデバッグ情報           | 関数の入退出点               |
| info      | 一般的な情報メッセージ       | 処理の進捗更新               |
| notice    | 通常だが重要なイベント       | 設定変更                     |
| warning   | 警告状態                     | 廃止予定機能の使用           |
| error     | エラー状態                   | 処理の失敗                   |
| critical  | 重大な障害                   | システムコンポーネントの故障 |
| alert     | 直ちに対処が必要             | データ破損の検出             |
| emergency | システムが使用不能           | システム全体の障害           |

<div id="protocol-messages">
  ## プロトコルメッセージ
</div>

<div id="setting-log-level">
  ### ログレベルの設定
</div>

最小ログレベルを設定するために、クライアントは `logging/setLevel` リクエストを送信してもよい（MAY）:

**リクエスト:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "logging/setLevel",
  "params": {
    "level": "info"
  }
}
```

<div id="log-message-notifications">
  ### ログメッセージ通知
</div>

サーバーは `notifications/message` 通知を使用してログメッセージを送信します。

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/message",
  "params": {
    "level": "error",
    "logger": "database",
    "data": {
      "error": "Connection failed",
      "details": {
        "host": "localhost",
        "port": 5432
      }
    }
  }
}
```

<div id="message-flow">
  ## メッセージフロー
</div>

```mermaid
sequenceDiagram
    participant Client
    participant Server

    Note over Client,Server: ログ設定
    Client->>Server: logging/setLevel (info)
    Server-->>Client: 空の結果

    Note over Client,Server: サーバーの活動
    Server--)Client: notifications/message (info)
    Server--)Client: notifications/message (warning)
    Server--)Client: notifications/message (error)

    Note over Client,Server: レベル変更
    Client->>Server: logging/setLevel (error)
    Server-->>Client: 空の結果
    Note over Server: error レベル以上のみ送信<br/>する
```

<div id="error-handling">
  ## エラー処理
</div>

サーバーは、一般的な失敗ケースに対して標準のJSON-RPCエラーを返すべきです（SHOULD）:

- 無効なログレベル: `-32602`（Invalid params）
- 設定エラー: `-32603`（Internal error）

<div id="implementation-considerations">
  ## 実装に関する考慮事項
</div>

1. サーバーは**SHOULD（推奨）**:
   - ログメッセージにレート制限をかける
   - dataフィールドに関連する文脈情報を含める
   - 一貫したロガー名を使用する
   - 機密情報を除去する

2. クライアントは**MAY（任意）**:
   - UIでログメッセージを表示する
   - ログのフィルタリング／検索を実装する
   - 重大度を視覚的に示す
   - ログメッセージを永続化する

<div id="security">
  ## セキュリティ
</div>

1. ログメッセージに含めてはならない（MUST NOT）もの:
   - 資格情報やシークレット
   - 個人を特定できる情報
   - 攻撃の助けとなり得る内部システムの詳細

2. 実装で行うべきこと（SHOULD）:
   - メッセージのレート制限
   - すべてのデータフィールドの検証
   - ログへのアクセス制御
   - 機微情報の監視
