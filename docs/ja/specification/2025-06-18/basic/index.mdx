<Info>**プロトコル改訂**: 2025-06-18</Info>

Model Context Protocol（MCP）は、相互に連携する複数の主要コンポーネントで構成されます:

* **ベースプロトコル**: コアの JSON-RPC 2.0 メッセージ型
* **ライフサイクル管理**: 接続初期化、機能ネゴシエーション、セッション制御
* **認可**: ストリーム対応HTTPベースのトランスポート向け認証・認可フレームワーク
* **サーバー機能**: サーバーが公開するリソース、プロンプト、ツール
* **クライアント機能**: クライアントが提供するサンプリングとルーツ一覧
* **ユーティリティ**: ロギングや引数補完などの横断的関心事

すべての実装は、ベースプロトコルとライフサイクル管理コンポーネントをサポートすることが**必須**です。その他のコンポーネントは、アプリケーションのニーズに応じて**任意**に実装できます。

これらのプロトコル層は、関心の分離を明確にしつつ、クライアントとサーバー間の豊かな相互作用を可能にします。モジュール式の設計により、実装は必要な機能だけを的確にサポートできます。

<div id="messages">
  ## メッセージ
</div>

MCPクライアントとサーバー間のすべてのメッセージは、[JSON-RPC 2.0](https://www.jsonrpc.org/specification) の仕様に必ず従う**必要があります**。このプロトコルでは、次の種類のメッセージが定義されています：

<div id="requests">
  ### リクエスト
</div>

リクエストは、処理を開始するためにクライアントからサーバーへ、またはその逆方向に送信されます。

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* リクエストには、文字列または整数のIDを必ず含めなければなりません（MUST）。
* ベースのJSON-RPCとは異なり、IDを`null`にしてはなりません（MUST NOT）。
* リクエストIDは、同一セッション内でリクエスト送信者が以前に使用したものであってはなりません（MUST NOT）。

<div id="responses">
  ### 応答
</div>

応答はリクエストへの返信として送信され、操作の結果またはエラーを含みます。

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* 応答は、対応するリクエストと同じIDを含めることが**必須**です。
* **応答**はさらに、**成功結果**または**エラー**のいずれかに分類されます。`result`または`error`のいずれかを設定することが**必須**です。両方を設定しては**なりません**。
* 結果は任意のJSONオブジェクト構造を取ることが**可能**ですが、エラーは少なくともエラーコードとメッセージを含めることが**必須**です。
* エラーコードは整数であることが**必須**です。

<div id="notifications">
  ### 通知
</div>

通知は、クライアントからサーバー、またはその逆に送信される一方向メッセージです。
受信側は応答を送ってはなりません（MUST NOT）。

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* 通知にIDを含めてはなりません（MUST NOT）。

<div id="auth">
  ## 認証・認可
</div>

MCPはHTTPでの利用に向けた[Authorization](/ja/specification/2025-06-18/basic/authorization)フレームワークを提供します。
HTTPベースのトランスポートを用いる実装は、この仕様に準拠することが**推奨されます（SHOULD）**。
一方、STDIOトランスポートを用いる実装は、この仕様に**従うべきではありません（SHOULD NOT）**。
その場合は、環境からクレデンシャルを取得してください。

また、クライアントとサーバーは、独自のカスタム認証・認可戦略を交渉**しても構いません（MAY）**。

MCPの認証・認可メカニズムの今後の議論や改善にご参加ください。
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
で、プロトコルの将来像づくりにご協力ください！

<div id="schema">
  ## スキーマ
</div>

プロトコルの完全な仕様は
[TypeScript スキーマ](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts)
として定義されています。これは、すべてのプロトコルメッセージおよび構造の正確な基準となる唯一の情報源（ソース・オブ・トゥルース）です。

また、
[JSON Schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.json)
もあり、各種自動化ツールでの利用のために、上記の TypeScript のソース・オブ・トゥルースから自動生成されています。

<div id="general-fields">
  ### 一般的なフィールド
</div>

<div id="_meta">
  #### `_meta`
</div>

`_meta` プロパティ／パラメータは MCP によって予約されており、クライアントとサーバーが
やり取りに追加のメタデータを付与できるようにするものです。

以下のとおり、特定のキー名はプロトコルレベルのメタデータ用に MCP によって予約されています。
実装はこれらのキーにおける値について推測してはなりません（MUST NOT）。

さらに、[schema](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts) の定義において、
用途特化のメタデータのために特定の名前が予約されている場合があります。これは各定義内で宣言されます。

**キー名の形式:** 有効な `_meta` のキー名は 2 つのセグメントで構成されます。任意の**プレフィックス**と**名前**です。

**プレフィックス:**

* 指定する場合は、ドット（`.`）で区切られたラベルの並びの後にスラッシュ（`/`）を続けた形式でなければなりません（MUST）。
  * ラベルは文字で始まり、文字または数字で終わる必要があります。内部の文字には、文字、数字、またはハイフン（`-`）を用いることができます。
* 先頭から 0 個以上の有効なラベルに続いて `modelcontextprotocol` または `mcp` があり、さらに任意の有効なラベルが続く
  いかなるプレフィックスも、MCP 用に**予約**されています。
  * 例: `modelcontextprotocol.io/`、`mcp.dev/`、`api.modelcontextprotocol.org/`、`tools.mcp.com/` はすべて予約済みです。

**名前:**

* 空でない限り、英数字（`[a-z0-9A-Z]`）で始まり、英数字で終わらなければなりません。
* その中間にはハイフン（`-`）、アンダースコア（`_`）、ドット（`.`）、および英数字を含めてもかまいません（MAY）。
