---
title: ガバナンスとスチュワードシップ
description: Model Context Protocol（MCP）のガバナンス体制とコミュニティへの参加方法について学ぶ
---

Model Context Protocol（MCP）は、透明性の高い意思決定とコミュニティ参加を担保するため、正式なガバナンスモデルに従っています。本書では、プロジェクトの運営体制と意思決定のプロセスを概説します。

<div id="technical-governance">
  ## 技術ガバナンス
</div>

MCPプロジェクトは、Python、PyTorch をはじめとするオープンソースプロジェクトに見られるのと同様の階層構造を採用しています。

* 課題の提出、プルリクエストの作成、プロジェクトへの貢献を行う**コントリビューター**のコミュニティ。
* SDK やドキュメントなど、MCPプロジェクト内の各コンポーネントを推進する少数の**メンテナー**。
* コントリビューターとメンテナーを監督し、プロジェクト全体の方向性をリードする**コアメンテナー**。
* コアメンテナーには、最終的な意思決定を担う 2 名の**リードコアメンテナー**がいる。
* メンテナー、コアメンテナー、リードコアメンテナーで**MCP ステアリンググループ**を構成する。

すべてのメンテナーは、MCP の設計哲学に強く共感し、それを重視する姿勢が期待されます。技術ガバナンスへの参加は企業ではなく個人が対象です。つまり、特定の企業に割り当てられた枠はなく、メンバーシップは所属企業ではなく個人に紐づきます。これにより、メンテナーがプロトコルそのものとオープンソースコミュニティの最善の利益のために行動できるようにしています。

<div id="channels">
  ### チャンネル
</div>

Technical Governance は、すべての**maintainers、core maintainers**、および**lead maintainers**が参加する共通の[Discordサーバー](/ja/community/communication#discord)上で運営されます。各 maintainer グループは追加のコミュニケーションチャネルを選択できますが、すべての意思決定とそれに関する議論は、Discordサーバー上で記録され、透明性をもって公開されなければなりません。

<div id="maintainers">
  ### メンテナー
</div>

メンテナーは、MCPプロジェクト内の個別プロジェクトや技術ワーキンググループに責任を持ちます。これらは通常、言語別SDKのような独立したリポジトリですが、MCPドキュメントのようにリポジトリ内のサブディレクトリに及ぶ場合もあります。メンテナーは意思決定のための独自のルールや手続きを採用できます。各メンテナーは担当プロジェクトについて独立して意思決定することが期待されますが、必要に応じてコアメンテナーに委任またはエスカレーションできます。

メンテナーは次の事項に責任を負います:

* コミュニティコントリビューターとの、思慮深く生産的な関わり
* MCPプロジェクトにおける担当領域の維持・改善
* ドキュメント、ロードマップ、その他MCPプロジェクトに隣接する領域の支援
* コミュニティからのアイデアをコアに提案・共有

メンテナーは、必要に応じて追加のメンテナーを提案することが推奨されます。メンテナーの任命および解任は、理由の如何を問わず、コアメンテナーまたはリードコアメンテナーのみがいつでも行えます。

メンテナーは、それぞれのリポジトリに対して書き込みおよび/または管理アクセス権を持ちます。

<div id="core-maintainers">
  ### コアメンテナー
</div>

コアメンテナーは、Model Context Protocol（MCP）とその仕様について深い理解を有していることが期待されます。彼らの責務には次が含まれます:

* MCPの仕様およびドキュメントなどMCPプロジェクトの他のすべての部分の設計・レビュー・進化の舵取り
* プロジェクトに対する一貫した長期的なビジョンの提示
* 公平かつ透明性をもって対立する問題を仲裁・解決し、可能な限り合意形成を図りつつ、必要に応じて断固とした意思決定を行うこと
* メンテナーの任命または解任
* MCPの最善の利益にかなう形でのMCPプロジェクトのスチュワードシップ

コアメンテナーはグループとして、メンテナーによるあらゆる決定に対し、過半数の投票により拒否権を行使する権限を持ちます。コアメンテナーは、適切と判断する方法で紛争を解決する権限を持ちます。コアメンテナーは、自らの意思決定を公に説明すべきです。コアグループは、意思決定のための独自の手続きを採用する責任を負います。

コアメンテナーは一般に、すべてのMCPリポジトリへの書き込み権限および管理権限を持ちますが、外部コントリビューターと同じ貢献メカニズム（通常はpull request）を用いるべきです。セキュリティ上の観点から例外を設けることがあります。

<div id="lead-maintainers-bdfl">
  ### リードメンテナー（BDFL）
</div>

MCPには2人のリードメンテナーがいます：Justin Spahr-Summers と David Soria Parra。リードメンテナーは、コアメンテナーやメンテナーが下したあらゆる決定に対して拒否権を持ちます。このモデルは、オープンソースコミュニティでは Benevolent Dictator for Life（BDFL）として広く知られています。リードメンテナーは意思決定を公の場で説明し、その理由を明確に示すべきです。リードメンテナーはコアメンテナーグループの一員です。

リードメンテナーは、コアメンテナーの任命および解任の最終責任を負います。

リードメンテナーは、可能な限りMCPプロジェクトのすべてのインフラストラクチャにおける管理者です。これには、すべてのコミュニケーションチャネル、GitHubの組織およびリポジトリが含まれますが、これらに限定されません。

<div id="decision-process">
  ### 意思決定プロセス
</div>

コアメンテナーグループは隔週で集まり、提案の検討と投票を行い、必要なトピックについて協議します。必要に応じて、共有のDiscordサーバーを使って小規模な提案の議論および投票を行うこともできます。

リードメンテナー、コアメンテナー、ならびにメンテナーグループは、3〜6か月ごとに対面での会合を持つよう努めるべきです。

<div id="processes">
  ## プロセス
</div>

コアメンテナーおよびリードメンテナーは、ドキュメント、課題、コンテンツに関する提案、ならびに [MCP プロジェクト](https://github.com/modelcontextprotocol) 配下のその他すべての要素を含む Model Context Protocol のあらゆる側面に責任を負います。メンテナーは、MCP プロジェクト内の担当領域におけるドキュメント、課題、コンテンツ提案に責任を負いますが、MCP プロジェクト全体の保守にも積極的に参加することが推奨されます。メンテナー、コアメンテナー、リードメンテナーは、リポジトリへ直接変更を加えるのではなく、外部コントリビューターと同じコントリビューションプロセスを用いるべきです。これにより、意図の可視化と議論の機会が生まれます。

<div id="projects-and-working-groups">
  ### プロジェクトとワーキンググループ
</div>

MCPプロジェクトは、プロジェクトとワーキンググループという2つの主要な枠組みで構成されています。

プロジェクトは、専用のリポジトリでメンテナンスされる具体的なコンポーネントです。これにはSpecification、TypeScript SDK、Go SDK、Inspector、その他の実装アーティファクトが含まれます。

ワーキンググループは、コードリポジトリを保持せず、関係者がMCPの特定の側面について議論・協働するための場です。これには、トランスポートプロトコル、クライアント実装、その他の横断的な課題に焦点を当てたグループが含まれます。

<div id="governance-principles">
  #### ガバナンス原則
</div>

すべてのプロジェクトとワーキンググループは、以下の中核原則を順守しつつ自律的に運営されます。

1. 明確な貢献プロセスと意思決定プロセス
2. オープンなコミュニケーションと透明性のある意思決定

両者は次を満たす必要があります。

* 貢献プロセスを文書化する
* 透明性のあるコミュニケーションを維持する
* 意思決定を公開で行う（ワーキンググループは議事録と提案を公開すること）

プロセスが明示されていないプロジェクトおよびワーキンググループは、以下をデフォルトとします。

* 貢献には GitHub のプルリクエストと issue を使用する
* 公式の [MCP Contributor Discord](/ja/community/communication#discord) の公開チャンネルを使用する

<div id="maintenance-responsibilities">
  #### メンテナンスの責務
</div>

専任のメンテナーがいないコンポーネント（ドキュメントなど）は、コアメンテナーの責務となります。これらはプルリクエストによる標準的なコントリビューションガイドラインに従い、メンテナーがレビューを行い、重要な変更はコアメンテナーのレビューへエスカレートします。

コアメンテナーおよびメンテナーは、正式な担当範囲にかかわらず、Model Context Protocol（MCP）プロジェクトのあらゆる部分の改善に取り組むことが推奨されています。

<div id="specification-project">
  ### 仕様プロジェクト
</div>

<div id="specification-enhancement-proposal-sep">
  #### 仕様拡張提案（SEP）
</div>

仕様への変更提案は、提案の要約から始まる書面で提出してください。その中で、解決しようとする**問題**、提案する**解決策**、**代替案**、**考慮事項・想定される成果**、**リスク**を明確に示します。[SEP ガイドライン](/ja/community/sep-guidelines)には、SEP に求められる構成が記載されています。SEP は[仕様リポジトリ](https://github.com/modelcontextprotocol/specification)で Issue として作成し、`proposal, sep` のラベルを付与してください。

すべての提案には、MCP ステアリンググループ（メンテナー、コアメンテナー、リードコアメンテナー）からの**スポンサー**が必要です。スポンサーは、提案が継続的に検討・改善され、提案の品質基準を満たすように管理し、コアメンテナー会合での提示と議論を担います。メンテナーおよびコアメンテナーのグループは、スポンサーのいない公開中の提案を定期的にレビューすべきです。6 か月以内にスポンサーが見つからない提案は自動的に却下されます。

提案にスポンサーが付いたら、そのスポンサーに割り当てられ、`draft` としてタグ付けされます。

<div id="core-maintainer-meetings">
  ### コアメンテナー会合
</div>

コアメンテナーグループは隔週で集まり、提案やプロジェクトについて議論します。提案に関するメモは公開されます。コアメンテナーグループは3〜6か月ごとに、対面での会合の実施に努めます。

<div id="public-chat">
  ### 公開チャット
</div>

MCPプロジェクトは、関心のあるグループ向けのオープンチャットを備えた[公開Discordサーバー](/ja/community/communication#discord)を運営しています。MCPプロジェクトでは、特定の連絡や調整のために非公開チャンネルを設ける場合があります。

<div id="nominating-confirming-and-removing-maintainers">
  ## メンテナーの指名、承認、解任
</div>

<div id="the-principles">
  ### 原則
</div>

* モジュールメンテナーグループのメンバーシップは、貢献、レビュー、議論を通じて担当分野で高い専門性を示し、かつMCP全体の方向性に合致していることを踏まえ、功績に基づいて**個人**に付与されます。
* **メンテナー**グループのメンバーシップには、MCP全体の原則との強固で継続的な整合を個人が示す必要があります。
* モジュールメンテナーおよびコアメンテナーに任期制限はありません。
* 作業部会やサブプロジェクトの保守について、長期間積極的な参加がない場合には「名誉（emeritus）」ステータスへ移行するための緩やかな基準を設けます。各メンテナーグループは、自身の領域に適した非活動期間を定義できます。
* メンバーシップは企業ではなく個人に対するものです。

<div id="nomination-and-removal">
  ### 指名と解任
</div>

* コアメンテナーはメンテナーの追加と解任を担当し、既存のメンテナーの意見を考慮します。
* リードメンテナーはコアメンテナーの追加と解任を担当します。

<div id="nomination-process">
  #### 推薦プロセス
</div>

Maintainer（または Core / Lead Maintainer）が Core / Lead Maintainer の検討に付すために推薦を行いたい場合は、次のプロセスに従ってください。

1. 推薦の根拠となる証拠を集めます。一般的には、メンテナーシップを検討しているリポジトリでマージされたPRの履歴が該当します。
2. 関連するグループのメンテナー間で、推薦の承認に賛同できるかを議論します。
3. Community Moderator もしくは Core Maintainer にDMし、Discordで `nomination-{name}-{group}` という形式のプライベートチャンネルを作成してもらいます。関連グループのすべての Core Maintainer、Lead Maintainer、Co-maintainer を追加します。
4. 推薦対象者に関する情報（コンテキスト）を提供します。含める内容の例は以下を参照してください。
5. Discordで投票（Poll）を作成し、Core / Lead Maintainer に対して推薦についての賛否（Yes / No）を投票してもらいます。合意形成が望ましいものの必須ではありません。
6. Core / Lead Maintainer による議論および／または投票の結果、推薦が可決された場合は、GitHubおよびDiscordのロールを更新する権限を持つ該当メンバーが候補者を適切なグループに追加します。推薦者は、該当するDiscordチャンネルで新しいメンテナーシップをアナウンスしてください。
7. この一時的なDiscordチャンネルは1週間後に削除されます。

誰かを推薦する際に Core Maintainer と共有すべき情報の例:

* GitHubプロフィールのリンク、LinkedInプロフィールのリンク、Discordユーザー名
* どのグループのメンテナーシップに推薦しているか
* グループとして当該人物のメンテナー昇格に同意しているか
* これまでの貢献内容の説明（主要な貢献へのリンクを含む）
* 今後見込まれる貢献の説明（例: メンテナーとしての意欲はあるか？対応できる時間的余裕はあるか？）
* 当該人物に関するその他の情報（例: 現在の所属企業、Model Context Protocol（MCP）への関与動機）
* 推薦の検討にあたり有益と思われるその他の事項
