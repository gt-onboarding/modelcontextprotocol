<div id="what-is-a-sep">
  ## SEP とは何ですか？
</div>

SEP は Specification Enhancement Proposal（仕様拡張提案）の略称です。SEP は、MCP コミュニティに情報を提供したり、Model Context Protocol およびそのプロセスや環境に関する新機能を記述したりするための設計ドキュメントです。SEP では、その機能の簡潔な技術仕様と、その機能の採用理由（根拠）を示す必要があります。

私たちは、SEP を大規模な新機能を提案するための主要な手段、特定の課題に関するコミュニティの意見を集約するための手段、そして MCP における設計上の意思決定を記録するための手段として位置づけています。SEP の著者は、コミュニティ内で合意形成を進め、異論も併記して記録する責任を負います。

SEP はバージョン管理されたリポジトリ（GitHub の Issue）内のテキストファイルとして管理されるため、その改訂履歴が機能提案の歴史的記録となります。

<div id="what-qualifies-a-sep">
  ## どのような場合に SEP が適用されますか？
</div>

このプロセスの目的は、広範なコミュニティでの議論、正式な設計ドキュメント、意思決定プロセスの履歴を要する、十分に大きな変更に限って SEP プロセスを適用することです。より小規模で直接的な変更には、通常の GitHub Issue や Pull Request のほうが適切な場合が多くあります。

以下のいずれかに該当する変更を行う場合は、SEP の提案を検討してください。

- **新機能またはプロトコルの変更**: Model Context Protocol における機能の追加・変更・削除を伴うあらゆる変更。例:
  - 新しい API エンドポイントやメソッドの追加
  - 既存のデータ構造やメッセージの構文・セマンティクスの変更
  - さまざまな MCP 互換ツール間の相互運用性に関する新たな標準の導入
  - 仕様そのものの定義・提示・検証方法に関する重大な変更
- **Breaking Change**: 後方互換性のないあらゆる変更
- **ガバナンスまたはプロセスの変更**: プロジェクトの意思決定や貢献ガイドライン（本ドキュメント自体を含む）を変更する提案
- **複雑または論争を招くトピック**: 複数の妥当な解決策が想定される、または大きな議論を生む可能性がある変更の場合、SEP プロセスは代替案の検討、根拠の文書化、実装開始前のコミュニティ合意形成に必要な枠組みを提供します。

<div id="sep-types">
  ## SEPの種類
</div>

SEPには3つの種類があります:

1. **Standards Track** SEPは、Model Context Protocol（MCP）の新機能や実装を記述します。コアのプロトコル仕様の範囲外でサポートされる相互運用性標準について述べる場合もあります。
2. **Informational** SEPは、Model Context Protocol（MCP）の設計上の課題を記述するか、MCPコミュニティに一般的なガイドラインや情報を提供しますが、新機能は提案しません。Informational SEPは、必ずしもMCPコミュニティの合意や推奨を示すものではありません。
3. **Process** SEPは、MCPを取り巻くプロセスを記述するか、プロセスの変更（またはプロセスにおける事象）を提案します。Process SEPはStandards Track SEPに似ていますが、MCPプロトコル自体以外の領域に適用されます。

<div id="submitting-a-sep">
  ## SEP の提出
</div>

SEP プロセスは、Model Context Protocol に関する新しいアイデアから始まります。1 つの SEP には 1 つの主要な提案または新しいアイデアのみを含めることが強く推奨されます。小規模な改良やパッチは多くの場合 SEP を必要とせず、MCP リポジトリへの pull request で MCP の開発ワークフローに取り込めます。SEP は焦点が絞られているほど、成功しやすい傾向があります。

各 SEP には必ず「SEP の著者」が必要です。これは、以下で説明するスタイルと形式に従って SEP を執筆し、適切なフォーラムでの議論を取りまとめ、当該アイデアに関するコミュニティの合意形成を図る役割の人を指します。SEP の著者はまず、そのアイデアが SEP の対象になり得るかを確認するべきです。MCP コミュニティのフォーラム（Discord、GitHub Discussions）に投稿するのが最良の方法です。

<div id="sep-workflow">
  ### SEPのワークフロー
</div>

SEPは[仕様リポジトリ](https://github.com/modelcontextprotocol/modelcontextprotocol)のGitHub Issueとして提出してください。標準的なSEPのワークフローは次のとおりです。

1. あなた（SEPの著者）は、`SEP` と `proposal` のタグを付けた[適切な形式](#sep-format)のGitHub Issueを作成します。SEP番号はGitHub Issue番号と同一で、相互に置き換えて使用できます。
2. 提案を後援するCore MaintainerまたはMaintainerを見つけます。Core MaintainerおよびMaintainerは、どの提案をスポンサーするかを決めるため、オープンな提案一覧を定期的に確認します。提案内で[メンテナ一覧](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/MAINTAINERS.md)から該当するメンテナをタグ付けできます。
3. スポンサーが見つかったら、GitHub Issueはそのスポンサーにアサインされます。スポンサーは `draft` タグを追加し、タイトルにSEP番号が含まれていることを確認し、マイルストーンを設定します。
4. スポンサーは提案を非公式にレビューし、コミュニティからのフィードバックに基づいて変更を求める場合があります。正式なレビューの準備が整ったら、スポンサーは `in-review` タグを追加します。
5. `in-review` タグが追加されると、SEPはCore Maintainersチームによる正式なレビューに入ります。SEPは採択、却下、または修正のため差し戻される場合があります。
6. SEPが3か月以内にスポンサーを見つけられない場合、Core MaintainersはSEPを `dormant` としてクローズすることがあります。

<div id="sep-format">
  ### SEPのフォーマット
</div>

各SEPには以下のパートが必要です:

1. **前文（Preamble）** -- 短い説明的なタイトル、各著者の氏名と連絡先情報、現在のステータス。
2. **要約（Abstract）** -- 取り扱う技術的課題についての短い（約200語）説明。
3. **動機（Motivation）** -- 既存のプロトコル仕様では、当該SEPが解決しようとする問題に対応しきれない理由を明確に説明する必要があります。Model Context Protocol（MCP）の変更を提案するSEPにとって、動機は極めて重要です。十分な動機付けのないSEPは、そのまま却下される場合があります。
4. **仕様（Specification）** -- 技術仕様には、新しいプロトコル機能の構文とセマンティクスを記述します。競合しつつも相互運用可能な実装が可能となるよう、十分に詳細であるべきです。仕様への変更を含むPRも併せて提供してください。
5. **根拠（Rationale）** -- 特定の設計判断がなぜ行われたのかを説明します。検討された代替設計や関連する取り組みも記述してください。コミュニティ内の合意の証拠を示し、議論の過程で提起された重要な異議や懸念についても論じる必要があります。
6. **後方互換性（Backward Compatibility）** -- 後方互換性に関する非互換を導入するすべてのSEPは、それらの非互換とその重大性を説明するセクションを含めなければなりません。著者がそれらの非互換へどのように対処する提案かも説明する必要があります。
7. **リファレンス実装（Reference Implementation）** -- いかなるSEPも、「Final」のステータスが付与される前にリファレンス実装が完了していなければなりませんが、SEPが受理される前に完了している必要はありません。コードを書く前に仕様と根拠について合意形成を図るアプローチには価値がありますが、「ラフコンセンサスと動くコード」という原則は、プロトコル詳細に関する多くの議論を解決するうえでも有用です。
8. **セキュリティへの影響（Security Implications）** -- 当該SEPに関連するセキュリティ上の懸念がある場合、レビュー担当者が認識できるよう、それらを明示的に記述してください。

<div id="sep-states">
  ### SEPの状態
</div>

SEPは次のいずれかの状態になります

- `proposal`: スポンサーのいないSEP提案。
- `draft`: スポンサーのいるSEP提案。
- `in-review`: レビュー可能な状態のSEP提案。
- `accepted`: コアメンテナーにより受理されたSEP。ただし、最終的な文言と参照実装がまだ必要。
- `rejected`: コアメンテナーにより却下されたSEP。
- `withdrawn`: 取り下げられたSEP。
- `final`: 最終化されたSEP。
- `superseded`: より新しいSEPに置き換えられたSEP。
- `dormant`: スポンサーが見つからず、その後クローズされたSEP。

<div id="sep-review-resolution">
  ### SEPのレビューと決定
</div>

SEPは隔週でMCP Core Maintainersチームによってレビューされます。

SEPが採択されるには、以下の最低条件を満たす必要があります:

- 提案内容を実証するプロトタイプ実装
- MCPエコシステムに対する明確なメリット
- コミュニティの支持と合意

SEPが採択されたら、リファレンス実装を完了する必要があります。リファレンス実装が完了し、メインのソースコードリポジトリに統合されると、ステータスは「Final」に変更されます。

SEPは「Rejected」または「Withdrawn」となる場合もあります。「Withdrawn」となったSEPは、後日に再提出できます。

<div id="reporting-sep-bugs-or-submitting-sep-updates">
  ## SEPのバグ報告やSEP更新の提出
</div>

バグの報告やSEPの更新の提出方法は、SEPの成熟度、SEPの著者の意向、コメントの性質など、いくつかの要因によって異なります。まだ`final`状態に達していないSEPについては、コメントや変更をSEPの著者に直接送るのが最善でしょう。SEPが最終化された後は、参照実装のIssueやPull Requestに対するGitHubのコメントとして修正を提出することを検討してください。

<div id="transferring-sep-ownership">
  ## SEPの所有権移転
</div>

ときには、SEPの所有権を新しいSEPの著者へ移転する必要が生じることがあります。一般的には、移転後のSEPにも元の著者を共同著者として残したいと考えていますが、最終的な判断は元の著者に委ねられます。所有権を移転する妥当な理由としては、元の著者が更新やSEPプロセスの遂行に割く時間や関心を失っている場合、あるいはネット上から姿を消してしまい（つまり連絡が取れない、メールにも応答しない）場合などが挙げられます。所有権を移転する不適切な理由は、単にそのSEPの方向性に同意できないという理由です。私たちはSEPを巡ってコンセンサスの形成に努めますが、それが難しい場合は、いつでも競合するSEPを提出できます。
