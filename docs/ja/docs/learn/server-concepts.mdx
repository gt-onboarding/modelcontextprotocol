---
title: MCPサーバーの概要
sidebarTitle: サーバー
---

MCPサーバーは、標準化されたプロトコルインターフェースを通じて、AIアプリケーションに特定の機能を提供するプログラムです。

代表的な例としては、ドキュメントへのアクセスを行うファイルシステムサーバー、データを照会するデータベースサーバー、コード管理のためのGitHubサーバー、チームのコミュニケーションに用いるSlackサーバー、スケジュール管理のためのカレンダーサーバーなどがあります。

<div id="core-server-features">
  ## コアサーバー機能
</div>

サーバーは次の3つの構成要素を通じて機能を提供します:

| 機能          | 説明                                                                                                                                                                                      | 例                                                                | 誰が制御するか |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- | --------------- |
| **ツール**     | LLMが能動的に呼び出し、ユーザーのリクエストに基づいて使用タイミングを判断する関数。ツールはデータベースへの書き込み、外部APIの呼び出し、ファイルの変更、その他のロジックの実行を行えます。 | Search flights<br />Send messages<br />Create calendar events      | Model           |
| **リソース** | ファイル内容、データベーススキーマ、APIドキュメントなど、コンテキスト提供のために読み取り専用アクセスを与える受動的なデータソース。                                                         | Retrieve documents<br />Access knowledge bases<br />Read calendars | Application     |
| **プロンプト**   | 特定のツールやリソースと連携してモデルに作業させるための、再利用可能な事前定義の指示テンプレート。                                                                                           | Plan a vacation<br />Summarize my meetings<br />Draft an email     | User            |

これらの各機能の役割と、それらがどのように連携できるかを示すために、仮想のシナリオを用いて説明します。

<div id="tools">
  ### ツール
</div>

ツールはAIモデルにアクションの実行機能を提供します。各ツールは、型付きの入力と出力をもつ特定の操作を定義します。モデルはコンテキストに応じてツールの実行をリクエストします。

<div id="how-tools-work">
  #### ツールの仕組み
</div>

ツールは、LLMが呼び出せるスキーマ定義のインターフェースです。MCPは検証にJSON Schemaを用います。各ツールは、明確に定義された入力と出力を持つ単一の処理を実行します。ツールは実行前にユーザーの同意を求める場合があり、これによりモデルによる操作に対するユーザーのコントロールを維持できます。

**プロトコルの操作:**

| Method       | Purpose                 | Returns                                |
| ------------ | ----------------------- | -------------------------------------- |
| `tools/list` | 利用可能なツールの一覧取得 | スキーマ付きツール定義の配列           |
| `tools/call` | 指定したツールの実行     | ツールの実行結果                       |

**ツール定義の例:**

```typescript
{
  name: "searchFlights",
  description: "Search for available flights",
  inputSchema: {
    type: "object",
    properties: {
      origin: { type: "string", description: "Departure city" },
      destination: { type: "string", description: "Arrival city" },
      date: { type: "string", format: "date", description: "Travel date" }
    },
    required: ["origin", "destination", "date"]
  }
}
```

<div id="example-travel-booking">
  #### 例: 旅行予約
</div>

ツールは、ユーザーに代わってAIアプリケーションが操作を実行できるようにします。旅行計画のシナリオでは、AIアプリケーションは休暇の手配を支援するために複数のツールを使うことがあります。

**フライト検索**

```
searchFlights(origin: "NYC", destination: "Barcelona", date: "2024-06-15")
```

複数の航空会社を横断的に検索し、構造化されたフライト候補を返します。

**カレンダーの予定ブロック**

```
createCalendarEvent(title: "Barcelona Trip", startDate: "2024-06-15", endDate: "2024-06-22")
```

ユーザーのカレンダーに旅行日程を登録します。

**メール通知**

```
sendEmail(to: "team@work.com", subject: "Out of Office", body: "...")
```

同僚に自動の不在通知メールを送信します。

<div id="user-interaction-model">
  #### ユーザーインタラクションモデル
</div>

ツールはモデル主導で制御され、AIモデルが自動的に発見して呼び出せます。ただし、MCPは複数の仕組みによって人間の関与と監督を重視しています。

信頼性と安全性の観点から、アプリケーションは次のような手段でユーザーによるコントロールを実装できます。

- 利用可能なツールをUIに表示し、特定の対話でツールを利用可能にするかどうかをユーザーが設定できるようにする
- 個別のツール実行ごとの承認ダイアログ
- 安全と判断された特定の操作を事前承認するための権限設定
- すべてのツール実行とその結果を可視化するアクティビティログ

<div id="resources">
  ### リソース
</div>

リソースは、AIアプリケーションが取得し、モデルへのコンテキストとして提供できる情報へ体系的にアクセスする手段を提供します。

<div id="how-resources-work">
  #### リソースの仕組み
</div>

リソースは、ファイル、API、データベース、またはAIがコンテキストを理解するために必要なあらゆる情報源からデータを公開します。アプリケーションはこの情報に直接アクセスでき、どのように活用するかを判断できます—関連部分の抽出、埋め込みによる検索、あるいはすべてをモデルに渡すなど。

各リソースは一意のURI（例: `file:///path/to/document.md`）を持ち、適切なコンテンツ処理のためにMIMEタイプを宣言します。また、次の2つのディスカバリーパターンをサポートします:

- **Direct Resources** - 特定のデータを指す固定URI。例: `calendar://events/2024` - 2024年のカレンダーの空き状況を返す
- **Resource Templates** - 柔軟なクエリのためのパラメータ付き動的URI。例:
  - `travel://activities/{city}/{category}` - 都市とカテゴリごとのアクティビティを返す
  - `travel://activities/barcelona/museums` - バルセロナのすべての博物館を返す

Resource Templatesには、タイトル、説明、想定MIMEタイプといったメタデータが含まれ、発見可能で自己記述的です。

**プロトコル操作:**

| Method                     | Purpose                           | Returns                                  |
| -------------------------- | --------------------------------- | ---------------------------------------- |
| `resources/list`           | 利用可能なDirect Resourcesの一覧 | リソース記述子の配列                    |
| `resources/templates/list` | Resource Templatesの発見          | リソーステンプレート定義の配列          |
| `resources/read`           | リソース内容の取得                | メタデータ付きリソースデータ            |
| `resources/subscribe`      | リソース変更の監視                | 購読の確認                               |

<div id="example-getting-travel-planning-context">
  #### 例: 旅行計画のコンテキスト取得
</div>

旅行計画の例を続けると、リソースはAIアプリケーションに関連情報へのアクセスを提供します：

- **カレンダーデータ**（`calendar://events/2024`）— ユーザーの空き状況を確認
- **旅行書類**（`file:///Documents/Travel/passport.pdf`）— 重要な書類にアクセス
- **過去の旅程**（`trips://history/barcelona-2023`）— 過去の旅行や嗜好を参照

AIアプリケーションはこれらのリソースを取得し、埋め込みやキーワード検索でデータの一部を選別するか、生データをそのままモデルに渡すかなど、処理方法を判断します。

このケースでは、カレンダーデータ、天気情報、旅行の嗜好をモデルに提供し、空き状況の確認、天候パターンの参照、過去の旅行嗜好の参照を可能にします。

**リソーステンプレートの例:**

```json
{
  "uriTemplate": "weather://forecast/{city}/{date}",
  "name": "weather-forecast",
  "title": "Weather Forecast",
  "description": "Get weather forecast for any city and date",
  "mimeType": "application/json"
}

{
  "uriTemplate": "travel://flights/{origin}/{destination}",
  "name": "flight-search",
  "title": "Flight Search",
  "description": "Search available flights between cities",
  "mimeType": "application/json"
}
```

これらのテンプレートにより、柔軟なクエリが可能になります。天気データでは、任意の都市と日付の組み合わせの予報にアクセスできます。フライトでは、任意の2つの空港間の便を検索できます。ユーザーが`origin`空港として「NYC」を入力し、`destination`空港に「Bar」と入力し始めた場合、システムは「Barcelona（BCN）」や「Barbados（BGI）」を提案できます。

<div id="parameter-completion">
  #### パラメータ補完
</div>

動的リソースはパラメータ補完をサポートします。例えば:

- `weather://forecast/{city}` に「Par」と入力すると、「Paris」や「Park City」が提案される場合があります
- `flights://search/{airport}` に「JFK」と入力すると、「JFK - John F. Kennedy International」が提案される場合があります

この仕組みにより、厳密な書式を知らなくても妥当な値を見つけやすくなります。

<div id="user-interaction-model">
  #### ユーザーインタラクションモデル
</div>

リソースはアプリケーション主導で扱われ、取得・処理・提示の方法に柔軟性があります。一般的なインタラクションパターンには次のようなものがあります。

- フォルダに近い馴染みのある構造でリソースを閲覧できるツリー／リストビュー
- 特定のリソースを見つけるための検索・フィルター機能
- ヒューリスティクスやAIによる選択に基づいた自動コンテキストの取り込みやスマートな提案
- 単一または複数のリソースを取り込むための手動選択／一括選択インターフェース

アプリケーションは、ニーズに合った任意のインターフェースパターンでリソース探索を実装できます。プロトコルは特定のUIパターンを規定しないため、プレビュー機能付きのリソースピッカー、現在の会話コンテキストに基づくスマート提案、複数リソースの一括取り込み、既存のファイルブラウザやデータエクスプローラとの統合などが可能です。

<div id="prompts">
  ### プロンプト
</div>

プロンプトは再利用可能なテンプレートです。これにより、MCPサーバーの作者は特定のドメイン向けにパラメータ化されたプロンプトを提供したり、MCPサーバーの効果的な使い方を示したりできます。

<div id="how-prompts-work">
  #### プロンプトの仕組み
</div>

プロンプトは、想定される入力と対話パターンを定義する構造化テンプレートです。自動的に起動されるのではなく、明示的に呼び出すユーザー主導の機能です。プロンプトはコンテキストを認識し、利用可能なリソースやツールを参照して包括的なワークフローを構築できます。リソースと同様に、プロンプトはパラメータ補完をサポートし、有効な引数値の探索を支援します。

**プロトコル操作:**

| Method         | Purpose                 | Returns                               |
| -------------- | ----------------------- | ------------------------------------- |
| `prompts/list` | 利用可能なプロンプトの一覧 | プロンプト記述子の配列                 |
| `prompts/get`  | プロンプトの詳細の取得     | 引数を含む完全なプロンプト定義         |

<div id="example-streamlined-workflows">
  #### 例: ワークフローの効率化
</div>

プロンプトは、よくあるタスク向けに構造化されたテンプレートを提供します。旅行計画の文脈では:

**「休暇を計画する」プロンプト:**

```json
{
  "name": "plan-vacation",
  "title": "Plan a vacation",
  "description": "Guide through vacation planning process",
  "arguments": [
    { "name": "destination", "type": "string", "required": true },
    { "name": "duration", "type": "number", "description": "days" },
    { "name": "budget", "type": "number", "required": false },
    { "name": "interests", "type": "array", "items": { "type": "string" } }
  ]
}
```

自由記述の自然言語入力ではなく、プロンプトシステムにより次のことが可能になります:

1. 「休暇を計画する」テンプレートの選択
2. 構造化された入力: バルセロナ、7日、$3000、["beaches", "architecture", "food"]
3. テンプレートに基づく一貫したワークフローの実行

<div id="user-interaction-model">
  #### ユーザーインタラクションモデル
</div>

プロンプトはユーザーの管理下にあり、明示的に呼び出す必要があります。プロトコルは、実装者が自分たちのアプリケーションになじむ自然なインターフェースを設計できるよう柔軟性を提供します。主な原則は次のとおりです。

- 利用可能なプロンプトを見つけやすくする
- 各プロンプトの挙動を明確に説明する
- 検証付きで自然に引数を入力できること
- プロンプトの基礎となるテンプレートを透明性高く表示する

アプリケーションは通常、次のようなさまざまなUIパターンでプロンプトを提供します。

- スラッシュコマンド（「/」を入力して、/plan-vacation のような利用可能なプロンプトを表示）
- 検索しやすいアクセスのためのコマンドパレット
- 使用頻度の高いプロンプト向けの専用UIボタン
- 関連するプロンプトを提案するコンテキストメニュー

<div id="bringing-servers-together">
  ## サーバーをつなぐ
</div>

複数のサーバーが連携し、統一されたインターフェースを通じてそれぞれの専門機能を組み合わせたとき、MCPの真価が発揮されます。

<div id="example-multi-server-travel-planning">
  ### 例: 複数サーバーによる旅行計画
</div>

パーソナライズされたAI旅行プランナーアプリケーションを考えてみましょう。3つのサーバーが接続されています：

- **Travel Server** - フライト、ホテル、旅程を扱う
- **Weather Server** - 気象データと予報を提供
- **Calendar/Email Server** - スケジュールとメッセージの管理

<div id="the-complete-flow">
  #### 完全なフロー
</div>

1. **ユーザーがパラメータ付きでプロンプトを実行:**

   ```json
   {
     "prompt": "plan-vacation",
     "arguments": {
       "destination": "Barcelona",
       "departure_date": "2024-06-15",
       "return_date": "2024-06-22",
       "budget": 3000,
       "travelers": 2
     }
   }
   ```

2. **ユーザーが含めるリソースを選択:**
   - `calendar://my-calendar/June-2024`（Calendar Server）
   - `travel://preferences/europe`（Travel Server）
   - `travel://past-trips/Spain-2023`（Travel Server）

3. **AI がツールを使ってリクエストを処理:**

   まず AI は選択されたすべてのリソースを読み取り、コンテキストを収集します。カレンダーから利用可能な日付を特定し、旅行の嗜好から好みの航空会社やホテルタイプを把握し、過去の旅行から気に入った場所を抽出します。

   このコンテキストを基に、AI は一連のツールを実行します:
   - `searchFlights()` - NYC から Barcelona へのフライトを航空会社に照会
   - `checkWeather()` - 旅行日程の気象予報を取得

   続いて AI はこの情報を用いて予約と後続の手順を進め、必要に応じてユーザーの承認を求めます:
   - `bookHotel()` - 指定の予算内でホテルを検索
   - `createCalendarEvent()` - ユーザーのカレンダーに旅行を追加
   - `sendEmail()` - 旅行の詳細を含む確認メールを送信

**結果:** 複数の MCPサーバーを通じて、ユーザーは自分のスケジュールに合わせた Barcelona 旅行の調査と予約を完了しました。"Plan a Vacation" プロンプトが、異なるサーバーにわたってリソース（カレンダーの空き状況や旅行履歴）とツール（フライト検索、ホテル予約、カレンダー更新）を組み合わせるよう AI を導き、コンテキスト収集から予約までを実行しました。数時間かかり得た作業が、MCP により数分で完了しました。
