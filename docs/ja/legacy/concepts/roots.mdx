---
title: "ルーツ"
description: "MCP におけるルーツの理解"
---

ルーツは、MCP においてサーバーが動作できる範囲の境界を定義する概念です。これにより、クライアントは関連するリソースとその所在をサーバーに伝えられます。

<div id="what-are-roots">
  ## ルーツとは？
</div>

ルーツは、クライアントがサーバーに対して処理対象として提示するURIです。クライアントがサーバーに接続すると、サーバーが扱うべきルーツを宣言します。主にファイルシステムのパスに使われますが、HTTP URL を含む任意の有効なURIを指定できます。

例えば、ルーツは次のようになります:

```
file:///home/user/projects/myapp
https://api.example.com/v1
```

<div id="why-use-roots">
  ## なぜルーツを使うのか？
</div>

ルーツにはいくつかの重要な役割があります。

1. **ガイダンス**: 関連するリソースや場所をサーバーに示します
2. **明確化**: どのリソースが自分のワークスペースに含まれるかを明確にします
3. **整理**: 複数のルーツを使うことで、異なるリソースを同時に扱えます

<div id="how-roots-work">
  ## ルーツの仕組み
</div>

クライアントがルーツをサポートしている場合、次を行います：

1. 接続時に `roots` 機能を宣言する
2. サーバーに推奨するルーツの一覧を提供する
3. （対応している場合は）ルーツが変更されたときにサーバーへ通知する

ルーツはあくまで情報提供であり厳密な強制力はありませんが、サーバーは次を行うべきです：

1. 提供されたルーツを尊重する
2. ルーツのURIを用いてリソースを特定・アクセスする
3. ルーツの境界内での操作を優先する

<div id="common-use-cases">
  ## 一般的なユースケース
</div>

ルーツは一般的に次のような範囲を定義するために使われます:

- プロジェクトディレクトリ
- リポジトリの場所
- APIエンドポイント
- 設定の場所
- リソースの境界

<div id="best-practices">
  ## ベストプラクティス
</div>

ルーツを扱うときは:

1. 必要なリソースだけを提案する
2. ルーツには分かりやすく具体的な名前を付ける
3. ルーツへのアクセス状況を監視する
4. ルーツの変更には丁寧に対応する

<div id="example">
  ## 例
</div>

一般的なMCPクライアントがルーツをどのように公開するかは次のとおりです:

```json
{
  "roots": [
    {
      "uri": "file:///home/user/projects/frontend",
      "name": "Frontend Repository"
    },
    {
      "uri": "https://api.example.com/v1",
      "name": "API Endpoint"
    }
  ]
}
```

この設定は、サーバーがローカルのリポジトリとAPIエンドポイントの双方を対象としつつ、それぞれを論理的に分離して扱うべきであることを示しています。
