<Info>**Révision du protocole** : 2025-03-26</Info>

Le Model Context Protocol se compose de plusieurs composants clés qui fonctionnent ensemble :

* **Protocole de base** : Types de messages JSON-RPC de base
* **Gestion du cycle de vie** : Initialisation de la connexion, négociation des capacités et
  contrôle de session
* **Fonctionnalités du serveur** : Ressources, invites et outils exposés par les serveurs
* **Fonctionnalités du client** : Échantillonnage et listes de répertoires racines fournies par les clients
* **Utilitaires** : Aspects transversaux comme la journalisation et la complétion des arguments

Toutes les implémentations **DOIVENT** prendre en charge le protocole de base et les composants de gestion du cycle de vie.
D’autres composants **PEUVENT** être implémentés selon les besoins spécifiques de l’application.

Ces couches du protocole établissent une séparation claire des responsabilités, tout en permettant des interactions riches entre clients et serveurs. La conception modulaire permet aux implémentations de prendre en charge exactement les fonctionnalités dont elles ont besoin.

<div id="messages">
  ## Messages
</div>

Tous les messages entre les Clients MCP et les Serveurs MCP **DOIVENT** respecter la spécification
[JSON-RPC 2.0](https://www.jsonrpc.org/specification). Le protocole définit
les types de messages suivants :

<div id="requests">
  ### Requêtes
</div>

Les requêtes sont envoyées du client au serveur, ou l’inverse, pour initier une opération.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Les requêtes **DOIVENT** inclure un identifiant de type chaîne ou entier.
* Contrairement à JSON-RPC de base, l’identifiant **NE DOIT PAS** être `null`.
* L’identifiant de la requête **NE DOIT PAS** avoir été utilisé auparavant par l’émetteur au cours de la même session.

<div id="responses">
  ### Réponses
</div>

Les réponses sont envoyées à la suite des requêtes et contiennent le résultat ou l’erreur de l’opération.

```typescript
{
  jsonrpc: "2.0";
  id: string | number;
  result?: {
    [key: string]: unknown;
  }
  error?: {
    code: number;
    message: string;
    data?: unknown;
  }
}
```

* Les réponses **DOIVENT** inclure le même ID que la requête à laquelle elles correspondent.
* Les **réponses** sont ensuite catégorisées comme **résultats** ou **erreurs**. Soit un `result`, soit une `error` **DOIT** être défini. Une réponse **NE DOIT PAS** définir les deux.
* Les résultats **PEUVENT** suivre n’importe quelle structure d’objet JSON, tandis que les erreurs **DOIVENT** inclure au minimum un code d’erreur et un message.
* Les codes d’erreur **DOIVENT** être des nombres entiers.

<div id="notifications">
  ### Notifications
</div>

Les notifications sont envoyées du client au serveur, ou l’inverse, sous forme de message unidirectionnel.
Le destinataire **NE DOIT PAS** envoyer de réponse.

```typescript
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

* Les notifications **NE DOIVENT PAS** inclure d’ID.

<div id="batching">
  ### Regroupement
</div>

JSON-RPC définit aussi un moyen de
[regrouper plusieurs requêtes et notifications](https://www.jsonrpc.org/specification#batch),
en les envoyant dans un tableau. Les implémentations MCP **PEUVENT** prendre en charge l’envoi de lots JSON-RPC,
mais **DOIVENT** prendre en charge la réception de lots JSON-RPC.

<div id="auth">
  ## Auth
</div>

Le Model Context Protocol (MCP) fournit un cadre d’[autorisation](/fr-CA/specification/2025-03-26/basic/authorization) pour une utilisation avec HTTP.
Les implémentations utilisant un transport HTTP **DEVRAIENT** se conformer à cette spécification,
tandis que celles utilisant le transport STDIO **NE DEVRAIENT PAS** suivre cette spécification
et devraient plutôt récupérer les informations d’identification à partir de l’environnement.

De plus, les clients et les serveurs **PEUVENT** négocier leurs propres stratégies d’authentification et
d’autorisation personnalisées.

Pour poursuivre les discussions et contribuer à l’évolution des mécanismes d’auth de MCP, joignez-vous
à nous dans
[GitHub Discussions](https://github.com/modelcontextprotocol/specification/discussions)
pour aider à façonner l’avenir du protocole!

<div id="schema">
  ## Schéma
</div>

La spécification complète du protocole est définie dans un
[schéma TypeScript](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.ts).
Il s’agit de la source de vérité pour tous les messages et structures du protocole.

Il existe aussi un
[schéma JSON](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-03-26/schema.json),
généré automatiquement à partir de cette source de vérité en TypeScript, pour utilisation avec
divers outils automatisés.
