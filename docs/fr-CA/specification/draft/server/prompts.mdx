Le Protocole de contexte de modèle (MCP) fournit une façon normalisée pour les serveurs d’exposer des modèles d’invite aux clients. Les invites permettent aux serveurs de fournir des messages structurés et des instructions pour interagir avec des modèles de langue. Les clients peuvent découvrir les invites disponibles, en récupérer le contenu et fournir des paramètres pour les personnaliser.

<div id="user-interaction-model">
  ## Modèle d’interaction utilisateur
</div>

Les Invites sont conçues pour être **contrôlées par l’utilisateur**, ce qui signifie qu’elles sont exposées par les serveurs aux clients afin que l’utilisateur puisse les sélectionner explicitement pour les utiliser.

En général, les invites sont déclenchées au moyen de commandes lancées par l’utilisateur dans l’interface utilisateur, ce qui permet aux utilisateurs de découvrir et d’appeler naturellement les invites disponibles.

Par exemple, sous forme de commandes slash :

![Example of prompt exposed as slash command](/specification/draft/server/slash-command.png)

Cependant, les personnes qui mettent en œuvre le système sont libres d’exposer des invites par tout modèle d’interface qui convient à leurs besoins—le protocole en soi n’impose aucun modèle d’interaction utilisateur spécifique.

<div id="capabilities">
  ## Capacités
</div>

Les serveurs qui prennent en charge les Invites **DOIVENT** déclarer la capacité `prompts` lors de
l’[initialisation](/fr-CA/specification/draft/basic/lifecycle#initialization) :

```json
{
  "capabilities": {
    "prompts": {
      "listChanged": true
    }
  }
}
```

`listChanged` indique si le serveur enverra des notifications lorsque la liste des
Invites disponibles change.

<div id="protocol-messages">
  ## Messages du protocole
</div>

<div id="listing-prompts">
  ### Lister les invites
</div>

Pour récupérer les invites disponibles, les clients envoient une requête `prompts/list`. Cette opération
prend en charge la [pagination](/fr-CA/specification/draft/server/utilities/pagination).

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "prompts/list",
  "params": {
    "cursor": "optional-cursor-value"
  }
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "prompts": [
      {
        "name": "code_review",
        "title": "Demander une révision de code",
        "description": "Demande au LLM d’analyser la qualité du code et de proposer des améliorations",
        "arguments": [
          {
            "name": "code",
            "description": "Code à réviser",
            "required": true
          }
        ],
        "icons": [
          {
            "src": "https://example.com/review-icon.svg",
            "mimeType": "image/svg+xml",
            "sizes": "any"
          }
        ]
      }
    ],
    "nextCursor": "next-page-cursor"
  }
}
```

<div id="getting-a-prompt">
  ### Récupérer une invite
</div>

Pour obtenir une invite précise, les clients envoient une requête `prompts/get`. Les arguments peuvent être
complétés automatiquement au moyen de [l’API de saisie semi-automatique](/fr-CA/specification/draft/server/utilities/completion).

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "prompts/get",
  "params": {
    "name": "code_review",
    "arguments": {
      "code": "def hello():\n    print('world')"
    }
  }
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "description": "Invite d’examen de code",
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "Veuillez examiner ce code Python :\ndef hello():\n    print('world')"
        }
      }
    ]
  }
}
```

<div id="list-changed-notification">
  ### Notification de modification de la liste
</div>

Lorsque la liste des invites disponibles change, les serveurs qui ont déclaré la capacité `listChanged` **DEVRAIENT** envoyer une notification :

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/prompts/list_changed"
}
```

<div id="data-types">
  ## Types de données
</div>

<div id="prompt">
  ### Invite
</div>

Une définition d’invite comprend :

* `name`: Identifiant unique de l’invite
* `title`: Nom facultatif et lisible par les humains de l’invite, à des fins d’affichage
* `description`: Description facultative et lisible par les humains
* `arguments`: Liste facultative d’arguments pour la personnalisation

<div id="promptmessage">
  ### MessageDInvite
</div>

Les messages d’une invite peuvent contenir :

* `role` : « user » ou « assistant » pour indiquer l’interlocuteur
* `content` : l’un des types de contenu suivants :

<Note>
  Tous les types de contenu des messages d’invite prennent en charge des
  [annotations](/fr-CA/specification/2025-06-18/server/resources#annotations) facultatives pour
  les métadonnées concernant l’auditoire, la priorité et les moments de modification.
</Note>

<div id="text-content">
  #### Contenu textuel
</div>

Le contenu textuel représente des messages en texte brut :

```json
{
  "type": "text",
  "text": "The text content of the message"
}
```

Il s’agit du type de contenu le plus courant pour les interactions en langage naturel.

<div id="image-content">
  #### Contenu d’image
</div>

Le contenu d’image permet d’inclure des informations visuelles dans les messages :

```json
{
  "type": "image",
  "data": "base64-encoded-image-data",
  "mimeType": "image/png"
}
```

Les données de l’image DOIVENT être encodées en base64 et inclure un type MIME valide. Cela permet
des interactions multimodales lorsque le contexte visuel est important.

<div id="audio-content">
  #### Contenu audio
</div>

Le contenu audio permet d’inclure des informations audio dans les messages :

```json
{
  "type": "audio",
  "data": "base64-encoded-audio-data",
  "mimeType": "audio/wav"
}
```

Les données audio DOIVENT être codées en base64 et inclure un type MIME valide. Cela permet des interactions multimodales lorsque le contexte audio est important.

<div id="embedded-resources">
  #### Ressources intégrées
</div>

Les ressources intégrées permettent de référencer directement des ressources côté serveur dans les messages :

```json
{
  "type": "resource",
  "resource": {
    "uri": "resource://example",
    "name": "example",
    "title": "My Example Resource",
    "mimeType": "text/plain",
    "text": "Resource content"
  }
}
```

Les ressources peuvent contenir du texte ou des données binaires (blob) et **DOIVENT** inclure :

* Un URI de ressource valide
* Le type MIME approprié
* Du contenu textuel ou des données blob encodées en base64

Les ressources intégrées permettent aux Invites d’intégrer sans heurt du contenu géré par le serveur, comme
de la documentation, des exemples de code ou d’autres ressources de référence, directement dans le flux de
conversation.

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les serveurs **DEVRAIENT** renvoyer des erreurs JSON-RPC standard pour les cas d’échec courants :

* Nom d’invite invalide : `-32602` (Paramètres invalides)
* Arguments requis manquants : `-32602` (Paramètres invalides)
* Erreurs internes : `-32603` (Erreur interne)

<div id="implementation-considerations">
  ## Considérations relatives à l’implémentation
</div>

1. Les serveurs **DEVRAIENT** valider les arguments d’invites avant le traitement
2. Les clients **DEVRAIENT** gérer la pagination pour de grandes listes d’invites
3. Les deux parties **DEVRAIENT** respecter la négociation des capacités

<div id="security">
  ## Sécurité
</div>

Les implémentations DOIVENT valider rigoureusement toutes les entrées et sorties d’Invites afin de prévenir les attaques par injection ou l’accès non autorisé aux Ressources.
