<div id="enable-section-numbers" />

<Info>**Révision du protocole** : ébauche</Info>

Le Protocole de contexte de modèle (MCP) propose une méthode normalisée permettant aux clients d’exposer des « racines » de système de fichiers aux serveurs. Les racines définissent les limites au sein desquelles les serveurs peuvent agir dans le système de fichiers, leur indiquant à quels répertoires et fichiers ils ont accès. Les serveurs peuvent demander la liste des racines aux clients compatibles et recevoir des notifications lorsque cette liste est modifiée.

<div id="user-interaction-model">
  ## Modèle d’interaction utilisateur
</div>

Les Racines dans le MCP sont généralement exposées par des interfaces de configuration d’espaces de travail ou de projets.

Par exemple, les implémentations peuvent offrir un sélecteur d’espace de travail/projet permettant aux utilisateurs de choisir les répertoires et fichiers auxquels le serveur devrait avoir accès. Cela peut être combiné à une détection automatique de l’espace de travail à partir des systèmes de contrôle de version ou des fichiers de projet.

Cependant, les implémentations sont libres d’exposer les Racines via tout modèle d’interface qui convient à leurs besoins — le protocole lui-même n’impose aucun modèle d’interaction utilisateur spécifique.

<div id="capabilities">
  ## Capacités
</div>

Les clients qui prennent en charge les Racines DOIVENT déclarer la capacité `roots` lors de
l’[initialisation](/fr-CA/specification/draft/basic/lifecycle#initialization) :

```json
{
  "capabilities": {
    "roots": {
      "listChanged": true
    }
  }
}
```

`listChanged` indique si le client émettra des notifications lorsque la liste des Racines
change.

<div id="protocol-messages">
  ## Messages du protocole
</div>

<div id="listing-roots">
  ### Répertorier les racines
</div>

Pour obtenir les racines, les serveurs envoient une requête `roots/list` :

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "roots/list"
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "roots": [
      {
        "uri": "file:///home/user/projects/myproject",
        "name": "My Project"
      }
    ]
  }
}
```

<div id="root-list-changes">
  ### Modifications à la liste des Racines
</div>

Lorsque les Racines changent, les clients qui prennent en charge `listChanged` **DOIVENT** envoyer une notification :

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/roots/list_changed"
}
```

<div id="data-types">
  ## Types de données
</div>

<div id="root">
  ### Racine
</div>

Une définition de racine inclut :

* `uri` : Identifiant unique de la racine. Il **DOIT** s’agir d’un URI `file://` dans la
  spécification actuelle.
* `name` : Nom facultatif, lisible par l’humain, à des fins d’affichage.

Exemples de racines pour différents cas d’utilisation :

<div id="project-directory">
  #### Répertoire du projet
</div>

```json
{
  "uri": "file:///home/user/projects/myproject",
  "name": "Mon projet"
}
```

<div id="multiple-repositories">
  #### Plusieurs dépôts
</div>

```json
[
  {
    "uri": "file:///home/user/repos/frontend",
    "name": "Dépôt frontend"
  },
  {
    "uri": "file:///home/user/repos/backend",
    "name": "Dépôt backend"
  }
]
```

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les clients DEVRAIENT renvoyer des erreurs JSON-RPC standard pour les cas d’échec courants :

* Le client ne prend pas en charge les Racines : `-32601` (Method not found)
* Erreurs internes : `-32603`

Exemple d’erreur :

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Roots not supported",
    "data": {
      "reason": "Client does not have roots capability"
    }
  }
}
```

<div id="security-considerations">
  ## Considérations de sécurité
</div>

1. Les clients **DOIVENT** :
   * N’exposer que des Racines avec les autorisations appropriées
   * Valider tous les URI de Racines pour prévenir les traversées de chemins
   * Mettre en place des contrôles d’accès adéquats
   * Surveiller l’accessibilité des Racines

2. Les serveurs **DEVRAIENT** :
   * Gérer les cas où les Racines deviennent indisponibles
   * Respecter les limites des Racines durant les opérations
   * Valider tous les chemins par rapport aux Racines fournies

<div id="implementation-guidelines">
  ## Directives de mise en œuvre
</div>

1. Les clients DEVRAIENT :
   * Demander le consentement des utilisateurs avant d’exposer des racines aux serveurs
   * Offrir des interfaces utilisateur claires pour la gestion des racines
   * Valider l’accessibilité des racines avant de les exposer
   * Surveiller les changements apportés aux racines

2. Les serveurs DEVRAIENT :
   * Vérifier la prise en charge des racines avant utilisation
   * Gérer de façon harmonieuse les modifications de la liste des racines
   * Respecter les limites des racines dans les opérations
   * Mettre en cache l’information sur les racines de manière appropriée
