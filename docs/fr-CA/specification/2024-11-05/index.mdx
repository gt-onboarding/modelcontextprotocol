[Model Context Protocol](https://modelcontextprotocol.io) (MCP) est un protocole ouvert qui
permet une intégration transparente entre les applications LLM et les sources de données externes ainsi que les
Outils. Que vous développiez un IDE propulsé par l’IA, amélioriez une interface de clavardage ou créiez
des flux de travail d’IA personnalisés, MCP offre une méthode normalisée pour connecter les LLM au contexte
dont ils ont besoin.

Cette spécification définit les exigences faisant autorité du protocole, basées sur le
schéma TypeScript dans
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2024-11-05/schema.ts).

Pour des guides de mise en œuvre et des exemples, visitez
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Les mots clés « MUST », « MUST NOT », « REQUIRED », « SHALL », « SHALL NOT », « SHOULD », « SHOULD
NOT », « RECOMMENDED », « NOT RECOMMENDED », « MAY » et « OPTIONAL » dans ce document doivent être
interprétés comme décrit dans [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] lorsqu’ils apparaissent, et seulement lorsqu’ils apparaissent,
en majuscules, comme indiqué ici.

<div id="overview">
  ## Aperçu
</div>

Le MCP propose une méthode normalisée permettant aux applications de :

* Partager des informations contextuelles avec des modèles de langue
* Exposer des outils et des capacités aux systèmes d’IA
* Créer des intégrations et des flux de travail composables

Le protocole utilise des messages [JSON-RPC](https://www.jsonrpc.org/) 2.0 pour établir
la communication entre :

* **Hôtes** : applications LLM qui initient les connexions
* **Clients** : connecteurs au sein de l’application hôte
* **Serveurs** : services qui fournissent du contexte et des capacités

Le MCP s’inspire en partie du
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), qui
normalise la façon d’ajouter la prise en charge de langages de programmation à l’échelle
d’un écosystème d’outils de développement. De façon similaire, le MCP normalise la
façon d’intégrer du contexte supplémentaire et des outils dans l’écosystème des
applications d’IA.

<div id="key-details">
  ## Principaux détails
</div>

<div id="base-protocol">
  ### Protocole de base
</div>

* Format de messages [JSON-RPC](https://www.jsonrpc.org/)
* Connexions avec état
* Négociation des capacités entre le serveur et le client

<div id="features">
  ### Fonctionnalités
</div>

Les serveurs peuvent offrir aux clients l’une ou l’autre des fonctionnalités suivantes :

* **Ressources** : Contexte et données, pour l’utilisateur ou le modèle d’IA
* **Invites** : Messages et flux de travail à base de modèles pour les utilisateurs
* **Outils** : Fonctions que le modèle d’IA peut exécuter

Les clients peuvent offrir la fonctionnalité suivante aux serveurs :

* **Échantillonnage** : Comportements agentiques initiés par le serveur et interactions LLM récursives

<div id="additional-utilities">
  ### Utilitaires supplémentaires
</div>

* Configuration
* Suivi de la progression
* Annulation
* Signalement des erreurs
* Journalisation

<div id="security-and-trust-safety">
  ## Sécurité et confiance
</div>

Le Model Context Protocol offre de puissantes capacités grâce à l’accès arbitraire aux données
et à des parcours d’exécution de code. Ce pouvoir s’accompagne d’importantes considérations
de sécurité et de confiance que tous les implémenteurs doivent aborder avec soin.

<div id="key-principles">
  ### Principes clés
</div>

1. **Consentement et contrôle des utilisateurs**
   * Les utilisateurs doivent donner un consentement explicite à tout accès aux données et à toute opération, et en comprendre la portée
   * Les utilisateurs doivent conserver le contrôle sur les données partagées et les actions effectuées
   * Les implémenteurs devraient fournir des interfaces claires pour examiner et autoriser les activités

2. **Confidentialité des données**
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’exposer des données d’utilisateur à des serveurs
   * Les hôtes ne doivent pas transmettre des données de Ressources ailleurs sans le consentement de l’utilisateur
   * Les données des utilisateurs devraient être protégées par des contrôles d’accès appropriés

3. **Sécurité des Outils**
   * Les Outils impliquent l’exécution de code arbitraire et doivent être traités avec une prudence appropriée
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’invoquer tout Outil
   * Les utilisateurs devraient comprendre ce que fait chaque Outil avant d’en autoriser l’utilisation

4. **Contrôles d’Échantillonnage LLM**
   * Les utilisateurs doivent approuver explicitement toute requête d’Échantillonnage LLM
   * Les utilisateurs devraient contrôler :
     * Si un Échantillonnage a lieu ou non
     * L’Invite réelle qui sera envoyée
     * Quels résultats le serveur peut voir
   * Le protocole limite intentionnellement la visibilité du serveur sur les Invites

<div id="implementation-guidelines">
  ### Lignes directrices de mise en œuvre
</div>

Bien que le MCP ne puisse pas appliquer ces principes de sécurité au niveau du protocole,
les implémenteurs **DEVRAIENT** :

1. Intégrer des parcours robustes de consentement et d’autorisation dans leurs applications
2. Fournir une documentation claire sur les implications en matière de sécurité
3. Mettre en place des contrôles d’accès et des protections des données appropriés
4. Suivre les pratiques exemplaires de sécurité dans leurs intégrations
5. Tenir compte des implications en matière de confidentialité dans la conception de leurs fonctionnalités
