<div id="enable-section-numbers" />

<Info>**Révision du protocole** : 2025-06-18</Info>

Le Protocole de contexte de modèle (MCP) offre une méthode normalisée permettant aux clients d’exposer aux serveurs des « racines » du système de fichiers. Les racines définissent les limites à l’intérieur desquelles les serveurs peuvent fonctionner dans le système de fichiers, leur permettant de savoir à quels répertoires et fichiers ils ont accès. Les serveurs peuvent demander la liste des racines aux clients qui la prennent en charge et recevoir des notifications lorsque cette liste change.

<div id="user-interaction-model">
  ## Modèle d’interaction utilisateur
</div>

Les Racines dans le MCP sont généralement exposées par l’entremise d’interfaces de configuration d’espaces de travail ou de projets.

Par exemple, les implémentations peuvent offrir un sélecteur d’espace de travail/projet qui permet aux utilisateurs de sélectionner les répertoires et les fichiers auxquels le serveur devrait avoir accès. Cela peut être combiné à une détection automatique de l’espace de travail à partir des systèmes de contrôle de version ou des fichiers de projet.

Cependant, les implémentations sont libres d’exposer des Racines par l’entremise de tout modèle d’interface qui répond à leurs besoins—le protocole lui-même n’impose aucun modèle d’interaction utilisateur spécifique.

<div id="capabilities">
  ## Capacités
</div>

Les clients qui prennent en charge les Racines **DOIVENT** déclarer la capacité `roots` lors de
l’[initialisation](/fr-CA/specification/2025-06-18/basic/lifecycle#initialization) :

```json
{
  "capabilities": {
    "roots": {
      "listChanged": true
    }
  }
}
```

`listChanged` indique si le client émettra des notifications lorsque la liste des Racines
change.

<div id="protocol-messages">
  ## Messages du protocole
</div>

<div id="listing-roots">
  ### Répertorier les racines
</div>

Pour récupérer les racines, les serveurs envoient une requête `roots/list` :

**Requête :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "roots/list"
}
```

**Réponse :**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "roots": [
      {
        "uri": "file:///home/user/projects/myproject",
        "name": "My Project"
      }
    ]
  }
}
```

<div id="root-list-changes">
  ### Modifications à la liste des Racines
</div>

Lorsque les Racines changent, les clients qui prennent en charge `listChanged` **DOIVENT** envoyer une notification :

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/roots/list_changed"
}
```

<div id="data-types">
  ## Types de données
</div>

<div id="root">
  ### Racine
</div>

Une définition de racine comprend :

* `uri` : Identifiant unique de la racine. Dans la présente spécification, il **DOIT** s’agir d’un URI `file://`.
* `name` : Nom convivial (facultatif) à des fins d’affichage.

Exemples de racines pour différents cas d’utilisation :

<div id="project-directory">
  #### Répertoire du projet
</div>

```json
{
  "uri": "file:///home/user/projects/myproject",
  "name": "Mon projet"
}
```

<div id="multiple-repositories">
  #### Dépôts multiples
</div>

```json
[
  {
    "uri": "file:///home/user/repos/frontend",
    "name": "Dépôt frontal"
  },
  {
    "uri": "file:///home/user/repos/backend",
    "name": "Dépôt dorsal"
  }
]
```

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les clients **DEVRAIENT** retourner des erreurs JSON-RPC standard pour les défaillances courantes :

* Le client ne prend pas en charge les Racines : `-32601` (Méthode introuvable)
* Erreurs internes : `-32603`

Exemple d’erreur :

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Roots not supported",
    "data": {
      "reason": "Client does not have roots capability"
    }
  }
}
```

<div id="security-considerations">
  ## Considérations de sécurité
</div>

1. Les clients **DOIVENT** :
   * N’exposer que les Racines avec les permissions appropriées
   * Valider tous les URI de Racine pour prévenir les traversées de chemin
   * Mettre en œuvre des contrôles d’accès adéquats
   * Surveiller l’accessibilité des Racines

2. Les serveurs **DEVRAIENT** :
   * Gérer les cas où les Racines deviennent indisponibles
   * Respecter les limites des Racines durant les opérations
   * Valider tous les chemins par rapport aux Racines fournies

<div id="implementation-guidelines">
  ## Directives de mise en œuvre
</div>

1. Les clients **DEVRAIENT** :
   * Demander le consentement des utilisateurs avant d’exposer les Racines aux serveurs
   * Fournir des interfaces utilisateur claires pour la gestion des Racines
   * Valider l’accessibilité des Racines avant de les exposer
   * Surveiller les changements apportés aux Racines

2. Les serveurs **DEVRAIENT** :
   * Vérifier la prise en charge des Racines avant utilisation
   * Gérer avec souplesse les modifications de la liste des Racines
   * Respecter les limites des Racines lors des opérations
   * Mettre en cache adéquatement les informations sur les Racines
