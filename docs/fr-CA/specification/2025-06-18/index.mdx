<div id="enable-section-numbers" />

Le [Model Context Protocol](https://modelcontextprotocol.io) (MCP) est un protocole ouvert qui
permet une intégration harmonieuse entre les applications LLM et les sources de données externes ainsi que les
outils. Que vous créiez un IDE propulsé par l’IA, bonifiiez une interface de clavardage ou mettiez en place
des flux de travail d’IA personnalisés, MCP offre une façon normalisée de connecter les LLM au contexte
dont ils ont besoin.

Cette spécification définit les exigences de protocole faisant autorité, fondées sur le
schéma TypeScript dans
[schema.ts](https://github.com/modelcontextprotocol/specification/blob/main/schema/2025-06-18/schema.ts).

Pour des guides de mise en œuvre et des exemples, consultez
[modelcontextprotocol.io](https://modelcontextprotocol.io).

Les mots‑clés « MUST », « MUST NOT », « REQUIRED », « SHALL », « SHALL NOT », « SHOULD », « SHOULD
NOT », « RECOMMENDED », « NOT RECOMMENDED », « MAY » et « OPTIONAL » dans ce document doivent être
interprétés comme décrit dans [BCP 14](https://datatracker.ietf.org/doc/html/bcp14)
[[RFC2119](https://datatracker.ietf.org/doc/html/rfc2119)]
[[RFC8174](https://datatracker.ietf.org/doc/html/rfc8174)] lorsque, et seulement lorsque, ils
apparaissent en lettres majuscules, comme indiqué ici.

<div id="overview">
  ## Aperçu
</div>

Le MCP offre une façon normalisée aux applications de :

* Partager des informations contextuelles avec des modèles de langage
* Exposer des outils et des capacités aux systèmes d’IA
* Créer des intégrations et des flux de travail composables

Le protocole utilise des messages [JSON-RPC](https://www.jsonrpc.org/) 2.0 pour établir
la communication entre :

* **Hôtes** : applications LLM qui initient des connexions
* **Clients** : connecteurs au sein de l’application hôte
* **Serveurs** : services qui fournissent du contexte et des capacités

Le MCP s’inspire en partie du
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/), qui
normalise la manière d’ajouter la prise en charge des langages de programmation à l’échelle d’un écosystème complet d’outils de développement. De la même façon, le MCP normalise la manière d’intégrer un contexte supplémentaire et des outils dans l’écosystème des applications d’IA.

<div id="key-details">
  ## Points clés
</div>

<div id="base-protocol">
  ### Protocole de base
</div>

* Format de messages [JSON-RPC](https://www.jsonrpc.org/)
* Connexions avec état
* Négociation des capacités entre serveur et client

<div id="features">
  ### Fonctionnalités
</div>

Les serveurs offrent aux clients les fonctionnalités suivantes :

* **Ressources** : Contexte et données, à utiliser par l’utilisateur ou le modèle d’IA
* **Invites** : Messages et flux de travail modèles pour les utilisateurs
* **Outils** : Fonctions que le modèle d’IA peut exécuter

Les clients peuvent offrir les fonctionnalités suivantes aux serveurs :

* **Échantillonnage** : Comportements agentiques initiés par le serveur et interactions LLM récursives
* **Racines** : Requêtes initiées par le serveur sur les limites d’URI ou de système de fichiers dans lesquelles opérer
* **Élicitation** : Requêtes initiées par le serveur pour obtenir des informations supplémentaires auprès des utilisateurs

<div id="additional-utilities">
  ### Utilitaires supplémentaires
</div>

* Configuration
* Suivi de l’avancement
* Annulation
* Signalement des erreurs
* Journalisation

<div id="security-and-trust-safety">
  ## Sécurité et fiabilité
</div>

Le Protocole de contexte de modèle (MCP) offre des fonctionnalités puissantes grâce à l’accès arbitraire aux données
et à des chemins d’exécution de code. Ce pouvoir s’accompagne d’importantes considérations de sécurité et de confiance
que tous les implémenteurs doivent examiner avec soin.

<div id="key-principles">
  ### Principes clés
</div>

1. **Consentement et contrôle de l’utilisateur**
   * Les utilisateurs doivent donner un consentement explicite à tous les accès aux données et opérations, et les comprendre
   * Les utilisateurs doivent conserver le contrôle sur les données partagées et les actions effectuées
   * Les implémenteurs devraient fournir des interfaces claires pour examiner et autoriser les activités

2. **Confidentialité des données**
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’exposer des données d’utilisateur aux serveurs
   * Les hôtes ne doivent pas transmettre des données de Ressources ailleurs sans le consentement de l’utilisateur
   * Les données des utilisateurs devraient être protégées par des contrôles d’accès appropriés

3. **Sécurité des Outils**
   * Les Outils représentent une exécution arbitraire de code et doivent être traités avec la prudence appropriée.
     * En particulier, les descriptions du comportement des Outils, comme les annotations, devraient être considérées comme non fiables, sauf si elles proviennent d’un Serveur MCP de confiance.
   * Les hôtes doivent obtenir le consentement explicite de l’utilisateur avant d’invoquer tout Outil
   * Les utilisateurs devraient comprendre ce que chaque Outil fait avant d’en autoriser l’utilisation

4. **Contrôles d’échantillonnage des LLM**
   * Les utilisateurs doivent approuver explicitement toute demande d’Échantillonnage LLM
   * Les utilisateurs devraient contrôler :
     * Si un échantillonnage a lieu ou non
     * L’Invite réelle qui sera envoyée
     * Quels résultats le serveur peut voir
   * Le protocole limite intentionnellement la visibilité du serveur sur les Invites

<div id="implementation-guidelines">
  ### Directives de mise en œuvre
</div>

Même si le MCP ne peut pas appliquer ces principes de sécurité au niveau du protocole,
les implémenteurs **DEVRAIENT** :

1. Intégrer des parcours robustes de consentement et d’autorisation dans leurs applications
2. Fournir une documentation claire sur les implications en matière de sécurité
3. Mettre en place des contrôles d’accès et des protections des données appropriés
4. Suivre les meilleures pratiques de sécurité dans leurs intégrations
5. Tenir compte des implications en matière de confidentialité dans la conception de leurs fonctionnalités
