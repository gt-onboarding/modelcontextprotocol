Les transports dans le Model Context Protocol (MCP) constituent la base de la communication entre les clients et les serveurs. Un transport gère la mécanique sous-jacente de l’envoi et de la réception des messages.

<div id="message-format">
  ## Format des messages
</div>

Le MCP utilise [JSON-RPC](https://www.jsonrpc.org/) 2.0 comme format d’échange. La couche de transport est chargée de convertir les messages du protocole MCP en JSON-RPC pour la transmission, puis de retransformer les messages JSON-RPC reçus en messages du protocole MCP.

Trois types de messages JSON-RPC sont utilisés :

<div id="requests">
  ### Requêtes
</div>

```typescript
{
  jsonrpc: "2.0",
  id: number | string,
  method: string,
  params?: object
}
```

<div id="responses">
  ### Réponses
</div>

```typescript
{
  jsonrpc: "2.0",
  id: number | string,
  result?: object,
  error?: {
    code: number,
    message: string,
    data?: unknown
  }
}
```

<div id="notifications">
  ### Notifications
</div>

```typescript
{
  jsonrpc: "2.0",
  method: string,
  params?: object
}
```

  ### Entrée/sortie standard (stdio)
</div>

Utilisez stdio lorsque :

- Vous créez des outils en ligne de commande
- Vous mettez en place des intégrations locales
- Vous avez besoin d’une communication simple entre processus
- Vous travaillez avec des scripts shell

<div id="streamable-http">
  ### HTTP diffusable
</div>

Le transport HTTP diffusable utilise des requêtes HTTP POST pour la communication du client vers le serveur et des flux optionnels d’Événements envoyés par le serveur (SSE) pour la communication du serveur vers le client.

Utilisez HTTP diffusable lorsque :

- Vous créez des intégrations web
- Vous avez besoin d’une communication client-serveur via HTTP
- Vous avez besoin de sessions avec état
- Vous devez prendre en charge plusieurs clients simultanés
- Vous implémentez des connexions pouvant être reprises

<div id="how-it-works">
  #### Fonctionnement
</div>

1. **Communication client-serveur** : Chaque message JSON-RPC du client vers le serveur est envoyé sous la forme d’une nouvelle requête HTTP POST vers le point de terminaison MCP.
2. **Réponses du serveur** : Le serveur peut répondre soit par :
   - Une réponse JSON unique (`Content-Type: application/json`)
   - Un flux SSE (`Content-Type: text/event-stream`) pour plusieurs messages
3. **Communication serveur-client** : Les serveurs peuvent envoyer des requêtes/notifications aux clients via :
   - Des flux SSE initiés par des requêtes du client
   - Des flux SSE issus de requêtes HTTP GET vers le point de terminaison MCP

<div id="server">
  #### Serveur
</div>

<CodeGroup>

```typescript TypeScript
import express from "express";

const app = express();

const server = new Server(
  {
    name: "example-server",
    version: "1.0.0",
  },
  {
    capabilities: {},
  },
);

// Le point de terminaison MCP gère à la fois POST et GET
app.post("/mcp", async (req, res) => {
  // Traiter la requête JSON-RPC
  const response = await server.handleRequest(req.body);

  // Retourner une réponse unique ou un flux SSE
  if (needsStreaming) {
    res.setHeader("Content-Type", "text/event-stream");
    // Envoyer des événements SSE...
  } else {
    res.json(response);
  }
});

app.get("/mcp", (req, res) => {
  // Facultatif : prise en charge des flux SSE initiés par le serveur
  res.setHeader("Content-Type", "text/event-stream");
  // Envoyer des notifications/demandes du serveur...
});

app.listen(3000);
```

<div id="session-management">
  #### Gestion des sessions
</div>

HTTP diffusable prend en charge des sessions avec état pour maintenir le contexte entre plusieurs requêtes :

1. **Initialisation de la session** : Les serveurs peuvent attribuer un ID de session lors de l’initialisation en l’incluant dans l’en-tête `Mcp-Session-Id`
2. **Persistance de la session** : Les clients doivent inclure l’ID de session dans toutes les requêtes subséquentes en utilisant l’en-tête `Mcp-Session-Id`
3. **Fin de la session** : Les sessions peuvent être explicitement terminées en envoyant une requête HTTP DELETE avec l’ID de session

Exemple de déroulement de session :

```typescript
// Le serveur attribue un ID de session lors de l’initialisation
app.post("/mcp", (req, res) => {
  if (req.body.method === "initialize") {
    const sessionId = generateSecureId();
    res.setHeader("Mcp-Session-Id", sessionId);
    // Enregistrer l’état de la session...
  }
  // Traiter la requête...
});

// Le client inclut l’ID de session dans les requêtes subséquentes
fetch("/mcp", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Mcp-Session-Id": sessionId,
  },
  body: JSON.stringify(request),
});
```

<div id="resumability-and-redelivery">
  #### Reprise et renvoi
</div>

Pour prendre en charge la reprise des connexions interrompues, HTTP diffusable offre :

1. **ID d’événement** : les serveurs peuvent associer des identifiants uniques aux événements SSE pour le suivi
2. **Reprise à partir du dernier événement** : les clients peuvent reprendre en envoyant l’en-tête `Last-Event-ID`
3. **Relecture des messages** : les serveurs peuvent rejouer les messages manqués à partir du point de déconnexion

Cela garantit une remise fiable des messages, même avec des connexions réseau instables.

<div id="security-considerations">
  #### Considérations de sécurité
</div>

Lors de la mise en œuvre du transport HTTP diffusable, suivez ces pratiques exemplaires en matière de sécurité :

1. **Valider les en-têtes Origin** : Validez toujours l’en-tête `Origin` pour toutes les connexions entrantes afin de prévenir les attaques de réaffectation DNS
2. **Lier à localhost** : En local, liez uniquement à localhost (127.0.0.1) plutôt qu’à toutes les interfaces réseau (0.0.0.0)
3. **Mettre en place l’authentification** : Utilisez une authentification appropriée pour toutes les connexions
4. **Utiliser HTTPS** : Utilisez toujours TLS/HTTPS pour les déploiements en production
5. **Valider les identifiants de session** : Assurez-vous que les identifiants de session sont sécurisés cryptographiquement et correctement validés

Sans ces protections, des attaquants pourraient utiliser la réaffectation DNS pour interagir avec des Serveurs MCP locaux à partir de sites web distants.

<div id="server-sent-events-sse-deprecated">
  ### Événements envoyés par le serveur (SSE) - Obsolète
</div>

<Note>
  Les SSE, en tant que transport autonome, sont obsolètes depuis la version du protocole 2024-11-05.
  Ils ont été remplacés par HTTP diffusable, qui intègre les SSE comme mécanisme
  de diffusion optionnel. Pour des informations sur la rétrocompatibilité, consultez la
  section [rétrocompatibilité](#backwards-compatibility) ci-dessous.
</Note>

L’ancien transport SSE permettait la diffusion du serveur vers le client, avec des requêtes HTTP POST pour la communication du client vers le serveur.

Auparavant utilisé lorsque :

- Seule la diffusion du serveur vers le client est nécessaire
- On travaille avec des réseaux restreints
- Pour mettre en œuvre des mises à jour simples

<div id="legacy-security-considerations">
  #### Considérations de sécurité héritées
</div>

Le transport SSE obsolète présentait des considérations de sécurité similaires à HTTP diffusable, notamment en ce qui concerne les attaques de réassociation DNS. Ces mêmes protections devraient être appliquées lors de l’utilisation de flux SSE dans le cadre du transport HTTP diffusable.

<div id="custom-transports">
  ## Transports personnalisés
</div>

MCP facilite la mise en œuvre de transports personnalisés pour des besoins spécifiques. Toute implémentation de transport doit simplement respecter l’interface Transport :

Vous pouvez mettre en œuvre des transports personnalisés pour :

- Protocoles réseau personnalisés
- Canaux de communication spécialisés
- Intégration à des systèmes existants
- Optimisation des performances

<CodeGroup>

```typescript TypeScript
interface Transport {
  // Démarrer le traitement des messages
  start(): Promise<void>;

  // Envoyer un message JSON-RPC
  send(message: JSONRPCMessage): Promise<void>;

  // Fermer la connexion
  close(): Promise<void>;

  // Rappels
  onclose?: () => void;
  onerror?: (error: Error) => void;
  onmessage?: (message: JSONRPCMessage) => void;
}
```

```python Python
# Notez que même si les serveurs MCP sont souvent implémentés avec asyncio,
# nous recommandons d’implémenter les interfaces bas niveau comme les transports
# avec `anyio` pour une compatibilité plus large.

@contextmanager
async def create_transport(
    read_stream: MemoryObjectReceiveStream[JSONRPCMessage | Exception],
    write_stream: MemoryObjectSendStream[JSONRPCMessage]
):
    """
    Interface de transport pour MCP.

    Args:
        read_stream: Flux pour lire les messages entrants
        write_stream: Flux pour écrire les messages sortants
    """
    async with anyio.create_task_group() as tg:
        try:
            # Démarrer le traitement des messages
            tg.start_soon(lambda: process_messages(read_stream))

            # Envoyer des messages
            async with write_stream:
                yield write_stream

        except Exception as exc:
            # Gérer les erreurs
            raise exc
        finally:
            # Nettoyage
            tg.cancel_scope.cancel()
            await write_stream.aclose()
            await read_stream.aclose()
```

</CodeGroup>

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les implémentations de transport devraient gérer divers scénarios d’erreur :

1. Erreurs de connexion
2. Erreurs d’analyse des messages
3. Erreurs de protocole
4. Expirations de délai réseau
5. Nettoyage des ressources

Exemple de gestion des erreurs :

<CodeGroup>

```typescript TypeScript
class ExampleTransport implements Transport {
  async start() {
    try {
      // Logique de connexion
    } catch (error) {
      this.onerror?.(new Error(`Échec de la connexion : ${error}`));
      throw error;
    }
  }

  async send(message: JSONRPCMessage) {
    try {
      // Logique d’envoi
    } catch (error) {
      this.onerror?.(new Error(`Échec de l’envoi du message : ${error}`));
      throw error;
    }
  }
}
```

```python Python
# Bien que les Serveurs MCP soient souvent implémentés avec asyncio, nous recommandons
# d’implémenter les interfaces bas niveau comme les transports avec « anyio » pour une compatibilité accrue.

@contextmanager
async def example_transport(scope: Scope, receive: Receive, send: Send):
    try:
        # Créer des flux pour une communication bidirectionnelle
        read_stream_writer, read_stream = anyio.create_memory_object_stream(0)
        write_stream, write_stream_reader = anyio.create_memory_object_stream(0)

        async def message_handler():
            try:
                async with read_stream_writer:
                    # Logique de traitement des messages
                    pass
            except Exception as exc:
                logger.error(f"Échec du traitement du message : {exc}")
                raise exc

        async with anyio.create_task_group() as tg:
            tg.start_soon(message_handler)
            try:
                # Fournir (yield) les flux pour la communication
                yield read_stream, write_stream
            except Exception as exc:
                logger.error(f"Erreur de transport : {exc}")
                raise exc
            finally:
                tg.cancel_scope.cancel()
                await write_stream.aclose()
                await read_stream.aclose()
    except Exception as exc:
        logger.error(f"Échec de l’initialisation du transport : {exc}")
        raise exc
```

<div id="best-practices">
  ## Meilleures pratiques
</div>

Lors de la mise en œuvre ou de l’utilisation du transport MCP :

1. Gérez correctement le cycle de vie de la connexion
2. Mettez en place une gestion adéquate des erreurs
3. Libérez les ressources à la fermeture de la connexion
4. Utilisez des délais d’attente appropriés
5. Validez les messages avant l’envoi
6. consignez les événements de transport à des fins de débogage
7. Implémentez une logique de reconnexion au besoin
8. Gérez la contre-pression dans les files de messages
9. Surveillez l’intégrité de la connexion
10. Mettez en place des mesures de sécurité appropriées

<div id="backwards-compatibility">
  ## Rétrocompatibilité
</div>

Pour préserver la compatibilité entre différentes versions du protocole :

<div id="for-servers-supporting-older-clients">
  ### Pour les serveurs qui prennent en charge des clients plus anciens
</div>

Les serveurs souhaitant prendre en charge des clients utilisant le transport HTTP+SSE désapprouvé devraient :

1. Héberger à la fois les anciens points de terminaison SSE et POST en plus du nouveau point de terminaison MCP
2. Traiter les requêtes d’initialisation sur les deux points de terminaison
3. Conserver une logique de traitement distincte pour chaque type de transport

<div id="for-clients-supporting-older-servers">
  ### Pour les clients prenant en charge des serveurs plus anciens
</div>

Les clients souhaitant prendre en charge des serveurs utilisant le transport désapprouvé devraient :

1. Accepter des URL de serveur pouvant utiliser l’un ou l’autre transport
2. Tenter d’envoyer une requête POST `InitializeRequest` avec des en-têtes `Accept` appropriés :
   - En cas de réussite, utiliser le transport HTTP diffusable
   - En cas d’échec avec un statut 4xx, revenir au transport SSE classique
3. Envoyer une requête GET en s’attendant à un flux SSE avec l’événement `endpoint` pour les serveurs classiques

Exemple de détection de compatibilité :

```typescript
async function detectTransport(serverUrl: string): Promise<TransportType> {
  try {
    // Essayer d’abord HTTP diffusable
    const response = await fetch(serverUrl, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        Accept: "application/json, text/event-stream",
      },
      body: JSON.stringify({
        jsonrpc: "2.0",
        method: "initialize",
        params: {
          /* ... */
        },
      }),
    });

    if (response.ok) {
      return "streamable-http";
    }
  } catch (error) {
    // Revenir au SSE classique
    const sseResponse = await fetch(serverUrl, {
      method: "GET",
      headers: { Accept: "text/event-stream" },
    });

    if (sseResponse.ok) {
      return "legacy-sse";
    }
  }

  throw new Error("Unsupported transport");
}
```
