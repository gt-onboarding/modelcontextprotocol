title: "Outils"

Les outils sont une primitive puissante du Model Context Protocol (MCP) qui permet aux serveurs d’exposer des fonctionnalités exécutables aux clients. Grâce aux outils, les LLM peuvent interagir avec des systèmes externes, effectuer des calculs et poser des actions dans le monde réel.

<Note>
  Les outils sont conçus pour être **contrôlés par le modèle**, c’est-à-dire qu’ils sont exposés par les serveurs aux clients afin que le modèle d’IA puisse les invoquer automatiquement (avec une intervention humaine pour accorder l’approbation).
</Note>

<div id="overview">
  ## Aperçu
</div>

Les outils dans le Model Context Protocol (MCP) permettent aux serveurs d’exposer des fonctions exécutables qui peuvent être invoquées par des clients et utilisées par des LLM pour effectuer des actions. Les principaux aspects des outils comprennent :

* **Découverte** : les clients peuvent obtenir une liste des outils disponibles en envoyant une requête `tools/list`
* **Invocation** : les outils sont appelés au moyen de la requête `tools/call`, où les serveurs exécutent l’opération demandée et renvoient les résultats
* **Flexibilité** : les outils peuvent aller de simples calculs à des interactions complexes avec des API

Comme les [Ressources](/fr-CA/legacy/concepts/resources), les outils sont identifiés par des noms uniques et peuvent inclure des descriptions pour guider leur utilisation. Cependant, contrairement aux ressources, les outils représentent des opérations dynamiques qui peuvent modifier l’état ou interagir avec des systèmes externes.

<div id="tool-definition-structure">
  ## Structure de définition d’outil
</div>

Chaque outil est défini avec la structure suivante :

```typescript
{
  name: string;          // Identifiant unique de l’outil
  description?: string;  // Description en langage clair
  inputSchema: {         // Schéma JSON des paramètres de l’outil
    type: "object",
    properties: { ... }  // Paramètres propres à l’outil
  },
  annotations?: {        // Indications facultatives sur le comportement de l’outil
    title?: string;      // Titre en langage clair de l’outil
    readOnlyHint?: boolean;    // Si true, l’outil ne modifie pas son environnement
    destructiveHint?: boolean; // Si true, l’outil peut effectuer des mises à jour destructrices
    idempotentHint?: boolean;  // Si true, des appels répétés avec les mêmes arguments n’ont aucun effet supplémentaire
    openWorldHint?: boolean;   // Si true, l’outil interagit avec des entités externes
  }
}
```

<div id="implementing-tools">
  ## Implémentation des outils
</div>

Voici un exemple d’implémentation d’un outil de base dans un serveur MCP :

<CodeGroup>
  ```typescript TypeScript
  const server = new Server(
    {
      name: "example-server",
      version: "1.0.0",
    },
    {
      capabilities: {
        tools: {},
      },
    },
  );

  // Définir les outils disponibles
  server.setRequestHandler(ListToolsRequestSchema, async () => {
    return {
      tools: [
        {
          name: "calculate_sum",
          description: "Additionner deux nombres",
          inputSchema: {
            type: "object",
            properties: {
              a: { type: "number" },
              b: { type: "number" },
            },
            required: ["a", "b"],
          },
        },
      ],
    };
  });

  // Gérer l’exécution des outils
  server.setRequestHandler(CallToolRequestSchema, async (request) => {
    if (request.params.name === "calculate_sum") {
      const { a, b } = request.params.arguments;
      return {
        content: [
          {
            type: "text",
            text: String(a + b),
          },
        ],
      };
    }
    throw new Error("Outil introuvable");
  });
  ```

  ```python Python
  app = Server("example-server")

  @app.list_tools()
  async def list_tools() -> list[types.Tool]:
      return [
          types.Tool(
              name="calculate_sum",
              description="Additionner deux nombres",
              inputSchema={
                  "type": "object",
                  "properties": {
                      "a": {"type": "number"},
                      "b": {"type": "number"}
                  },
                  "required": ["a", "b"]
              }
          )
      ]

  @app.call_tool()
  async def call_tool(
      name: str,
      arguments: dict
  ) -> list[types.TextContent | types.ImageContent | types.EmbeddedResource]:
      if name == "calculate_sum":
          a = arguments["a"]
          b = arguments["b"]
          result = a + b
          return [types.TextContent(type="text", text=str(result))]
      raise ValueError(f"Outil introuvable : {name}")
  ```
</CodeGroup>

<div id="best-practices">
  ## Meilleures pratiques
</div>

Lors de l’implémentation d’Outils :

1. Fournir des noms et des descriptions clairs et parlants
2. Utiliser des définitions JSON Schema détaillées pour les paramètres
3. Inclure des exemples dans les descriptions des Outils pour montrer comment le modèle doit les utiliser
4. Mettre en place une gestion des erreurs et une validation adéquates
5. Utiliser un suivi de progression pour les opérations longues
6. Garder les opérations des Outils ciblées et atomiques
7. Documenter les structures de valeurs de retour attendues
8. Mettre en place des délais d’expiration appropriés
9. Envisager une limitation de débit pour les opérations gourmandes en ressources
10. Consigner l’utilisation des Outils pour le débogage et la supervision

<div id="tool-name-conflicts">
  ### Conflits de noms d’outils
</div>

Les applications Client MCP et les proxys de Serveur MCP peuvent rencontrer des conflits de noms d’outils lorsqu’elles construisent leurs propres listes d’outils. Par exemple, deux Serveurs MCP connectés, `web1` et `web2`, peuvent tous deux exposer un outil nommé `search_web`.

Les applications peuvent lever toute ambiguïté entre les outils au moyen de l’une des stratégies suivantes (entre autres; liste non exhaustive) :

* Concaténer un nom de serveur unique défini par l’utilisateur avec le nom de l’outil, p. ex. `web1___search_web` et `web2___search_web`. Cette stratégie peut être préférable lorsque des noms de serveur uniques sont déjà fournis par l’utilisateur dans un fichier de configuration.
* Générer un préfixe aléatoire pour le nom de l’outil, p. ex. `jrwxs___search_web` et `6cq52___search_web`. Cette stratégie peut être préférable dans les proxys de serveur où des noms uniques définis par l’utilisateur ne sont pas disponibles.
* Utiliser l’URI du serveur comme préfixe du nom de l’outil, p. ex. `web1.example.com:search_web` et `web2.example.com:search_web`. Cette stratégie peut convenir lorsqu’on travaille avec des Serveurs MCP distants.

Notez que le nom fourni par le serveur lors de l’initialisation n’est pas garanti unique et n’est généralement pas approprié pour la désambiguïsation.

</div>

* N&#39;exposez pas d&#39;erreurs internes aux clients

<div id="tool-discovery-and-updates">
  ## Découverte et mises à jour des outils
</div>

Le MCP prend en charge la découverte dynamique des outils :

1. Les clients peuvent énumérer les outils disponibles en tout temps
2. Les serveurs peuvent avertir les clients lorsque les outils changent à l’aide de `notifications/tools/list_changed`
3. Des outils peuvent être ajoutés ou retirés pendant l’exécution
4. Les définitions d’outils peuvent être mises à jour (mais cela doit être fait avec prudence)

<div id="error-handling">
  ## Gestion des erreurs
</div>

Les erreurs d’outils doivent être signalées dans l’objet de résultat, et non comme des erreurs au niveau du protocole MCP. Cela permet au LLM de voir l’erreur et éventuellement de la gérer. Lorsqu’un outil rencontre une erreur :

1. Réglez `isError` à `true` dans le résultat
2. Incluez les détails de l’erreur dans le tableau `content`

Voici un exemple de bonne gestion des erreurs pour les outils :

<CodeGroup>
  ```typescript TypeScript
  try {
    // Tool operation
    const result = performOperation();
    return {
      content: [
        {
          type: "text",
          text: `Operation successful: ${result}`,
        },
      ],
    };
  } catch (error) {
    return {
      isError: true,
      content: [
        {
          type: "text",
          text: `Error: ${error.message}`,
        },
      ],
    };
  }
  ```

  ```python Python
  try:
      # Tool operation
      result = perform_operation()
      return types.CallToolResult(
          content=[
              types.TextContent(
                  type="text",
                  text=f"Operation successful: {result}"
              )
          ]
      )
  except Exception as error:
      return types.CallToolResult(
          isError=True,
          content=[
              types.TextContent(
                  type="text",
                  text=f"Error: {str(error)}"
              )
          ]
      )
  ```
</CodeGroup>

Cette approche permet au LLM de voir qu’une erreur s’est produite et de potentiellement prendre des mesures correctives ou de demander une intervention humaine.

<div id="tool-annotations">
  ## Annotations d’outil
</div>

Les annotations d’outil fournissent des métadonnées supplémentaires sur le comportement d’un outil, aidant les clients à comprendre comment présenter et gérer les outils. Ces annotations donnent des indications sur la nature et l’impact d’un outil, mais elles ne devraient pas être utilisées pour prendre des décisions en matière de sécurité.

<div id="purpose-of-tool-annotations">
  ### Finalité des annotations d’outil
</div>

Les annotations d’outil remplissent plusieurs objectifs clés :

1. Fournir des informations propres à l’UX sans influer sur le contexte du modèle
2. Aider les clients à catégoriser et à présenter les outils de manière appropriée
3. Transmettre des renseignements sur les effets secondaires potentiels d’un outil
4. Aider à concevoir des interfaces intuitives pour l’approbation des outils

<div id="available-tool-annotations">
  ### Annotations d’outil disponibles
</div>

La spécification MCP définit les annotations suivantes pour les outils :

| Annotation        | Type    | Par défaut | Description                                                                                                                                     |
| ----------------- | ------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`           | string  | -          | Un titre convivial pour l’outil, utile pour l’affichage dans l’interface utilisateur                                                            |
| `readOnlyHint`    | boolean | false      | Si vrai, indique que l’outil ne modifie pas son environnement                                                                                   |
| `destructiveHint` | boolean | true       | Si vrai, l’outil peut effectuer des mises à jour destructrices (pertinent uniquement lorsque `readOnlyHint` est false)                         |
| `idempotentHint`  | boolean | false      | Si vrai, appeler l’outil à répétition avec les mêmes arguments n’a aucun effet supplémentaire (pertinent uniquement lorsque `readOnlyHint` est false) |
| `openWorldHint`   | boolean | true       | Si vrai, l’outil peut interagir avec un « monde ouvert » d’entités externes                                                                    |

<div id="example-usage">
  ### Exemple d’utilisation
</div>

Voici comment définir des outils avec des annotations pour différents scénarios :

```typescript
// Un outil de recherche en lecture seule

// Un outil destructif de suppression de fichiers

  inputSchema: {
    type: "object",
    properties: {
      path: { type: "string" }
    },
    required: ["path"]
  },
  annotations: {
    title: "Supprimer un fichier",
    readOnlyHint: false,
    destructiveHint: true,
    idempotentHint: true,

// Un outil non destructif de création d’enregistrements dans la base de données
{
  name: "create_record",
  description: "Créer un nouvel enregistrement dans la base de données",
  inputSchema: {
    type: "object",
    properties: {
      table: { type: "string" },
      data: { type: "object" }
    },
    required: ["table", "data"]
  },
  annotations: {
    title: "Créer un enregistrement de base de données",
    readOnlyHint: false,
    destructiveHint: false,
    idempotentHint: false,
```

<div id="best-practices-for-tool-annotations">
  ### Bonnes pratiques pour les annotations d’outils
</div>

1. **Soyez précis quant aux effets secondaires** : Indiquez clairement si un outil modifie son environnement et si ces modifications sont destructrices.

2. **Utilisez des titres explicites** : Fournissez des titres compréhensibles pour les humains qui décrivent clairement la finalité de l’outil.

3. **Indiquez correctement l’idempotence** : Marquez les outils comme idempotents uniquement si des appels répétés avec les mêmes arguments n’ont vraiment aucun effet supplémentaire.

4. **Fournissez des indications adéquates pour les mondes ouvert/fermé** : Indiquez si un outil interagit avec un système fermé (comme une base de données) ou un système ouvert (comme le Web).

5. **Rappelez-vous que les annotations sont des indications** : Toutes les propriétés dans ToolAnnotations sont des indications et ne garantissent pas une description fidèle du comportement de l’outil. Les clients ne devraient jamais prendre de décisions critiques en matière de sécurité uniquement sur la base des annotations.

<div id="testing-tools">
  ## Mise à l’essai des outils
</div>

Une stratégie d’essai complète pour les outils MCP devrait couvrir :

* **Essais fonctionnels** : Vérifier que les outils s’exécutent correctement avec des entrées valides et qu’ils gèrent adéquatement les entrées invalides
* **Essais d’intégration** : Tester l’interaction des outils avec des systèmes externes en utilisant à la fois des dépendances réelles et simulées
* **Essais de sécurité** : Valider l’authentification, l’autorisation, l’assainissement des entrées et la gestion du débit
* **Essais de performance** : Vérifier le comportement sous charge, la gestion des délais d’attente et le nettoyage des ressources
* **Gestion des erreurs** : S’assurer que les outils signalent correctement les erreurs par le protocole MCP et nettoient les ressources
